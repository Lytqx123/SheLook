## 一、什么是"大模型"？为什么叫"大"？

"大模型"这个词在 2023 年之后被广泛使用，但很多人对"大"的理解停留在参数数量上——"参数多就是大模型"。实际上，"大"有三个维度，每一个维度都带来了质的变化。缺少任何一个维度，就不构成真正意义上的"大模型"。

### 1.1 参数大：从"记得住"到"想得通"

| 模型 | 参数规模 | 类比 |
|------|----------|------|
| ResNet-50（2015） | 2,500 万 | 一本字典——能查阅具体条目 |
| CLIP ViT-B/32（2021） | 8,600 万 | 一套百科全书——能理解概念间关系 |
| Stable Diffusion XL（2023） | 26 亿 | 一个小型图书馆——能从零生成新内容 |
| GPT-3（2020） | 1,750 亿 | 一个国家图书馆——能推理、翻译、编程 |
| GPT-4（2023） | 未公开（估计 1.8 万亿） | 所有图书馆加起来——接近人类的综合理解 |
| Gemini 2.0 Flash（2025） | 未公开（估计千亿级） | 所有图书馆 + 视频库 + 音频库 |

参数的变化不是量变，是质变。当参数从百万级跨到千亿级，模型突然"涌现"出训练时没有明确教过的能力——比如推理、代码生成、多语言翻译。

"涌现"（Emergence）是一个在复杂系统中被广泛观察到的现象：当系统的组成部分达到一定数量和复杂度时，会出现单个组成部分不具备的全局行为。蚂蚁个体的行为很简单，但蚁群展现出复杂的社会分工和路径优化能力——没有一个蚂蚁"知道"整个蚁群在做什么。大模型的涌现也是类似的逻辑：没有一个参数"知道"怎么做推理，但 1750 亿个参数的组合产生了一个能做推理的整体。

这种涌现能力的标志是：模型的表现在某些任务上突然从随机水平（接近瞎猜）跳变到接近人类水平，而且这个跳变不是渐进的、而是突然发生的。Google 和 DeepMind 的研究人员在 2022 年的论文《Emergent Abilities of Large Language Models》中系统记录了这种现象——模型从 10B 参数增长到 100B 参数的过程中，算术推理、多语言翻译、代码生成等能力几乎是"瞬间"出现的，而不是逐步提升。

这在工程上有直接的后果：你不能通过训练一个 1000 万参数的模型 100 次来获得一个 10 亿参数模型的能力。涌现需要一个足够大的网络来承载，这不是训练次数的问题，是网络容量的物理上限。SheLook 选择调用外部大模型 API 而非自己训练，这个决策的根本原因就在这里——训练一个能产生涌现能力的大模型需要千亿参数、数千张 GPU 运行数月、数百万美元的电力成本。对于任何非 AI 巨头来说，这不是"能不能做"的问题，而是"值不值得"的问题。

### 1.2 数据大：喂给模型的是整个互联网

CLIP 用了 4 亿图文对训练——这在 2021 年是天文数字。GPT-3 用了约 45TB 的文本数据，涵盖了维基百科、书籍、新闻、论坛、代码仓库等几乎所有公开可获取的文本来源。Gemini 的训练数据更进一步——包含文本、图片、视频、音频、代码，覆盖多种语言和文化背景。

训练一个千亿参数模型需要的资源是惊人的：

- 数千张 GPU（NVIDIA A100 或 H100，每张价格约 1-3 万美元）运行数月。以 GPT-3 为例，据估算使用了约 1 万张 V100 GPU 训练了数周。GPT-4 的训练规模至少是一个数量级以上的提升。
- 电力成本约数百万美元级别。这还不包括硬件采购和冷却系统的开销——一个千亿参数模型的训练集群，其散热系统的功率本身就是一个中小型数据中心的规模。
- 训练一次排放的碳相当于几十辆汽车一年的排放。OpenAI 和 Google 在这方面面临越来越大的环保压力，都在承诺使用可再生能源来抵消训练碳排放。
- 数十人的顶级工程师和研究员团队，他们的薪资成本在硅谷每年数百万美元——而市场上能做这个级别模型的团队全球不超过几十个。

这就是为什么 SheLook 不自己训练大模型，而是调用外部 API。这不是工程能力问题——SheLook 的技术团队完全有能力搭建训练基础设施。但这不是技术问题，而是经济模型问题：训练一个千亿参数大模型的一次性成本是数百万到数千万美元，而调用 API 的成本是按每次请求几分钱计算的。对于一个 SaaS 产品来说，后者是更理性的选择。把宝贵的工程资源投入到业务逻辑、用户体验、数据飞轮上，比投入到大模型训练上对最终用户的 ROI 更高。

SheLook 的策略是"站在巨人的肩膀上"：把大模型当做和水、电、网络一样的基础设施来使用，专注于如何将这些基础设施高效地编排进电商视觉决策的具体场景中。这也是当前大多数 AI 应用公司的共同选择——模型层由少数几家巨头垄断，应用层百花齐放。

### 1.3 能力大：从"执行指令"到"理解意图"

传统模型：一个模型做一件事。ResNet 分类图片、YOLO 检测物体、BERT 理解文本——每个模型有一个明确的输入格式和一个明确的输出格式，不能越界。

大模型：一个模型什么都会一点。你不需要告诉 Gemini"请执行图像分类任务，类别列表是……"，你只需要说"这张图的颜色搭配怎么样？"——它自己会理解你想做什么、需要输出什么格式、需要调用哪些内部"能力模块"。

这种"从指令中理解意图"的能力叫做 **In-Context Learning（上下文学习）**，是大模型区别于所有传统 ML 模型的核心特征。它的奇妙之处在于：这个能力不是被明确训练的。GPT-3 的训练目标很简单——给定前文，预测下一个 token（字词或子词）。但在海量数据和海量参数的训练过程中，模型"意外地"学会了理解人类指令的结构和意图——因为互联网上的文本本身就包含了大量的指令-执行对（教程、问答、对话、代码注释）。

In-Context Learning 意味着你不需要为每个新任务准备标注数据、微调模型。你只需要在 prompt 中写清楚要求和格式，模型就能适应。这种灵活性在 SheLook 中至关重要——因为电商视觉审核的标准可能每月都在变化（新的平台规则、新的审美趋势、新的合规要求），如果每次变化都需要重新训练模型，系统根本无法跟上业务节奏。而 In-Context Learning 让"更新审核标准"变成了"修改 prompt 中的一段文字"——五分钟完成，不需要重新训练。

## 二、多模态：为什么模型要"能看又能写"？

### 2.1 单模态 vs 多模态

要理解什么是多模态，先要理解什么是"模态"（Modality）。模态就是信息的载体形式——文字是一种模态，图片是另一种，音频是第三种，视频是第四种。不同模态的信息在物理形态上完全不同：文字是由离散的符号（字、词）组成的序列，图片是由连续的像素值组成的矩阵，音频是由随时间变化的波形组成的信号。

单模态模型只理解一种信息形式，对其他形式的"语言"一窍不通：

```
GPT-2（纯文本模型）：
  你给它看一张猫的照片 → 它不知道这是什么，可能会直接报错。
  你告诉它"前面有一张猫的照片" → 它只能从文字层面理解这句话，无法"看"到猫。

ResNet（纯视觉模型）：
  你给它一段文字 "这是一只猫" → 它不知道这段文字在说什么，它只能处理像素。
  你告诉它"这张图里有猫" → 它只能输出"第 281 类（猫）"，无法解释为什么。
```

多模态模型打破了这种模态的隔离。它不是在"视觉模型旁边挂一个文本模型"，而是在训练阶段就让模型的内部神经元同时接受多种模态的刺激：

```
多模态模型（如 Gemini 2.0 Flash）：
  输入：一段文字 + 一张图片 + 可以选择性地加一段视频
  输出：分析报告 + 修改建议
  
  GPT-4V（GPT-4 with Vision）：
  输入：一张复杂的金融图表截图
  输出：图表中每个数据点的数值 + 趋势分析 + 调色建议
```

多模态训练的本质是在模型内部建立"跨模态的语义连接"。当模型在训练中同时看到"苹果"这个词和苹果的图片时，负责处理"苹果"文字 token 的神经元和负责处理苹果视觉特征的神经元会在反向传播中同时被激活，两者之间的连接权重被加强。经过足够多这样的跨模态共现，模型形成了一种内部"翻译"能力——看到图片能联想到对应的文字描述，看到文字能"想象"出对应的视觉特征。这和人类的学习过程高度类似：小孩学习"苹果"这个词时，通常是一边看到苹果的实物、一边听到大人说"苹果"。

### 2.2 SheLook 中三种多模态用法

SheLook 巧妙地利用了三种方向的多模态转换来覆盖商品视觉运营的不同环节：

| 模态组合 | 使用模型 | 在 SheLook 中的用途 |
|----------|----------|-------------------|
| 文→图 | FLUX.2 Pro / Gemini Flash Image | AI 生图：输入商品风格描述，输出高质量商品图 |
| 图+文→文 | Gemini 2.0 Flash | AI 审核：输入商品图+审核标准，输出多维度评分报告 |
| 图+文→视频 | Kling AI 3.0 / Runway Gen-4.5 | 视频生成：输入商品图+运动描述，输出展示视频 |

这三种方向的难度是递增的。文→图是从一个低维度的条件（一段文字，通常几十到几百个 token）生成一个高维度的输出（一张 1024×1024 的图片，约 300 万像素值）。这本质上是"从低信息密度到高信息密度"的生成——文字描述永远是稀疏的，图片永远是密集的。所以文生图模型必须学会"脑补"大量文字中没有明确描述的信息（"白色连衣裙"四个字没有描述裙子的褶皱细节、面料光泽、背景光影）。这是它难的地方。

图+文→文相对简单——从高信息密度到低信息密度，模型需要做的是总结和判断而不是"脑补"。这是大模型擅长的方向：给你大量信息（图片+审核标准），让你提炼出核心判断（评分+诊断）。

图+文→视频是最难的方向——因为视频是在图片的基础上增加了时间维度。一张 1024×1024 的图片是约 300 万像素值，一段 5 秒 30fps 的视频是 150 帧 × 300 万 = 约 4.5 亿像素值。生成的计算量直接翻了 150 倍。这就是为什么生视频需要 1-10 分钟而生成一张图只需要 5-60 秒。

## 三、文生图：从文字到像素的魔法

这是 SheLook 最核心的外部 AI 能力。所有运营流程的起点——"生成商品图"——都依赖文生图模型。用户输入商品信息和风格描述，系统输出可用的商品图片。

### 3.1 扩散模型原理：用"直觉"理解

当前所有主流的文生图模型（FLUX、Stable Diffusion、DALL·E、Midjourney）都基于**扩散模型（Diffusion Model）**。扩散模型的数学推导需要随机微分方程和马尔可夫链的知识，但它的核心直觉可以在几分钟内讲清楚。

想象你有一张清晰的商品图，你开始不断往上面撒椒盐（加噪点）：

```
t=0:  清晰的照片    ████████████████  （你可以看清每一个纽扣和每一道褶皱）
t=250: 轻微噪点     ████▓▓████▓██████  （还能看出连衣裙的轮廓和颜色）
t=500: 中等噪点     █▓█▓▓▓█▓▓▓█▓▓█▓▓  （只能看出大概是个服装类的物品）
t=750: 重度噪点     ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  （几乎看不清，只知道有一个蓝白相间的区域）
t=1000: 纯噪点      ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  （完全随机的噪声，和电视雪花一样）
```

这个过程是固定的——只要你知道原图，按照一个标准的高斯分布公式逐步加噪，t=1000 时一定是纯噪声。这不是在"破坏"，而是一个可逆的数学过程。

扩散模型的训练目标就是学习这个过程的反向——**如何从 t=1000 回到 t=0**。给定一张纯噪点图，再给一个文字条件作为"引导信号"，模型学习预测"往哪个方向调整像素值可以让图片看起来更像文字描述的东西"。

推理时的实际流程：

```
1. 从纯随机噪声开始（t=1000）
2. 输入文字提示："白色背景的红色连衣裙，韩式简约风格，平铺展示"
3. 模型根据当前噪声图和文字条件，预测去噪的方向
4. 在预测的方向上迈一小步：t=1000 → t=980（实际使用加速算法，不需要真的 1000 步）
5. 重复步骤 3-4，经过约 20-50 步
6. 输出清晰的商品图（t=0）
```

**关键直觉（这个理解比公式重要）**：扩散模型不是在"凭空创造"一张图，而是在"学习从噪点中恢复信号"。文字提示词的作用不是"构建"图片的每一部分，而是"引导"恢复的方向——告诉模型"往'白色背景''红色连衣裙'的方向去噪"。同样的初始噪点图，配上不同的提示词，会恢复出完全不同的图片。这在数学上等价于：模型学到了一个从噪声空间到图片空间的条件映射函数，这个函数以文字为条件。

为什么叫"扩散模型"？因为训练数据中模拟的加噪过程在数学上正是物理学中的扩散过程（比如一滴墨水滴入清水逐渐扩散均匀）。AI 研究者借用了物理学中已经研究了上百年的扩散方程来描述这个过程，所以模型被命名为"扩散模型"。

一个容易被忽略的细节：去噪是一个随机过程。即使使用完全相同的初始噪声、完全相同的提示词、完全相同的随机种子，不同的扩散步数配置（比如 20 步 vs 50 步）也会产生不同的结果。这是因为每一步的去噪方向预测都有一定的随机性（除非在推理时固定随机数生成器的种子）——扩散模型在理论上是一个随机微分方程的求解过程，而每次求解都有微小的数值差异。

### 3.2 为什么有这么多模型可选？

市面上有几十种文生图模型，SheLook 为什么选了 FLUX.2 Pro 作为主力、Gemini Flash Image 作为促销图通道、SD WebUI 作为备份？

**模型的差异不在于"谁的技术更先进"，而在于不同的工程取舍适应不同的使用场景。**

| 模型 | 类型 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|----------|
| FLUX.2 Pro | 开源扩散模型（Replicate 托管） | 质量最高、细节丰富、风格控制精准、支持高级参数（CFG scale、步数） | 慢（30-60秒）、贵（按秒计费）、需要联网 | 商品主图、需要极高质量的场景 |
| Gemini Flash Image | Google 多模态大模型 | 快（5-10秒）、理解复杂自然语言指令、可以一次生成多张变体 | 风格控制不如扩散模型精确、细节还原度略低 | 促销图、Banner、快速迭代试稿 |
| SD WebUI | 本地部署 Stable Diffusion | 完全离线可用、数据不出境、可以加载自定义模型权重和 LoRA | 需要本地 GPU、基础模型效果不如云端最新版本、需要自行维护 | 断网备用、数据安全要求高的场景、高度定制化的微调需求 |

这些差异不是偶然的，而是由底层技术架构决定的。FLUX.2 Pro 是专门的扩散模型，经过了从架构到训练策略的全链路优化，唯一目标是生成高质量图片——就像一个只会做一道菜但做到极致的大厨。Gemini Flash Image 是 Google 多模态大模型的一个子功能——它同时能做对话、翻译、编程、看图理解，生图只是它众多能力中的一个。大而全的模型在单一任务上往往不如专精模型表现好，这是模型容量分配的必然结果（一个千亿参数模型的大部分容量被用于理解语言、推理逻辑、记忆知识，只有一小部分用于图像生成）。

SD WebUI 作为备份通道的意义在于"最坏情况的保障"。当 Replicate 和 Google 同时不可用（虽然概率极小但并非不可能——比如某个基础网络设施故障影响了多条国际线路），系统至少还能用本地 GPU 生成一张"能看"的图。虽然质量可能不如云端，但这比"系统无响应"好太多。

### 3.3 SheLook 的多通道降级策略

这是整个生图系统最精妙的设计，也是真正优秀的系统区别于 Demo 的关键。Demo 级别的系统会说"我们接入了 FLUX API"，生产级别的系统会说"我们设计了三级降级，任何一级故障都有备份"。

```
用户发起生图请求
    │
    ├── 品类是 promo/banner？
    │     YES → 优先走 Gemini Flash Image
    │            （促销图需要速度：快速迭代、文字信息多、细节要求低）
    │     NO  → 优先走 Replicate FLUX.2 Pro
    │            （主图需要质量：一张主图用几个月，质量就是转化率）
    │
    ▼
优先通道尝试
    │
    ├── 成功 → 返回结果，流程结束 ✓
    │
    └── 失败（可能原因：API Key 未配置、网络超时、服务商返回空响应、配额耗尽、账单欠费）
          │
          ▼
      降级通道 1：SD WebUI（本地部署，Docker Compose 内）
          │     不依赖任何外部网络，只要服务器有电就能运行
          │
          ├── 成功 → 返回结果，附带降级标记 ✓
          │          （标记用于后续分析——如果降级频率异常高，说明优先通道有问题）
          │
          └── 失败 → 本地 GPU 挂了（驱动问题、显存溢出、模型文件损坏）
                │
                ▼
            终极降级判断：
                │
                ├── 开发环境(APP_ENV=development) → 返回 Mock 占位图
                │    标注 "Development mock - image generation unavailable"
                │    用途：前端开发不依赖生图 API，可以独立调试 UI 流程
                │
                └── 生产环境(APP_ENV=production) → 抛出 GenerationUnavailableError
                    任务状态标记为 FAILED，通知用户稍后重试
                    永远不会在生产环境返回 Mock 图
```

**为什么不在生产环境用 Mock？** 这是一个容易被忽视但后果严重的决策。如果生产环境返回 Mock 图，运营人员可能在不知情的情况下把一张写着"Development mock"的灰色占位图发布到 Amazon 商品后台。等消费者看到这张图，品牌形象和信任度会瞬间崩塌。相比之下，"系统暂时不可用，请稍后重试"虽然体验不好，但不会造成实际损害。

生产环境的错误处理哲学是：**宁可明确失败，也不静默降级。** 失败的请求会留下日志、触发告警、可以被重试机制自动恢复。而静默返回一个不正确的结果（如 Mock 图），错误会潜伏在系统中，等到被发现时可能已经造成了不可逆的损失。

### 3.4 品类路由：不同品类为什么用不同模型

```python
# 这个配置决定了一张促销图走 Gemini 还是走 FLUX
CATEGORY_MODEL_MAP = {
    "promo":  ("google", "gemini-3.1-flash-image"),   # 折扣标签、满减券、秒杀海报
    "banner": ("google", "gemini-3.1-flash-image"),    # 店铺首页横幅、活动头图
}
# 其他所有品类（dress, tops, shoes, accessories...）
# → ("replicate", "black-forest-labs/flux-2-pro")
```

这个路由决策背后有一套逻辑：

促销图（promo/banner）的特点是文字信息量极高——价格数字、折扣百分比、促销文案、"限时抢购"标签。这些文字需要精确呈现（"5 折"不能变成"50 折"），而扩散模型的文本渲染能力相对较弱——它本质上是学"去噪"，对精确字符的控制能力不如大模型（Gemini 在理解文字结构和排版意图方面更强，因为它的多模态训练数据中包含了大量带文字的图片）。

促销图的生命周期极短——一个"618 大促"的促销图通常只使用 3-7 天，之后就被替换。这意味着单张图的"使用价值"不如商品主图高。商品主图可能用几个月甚至几年，一张主图的生成质量直接影响所有见过这张图的消费者的转化决策。

促销图对细节的要求相对低——消费者看促销图主要关注"打折多少""什么时候结束"，不会像看商品主图那样放大观察面料纹理和缝线细节。所以用 Gemini 的速度优势来换 FLUX 的质量优势，在这个场景下是合理的取舍。

商品主图（dress/tops/shoes 等）的特点截然相反：文字信息极少甚至没有（Amazon 要求主图不能包含促销文字）、生命周期长（一张好主图可能用一年）、细节质量直接影响转化率（面料的质感、褶皱的自然度、光影的层次感直接影响消费者的购买信心）。这些特点决定了商品主图必须走质量最高的通道。

这个路由设计的本质是**经济学思维**：把有限的资源（API 调用费用、用户的等待时间）分配到价值最高的环节。促销图用便宜的、快速的方式生成（Gemini），把昂贵的、高质量的生成能力（FLUX）留给真正需要它的商品主图。如果一个系统对所有的图都一视同仁地使用最高质量通道，那它在成本上是不可持续的——就像送一张外卖也用劳斯莱斯。

## 四、AI 自动审核：用大模型做"裁判"

### 4.1 为什么有了 CLIP 质检还需要 Gemini 审核？

第一章介绍的 CLIP + 像素分析已经做了 L1 合规（格式、尺寸）、L2 技术质量（清晰度、光影、色彩）、L3 审美初筛（九维审美评分）。为什么在质检之后还需要 Gemini 再做一轮审核？

**因为 CLIP 懂的是"特征"，Gemini 懂的是"判断"。** 这是两者本质上的能力差异。

CLIP 能告诉你"这张图和'好构图'的文本向量有多接近"——它输出一个相似度数字，但它不知道这个数字意味着什么，不知道这个数字在电商的上下文中应该怎么解读，更不知道"这张图作为 Amazon 上的连衣裙主图会有吸引力吗"。CLIP 做的是模式匹配，不是决策判断。

Gemini 不同。它是一个能推理的大模型。给它一张图片和一段审核指令，它能做的是：理解这张图是一张电商商品图（有品类、有用途、有目标受众），理解审核指令中的每一个维度在电商上下文中意味着什么（"商品可见性"不是简单的画面占比，而是"消费者能不能从这张图了解商品的形状、材质和穿着效果"），综合考虑多个维度给出一致性的整体判断（不能说清晰度 90 分但审美 30 分就算通过），并且用自然语言解释评分的原因和改进建议。

CLIP 的质检回答了"这张图在数学上有什么特征"。Gemini 的审核回答了"这张图作为电商主图够不够好"。前者适合自动化流水线——速度快、成本低、吞吐量大；后者适合关键决策节点——信息密度高、需要综合判断。

这里有一个重要的性能参考数据。根据 2025 年发表的 SIGIR（信息检索顶会）研究，Gemini/Vision 级别的大型多模态模型与人工专业评审在审美评分上的 Spearman 秩相关系数达到了 0.85 以上。Spearman 秩相关系数衡量的是排序的一致性——0.85 意味着 Gemini 给出的评分排序，有 85% 的排序位置和人类评审是一致的。这个水平已经超过了"仅供参"的阈值，可以作为自动化决策的有效输入。

更重要的是，Gemini 的审核结果**可追溯**。它不仅给出一个分数，还给出了每个维度的具体评分、严重程度标注、文字诊断理由和改进建议。如果运营对 AI 给出的分数不满意，可以定位到具体的维度和理由进行质疑——而不是面对一个不可解释的黑盒分数。

### 4.2 Gemini 审核的完整工作流

Gemini 审核不是"扔一张图过去等分数回来"这么简单。Prompt 是它的核心——prompt 的质量直接决定了审核的质量。

```
输入给 Gemini 的内容：
  1. 系统角色设定（System Prompt）：
     "You are an expert e-commerce image quality auditor with 10 years of experience..."
     这告诉模型以什么"身份"和"专业水平"来处理后续请求
     
  2. 图片的二进制数据：
     直接嵌入请求中，Gemini 的多模态编码器同时处理图片和文字
     
  3. 审核指令（User Prompt）：
     "Evaluate this product image for the following 7 dimensions..."
     明确定义了7个评估维度、每个维度的评分标准、
     要求返回严格的 JSON 格式（便于后端程序化处理）

Gemini 内部的执行过程：
  Step 1: 视觉编码器将图片压缩为一系列视觉 token
          这些 token 的粒度比 CLIP 的 512 维向量细致得多——
          它不是一张图一个向量，而是保留了空间结构的 token 网格
          
  Step 2: 文本编码器将审核指令+系统角色设定编码为文本 token 序列
          角色设定在前，审核指令在后，形成"我是专家 → 这是任务"的逻辑结构
          
  Step 3: 多模态 Transformer 同时处理视觉 token 和文本 token
          Transformer 的自注意力机制让视觉 token 可以"读"到文本指令、
          文本 token 也可以"看"到图片的视觉特征
          ← 这就是原生多模态的核心能力：图文融合推理，不是先看图再读文字
          
  Step 4: 自回归生成 JSON 响应
          模型逐个 token 地输出结构化的审核报告
          生成时受到 prompt 中 JSON 格式约束的"压力"——
          模型倾向于生成符合之前在 prompt 中看到的格式

最终输出示例（经过解析后）：

### 4.3 三段阈值的决策矩阵

```
综合评分 (overall_score)

  0 ───────────────── 40 ────────────────── 70 ─────────── 100
  │                   │                      │               │
  │    自动驳回        │     人工复审          │   自动通过     │
  │    (< 40)         │     [40, 70)         │   (>= 70)     │
  │                   │                      │               │
  │  严重质量问题：     │  有瑕疵但可能可用：    │  质量合格：     │
  │  · 图片模糊        │  · 轻微偏色           │  · 可直接发布   │
  │  · 严重过曝        │  · 构图略有瑕疵        │  · 不需要人工   │
  │  · 商品被大面积遮挡  │  · 光影不够均匀     │    干预        │
  │  · 完全不符合       │  · 某个维度偏弱       │               │
  │    品牌调性        │    但其他维度优秀      │               │
```

为什么是 40 和 70 这两个阈值，而不是 30 和 80 或者 50 和 90？

这两条线对应的是决策质量的**错误成本不对称**。

低于 40 分的图片，通过的概率极小。如果让这类图进入人工审核，Reviewer 大概率会驳回——浪费了一个人的审核时间。自动驳回的成本是零（没有伤害），误驳回的成本是"可能错过一张还行的图"（但这种概率远低于 5%）。收益是"节省了所有 reviewer 的时间"。这是一个收益远大于成本的选择。

高于 70 分的图片，被驳回的概率极小。如果让这类图进入人工审核，Reviewer 大概率会点"通过"——同样是浪费了人工。自动通过的成本是"可能放行一张其实有问题的图"（概率很低，可能 10% 左右），但收益是"让 reviewer 专注于真正需要判断力的事"。这也是一个合理的取舍。

40-70 这个中间地带，是 AI 没有把握的区域。AI 觉得"可能还行但也可能有隐藏问题"——这时候需要人类的判断力。人类在这类模糊判断上比 AI 强——人类能感知到 AI 盲区中的因素（比如"这个风格符不符合老板的口味"这种无法量化的考量）。

这个三段设计的隐含前提是：**AI 在两端判断准确、中间不确定**。这符合前文提到的 Spearman 0.85 的数据——排序是准确的（极好和极差分别清楚），但具体的分值判断有噪音（中间地带模糊）。

### 4.4 Gemini 不可用时的降级

在任何依赖外部服务的系统中，"服务不可用"不是一个"如果"，而是一个"什么时候"的问题。Gemini API 可能因为各种原因不可用：Google Cloud 网络故障、API 配额耗尽（月初重置前的最后几个小时）、API Key 过期、Google 紧急维护。

当 Gemini API 不可用时，系统自动回退到启发式评分：

```
启发式评分 = 基础分(50) + 分辨率评分(0~30) + 宽高比评分(0~20) + 文件大小评分(0~20)

详细规则：
  - 图片分辨率最低边 >= 800px → 得满分 30
    (800px 是大多数电商平台的最低要求，低于 800px 按比例扣分)
  - 宽高比接近 1:1 → 得满分 20
    (偏离越多扣越多：1:1.2 扣 5 分，1:1.5 扣 10 分，1:2 以上扣 15 分)
  - 图片文件大小 >= 100KB → 得满分 20
    (文件过小说明图片被过度压缩，缺少细节)
```

这个降级方案在代码注释中自嘲为"先这样，后面有更好的降级方案再改"。它显然远不如 Gemini 的 7 维度审核精细——它只看三个可以客观测量的数值指标，不看任何审美相关的内容。但它体现了一个深刻的工程原则：**永远不要让用户看到"系统错误"。宁可给一个粗糙但有参考价值的评估，也不要让用户在黑盒中等一个永远不会来的结果。**

这三个粗暴的数值至少能过滤掉最糟糕的情况——一张 200×150 的模糊小图，或者一张极度拉伸变形的图。对于剩下的"还行"的图，启发式评分统统给 70-90 分（蒙混过关），让它们进入人工审核环节。虽然这意味着 Gemini 不可用时人工审核的工作量会增加（因为更多图进入 40-70 区间），但至少系统仍然在运转——用户不会收到"服务不可用"的提示，只是审核需要多等一会儿。

这个降级方案还有一个隐含的设计意图：它提醒开发团队"这个方案还不够好"。如果哪天有人想把这段"临时降级"变成永久方案，代码中清楚的 TODO 注释会提示后来的维护者——"这里是应该被替换的部分"。

## 五、图生视频：给静止的商品"注入生命"

### 5.1 图生视频的技术原理

图生视频是 2024-2025 年 AI 领域发展最快的方向之一，也是 SheLook 最新的功能模块。它的商业价值非常直接：Amazon 和 Shopee 等平台越来越多地支持商品视频，带视频的商品平均转化率比纯图片高 20-40%。但传统视频制作需要专业团队、灯光、模特、剪辑，成本动辄几百到几千美元。图生视频让这个过程变成了"上传一张图，等几分钟，拿到一段视频"。

图生视频的技术本质是把 2D 扩散模型的思想扩展到 3D（空间+时间）。这个过程分为三步：

**第一步：3D-VAE 时空编码。** VAE（Variational Autoencoder，变分自编码器）的作用是把高维的原始数据压缩为一个低维的"潜表示"。2D VAE 把一张 1024×1024 的图片压缩为一个 64×64 的潜变量网格。3D VAE 增加了时间维度——它把一段视频（比如 5 秒 30fps = 150 帧）压缩为一个时空潜变量张量（比如 16×64×64，其中 16 是压缩后的时间步数）。对于图生视频，输入端只有一帧（一张静态图），3D-VAE 需要"想象"这一帧在时间上展开会是什么样——它在编码时把这一帧在时间维度上复制，同时加入可学习的"运动先验"。

**第二步：时空扩散（Diffusion Transformer, DiT）。** 这不是原始的扩散模型，而是扩散模型和 Transformer 的结合。DiT 在 3D-VAE 产生的时空潜变量上执行扩散过程——从纯噪声开始，以文字条件和起始帧为引导，逐步去噪生成中间各帧的潜表示。文字条件在这里起到"导演"的作用——"360 度旋转展示"告诉模型每个时间步的潜表示应该让人感觉相机在绕物体转圈；"风吹裙摆"告诉模型下摆区域的潜表示应该有周期性的波动。

**第三步：3D-VAE 解码为视频帧。** 去噪完成后，时空潜变量通过 3D-VAE 的解码器还原为实际的视频帧。150 帧按时间顺序排列，就是一段 5 秒的视频。

为什么图生视频比图生图片慢 100-300 倍？因为时空潜变量的维度远大于 2D 潜变量（2D 是 H×W，3D 是 T×H×W，多了一个时间维度），扩散过程需要在更多维度上迭代去噪。再加上 3D-VAE 的编解码本身也比 2D VAE 更昂贵——所以一段 5 秒视频需要 1-10 分钟来生成。

### 5.2 Kling vs Runway：两条完全不同的技术路线

Kling AI 和 Runway 代表了图生视频领域两个不同方向的技术积累。SheLook 选择 Kling 作为主力、Runway 作为备份。

| 维度 | Kling AI 3.0 | Runway Gen-4.5 |
|------|-------------|----------------|
| 开发者 | 快手（中国互联网公司） | Runway（美国 AI 初创公司） |
| 技术路线 | 自研 3D-VAE + DiT，针对视频生成做了从零开始的架构设计 | 自研 Gen 系列模型，在图像生成基础上向视频扩展 |
| 中文理解 | 原生支持中文 prompt，"国潮""新中式""简约韩风"等本土化风格表达理解精准 | 需要英文 prompt，中文需翻译为英文后再输入 |
| 商品展示优化 | 专门针对"产品 360 度旋转展示"优化，物体运动轨迹稳定 | 更擅长创意和艺术类视频，某些场景下物体可能轻微变形 |
| API 认证方式 | API Key（国内版）或国际版 JWT Token | Bearer Token（标准 OAuth 方式） |
| 价格 | 较低（国内定价，面向中国开发者） | 较高（美国市场定价） |
| 生成速度 | 较快（针对商品展示场景做了推理优化） | 中等（通用的视频生成流程） |

SheLook 的选择逻辑很直接：Kling 在商品展示这个垂直场景上效果更好（快手的电商直播生态让它有大量商品旋转展示视频的训练数据），价格更低（国内市场的竞争导致价格优于美国同行），中文 prompt 理解更好（SheLook 的运营人员主要使用中文描述风格）。Runway 作为备选，覆盖 Kling 不可用的极端情况（比如快手法务调整 API 政策、国际版访问受限）。

### 5.3 异步轮询：等待的艺术

图生视频和图生图片最大的工程差异在于：不可能同步等待视频生成完成。

```
生图：
  提交生成请求 → 等待 5-60 秒 → 拿到图片 → 返回给用户
  可以同步等待，因为 60 秒在 HTTP 请求的超时范围内

生视频：
  提交生成请求 → 等待 1-10 分钟 → 拿到视频 → 返回给用户
  绝对不可能同步等待——HTTP 请求在 2 分钟不响应就会超时断开
```

SheLook 的异步轮询方案：

```
1. 前端发起 POST /api/video/generate
2. 后端提交任务到 Kling API → 拿到 task_id
3. 后端在数据库中创建一条 video_generation 记录，状态 = "pending"
4. 后端立即返回 {"task_id": "xxx", "status": "pending"} 给前端
   （这一步通常在 200ms 内完成——快的让用户感觉是"秒提交"）

5. 后端 Celery Worker 启动轮询任务：
   while status == "pending" or "processing":
       每 10 秒请求一次 GET /videos/image2video/{task_id}
          ├── 状态 = "completed" → 下载视频文件 → 存入 MinIO
          │                          → 更新数据库状态、视频 URL
          │                          → Redis Pub/Sub 通知前端
          ├── 状态 = "failed"    → 更新数据库状态、错误信息
          │                          → 通知前端
          └── 状态 = "pending"   → 继续等待，不通知

6. 最多轮询 30 次（Kling）或 60 次（Runway）
   （对应 5 分钟和 10 分钟的最大等待时间）
   超时后 → 状态保持 "pending"（不报错，让用户稍后手动查看）
```

轮询次数的上限是经过计算的。如果等了 5 分钟还没生成完，大概率是 API 端出了问题（任务被卡住了、队列积压严重），继续等下去也没有意义。但也不报错——因为任务其实可能还在处理中（只是特别慢），报错会让用户以为任务失败了而重新提交，造成重复任务。

前端接收到任务状态 = "pending" 后，显示一个进度条/等待动画。当 Redis Pub/Sub 推送"任务完成"消息时，前端自动刷新页面显示新生成的视频。用户的使用体验是：提交 → 看到进度条 → 去做别的事 → 几分钟后自动看到视频就绪的通知。这个体验比"卡在加载界面 5 分钟"好太多。

## 六、C2PA：给 AI 内容做"防伪认证"

### 6.1 为什么需要 C2PA？

这不是一个"Nice to Have"的锦上添花，而是 2024 年之后做 AI 生成内容的合规刚需。

2024 年 EU AI Act（欧盟人工智能法案）正式生效，2025 年多个国家跟进。核心要求之一：**AI 生成的内容必须有清晰、可验证的来源标识。** 违反规定的罚金最高可达全球年营收的 6%——对于任何服务欧洲市场的电商 SaaS 来说，这不是一个可以轻视的风险。

但 C2PA 不只是为了合规。它解决的是一个更加根本的信任问题。当消费者在 Amazon 上看到一张模特穿着连衣裙的美图，他们心里会产生一个疑问："这个模特真的穿了这件裙子吗？还是这是 AI 做出来的假图？"如果消费者发现收到的实物和图片差距巨大（因为图片是 AI 生成的、实物做不出同样的效果），他们不会骂 AI 模型——他们会给这个商品的卖家打差评、要求退款、甚至向平台投诉。

C2PA 的价值在于让这种"真假判断"变得可验证、可追溯。它不是阻止卖家使用 AI 生成图（这无可厚非——AI 生成图可以让商品展示更美观，消费者也是受益者），而是让消费者、平台审核方、监管机构有能力验证"这张图是否标注了 AI 生成"。透明的 AI 使用比隐藏的 AI 使用更有利于建立长期信任。

### 6.2 技术实现：在图片里嵌入"出生证明"

C2PA（Coalition for Content Provenance and Authenticity）由 Adobe、Microsoft、Intel、BBC、ARM 等发起的开放技术标准组织制定。它的工作方式很直接：

```
1. AI 生图引擎生成原始图片的二进制数据（JPEG/PNG bytes）

2. SheLook 构建一个 Manifest（内容清单），包含：
   {
     "assertions": [
       {
         "type": "c2pa.ai_generated",        ← 标记此为 AI 生成内容
         "model": "black-forest-labs/flux-2-pro",  ← 用的什么模型
         "provider": "replicate",            ← 生成服务商
         "prompt_hash": "sha256:e3b0c44298fc1c14...", ← 提示词哈希
         "timestamp": "2026-07-19T10:30:00Z" ← 生成时间（UTC）
       }
     ],
     "signature": "MEUCIQDx..."  ← 用 SheLook 的证书私钥对清单做数字签名
   }

3. 将 Manifest 和签名嵌入图片文件的元数据区域
   JPEG → 写入 XMP metadata header
   PNG  → 写入 iTXt chunk
   嵌入后的图片 = 原始图片数据 + Manifest 元数据（约 2-5KB）
   
4. 签名后的图片上传到 MinIO 存储
```

签名的核心价值在于**不可篡改性**。任何人都可以验证这张图片的 C2PA 签名是否有效——如果图片被任何编辑软件修改过（哪怕只改了 1 个像素），签名就会失效。因为数字签名是图片像素数据的密码学哈希加上 SheLook 私钥加密的结果——像素变了，哈希就变了，签名就匹配不上了。

### 6.3 为什么只存 prompt 的哈希

这是一个经过深思熟虑的隐私保护设计。Prompt 是运营人员输入的原始描述文字，它可能包含敏感的商业信息：

```
"模仿 ZARA 2025 Summer 最新款的碎花连衣裙设计，
  领口参考 H&M 春季款的 V 领设计，
  目标定价 $39.99，面向美国 18-35 岁女性"
```

这段 prompt 包含了竞争对手分析、产品定价策略、目标人群画像——如果在 C2PA 元数据中明文存储，任何人拿到这张图片就能读到这些信息，等于把商业策略在公开渠道曝光。

SHA-256 哈希解决了这个问题：

- **不可逆**：从哈希值无法反推出原始 prompt。SHA-256 的输出是 64 个十六进制字符，对应 256 位二进制。即使你用全球所有计算机从宇宙大爆炸开始做暴力破解，也试不完 2^256（约 10^77）种可能性。
- **可验证**：任何知道原始 prompt 的人（比如监管机构在调查时）可以把 prompt 做一次 SHA-256 哈希，和 C2PA 元数据中存储的哈希比对——如果一致，说明"这张图确实是用这个 prompt 生成的"。
- **防篡改**：如果有人修改了 prompt 的任何一个字符（比如把 $39.99 改成 $29.99），哈希值会完全不同，比对就会失败。

### 6.4 用户在 SheLook 中的体验

前端图片详情页的 C2PA 溯源面板让这个技术变得可见、可理解：

```
点击图片详情 → "C2PA 溯源" 标签页

┌───────────────────────────────────────────┐

## 七、大模型在 SheLook 中的"全家福"

总结一下，SheLook 的 AI 层由 **4 个外部大模型 API + 1 个本地模型** 组成，每一层各司其职：

```
                         SheLook 系统
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
   【生图工人】            【审核裁判】           【视频导演】
        │                     │                     │
   ┌────┴────┐          Gemini 2.0 Flash      ┌────┴────┐
   │         │          "这张图能打几分？"       │         │
  FLUX.2   Gemini       7维度综合评审          Kling   Runway
   Pro      Flash           +诊断建议          AI 3.0  Gen-4.5
   │         Image          +改进方向           │         │
   │         │                                  │         │
 主力生图  促销图                             主力视频  备选视频
 (质量优先) (速度优先)                        (成本优先) (国际备份)
```

### 技术路线全景对比

| 模型 | 核心技术 | 训练数据 | 推理方式 | 在 SheLook 中的位置 |
|------|----------|----------|----------|-------------------|
| FLUX.2 Pro | Diffusion Transformer | 图文对→去噪训练 | 迭代去噪，20-50 步 | 生图主力（质量最高）|
| Gemini Flash | Multimodal MoE | 图文+音视频+代码联合训练 | 自回归 Token 生成 | 生图（促销）+审核 |
| Kling AI 3.0 | 3D-VAE + DiT | 视频片段→时空去噪训练 | 时空迭代去噪 | 视频主力（成本最优）|
| Runway Gen-4.5 | 自研 Gen 系列 | 视频数据→时空扩散 | 时空迭代去噪 | 视频备选（国际覆盖）|
| CLIP ViT-B/32 | ViT + Transformer | 4 亿图文对→对比学习 | 单次前向传播，200ms | 本地感知底座 |

五种截然不同的技术路线，在 SheLook 中被编排成一个协同工作的整体。这个"编排"层正是 SheLook 作为产品的核心价值——不是训练更好的模型，而是让不同代际、不同架构、不同供应商的模型在一个统一的业务流中高效协作。

## 八、SheLook 的 "AI 编排层" 设计哲学

### 8.1 "永远有退路" 的韧性设计

这是整个编排层最重要的工程原则：

```
生图：   FLUX.2 Pro → Gemini Flash Image → SD WebUI → Mock（仅开发环境）
视频：   Kling AI → Runway → 显式不可用（返回 "service_unavailable"）
审核：   Gemini Flash → 启发式降级 → 人工兜底

不存在单点故障：
  - Replicate 挂了 → Gemini 顶上（可能质量略降，但可用）
  - Gemini 也挂了 → 本地 SD 顶上（完全离线，只要服务器有电）
  - AI 审核不可用 → 启发式规则兜底（粗糙但能过滤最差的图）
  - 只能人工 → 标记为需要人工审核（不影响其他系统功能）
```

这个设计不是为了"炫技"，而是来自对跨境电商运营场景的深刻理解：**跨境电商是 7×24 小时不间断运营的。** 北京的运营下班了，美国的消费者还在浏览商品；新加坡的团队周日休息，但 Shopee 上的商品并不会在周日下架。任何一个外部服务在凌晨 3 点宕机，都可能影响正在运行的批量生图任务——如果系统没有降级方案，第二天早上运营打开后台可能会看到一堆"生成失败"的记录。

多通道降级确保"最坏情况下系统仍然在运转"——哪怕质量打折扣，也比完全瘫痪好 100 倍。

### 8.2 "云端干重活、本地干轻活" 的分工哲学

```
外部 API 承担的任务（重活）：
  · 生图：30-60 秒 / 张，GPU 峰值算力密集
  · 生视频：5-10 分钟 / 段，GPU 持续高负载
  · AI 审核：复杂推理（看 7 个维度 + 写诊断报告），2-5 秒 / 张

本地模型承担的任务（轻活）：
  · CLIP 编码：200ms / 张，CPU 可以满足
  · 质量初筛：100ms / 张，传统像素分析
  · CTR 预测：<10ms / 张，GBDT 在 CPU 上极快
```

这个分工不是随意划分的，背后是精确的成本计算。如果 CLIP 编码也调云端 API：以图搜图每次要调 API（增加 200-500ms 网络延迟 + API 调用费用），每天可能数千次调用——一年下来的费用足够买好几台 GPU 服务器。如果把生图放本地：需要一张 A100/H100 GPU（约 1-3 万美元）+ 持续的电力和冷却成本——对于中小团队来说不现实。

"云端干重活、本地干轻活"的本质是：**把需要稀缺且昂贵资源（高端 GPU）的任务外包给专业的、已实现规模经济的外部服务商；把本地硬件就能搞定的任务留在本地，享受零延迟和零边际成本。**

### 8.3 "模型可替换" 的架构抽象

这是良好软件工程在大模型应用中的体现：

```
今天：FLUX.2 Pro 是生图主力
明天：Ideogram 4.0 据说效果更好 → 在 generation_service.py 中
       把优先级列表的第一个从 "replicate/flux-2-pro" 改为 "ideogram/v4"
       其他所有调用方（API 路由、Celery 任务、前端）完全不受影响

下周：可图（Kolors，快手开源）发布了 → 部署到本地 SD WebUI 的位置
      在降级通道列表中增加 "local_kolors"

替换一个模型不影响上游的原因是：
  上游调用方只知道 generation_service.generate(prompt, style, category)
  它们不知道也不关心 prompt 最终发给 Replicate 还是 Gemini 还是本地 SD
  
  这是依赖倒置原则 (Dependency Inversion) 在 AI 系统中的体现：
  高层模块（业务逻辑）不依赖低层模块（具体模型 API），
  两者都依赖抽象（GenerationService 接口）
```

这个抽象的价值在"多通道降级"中尤为明显：降级逻辑的变更（增加通道、调整优先级、修改超时时间）只需要在 GenerationService 内部完成，消费方（scheme_service、experiment_service 等）一行代码都不需要改。

## 九、多模态在 SheLook 中的深层价值

### 9.1 多模态不是备选

有些人可能会问：能不能不用多模态大模型，只用 CLIP + 规则引擎来做审核？

从技术上是可能的——CLIP 做每个维度的零样本评分（类似 L3 审美评分），然后制定规则（"所有维度平均分 < 40 就驳回"）。但这样做会失去大模型最重要的能力——综合上下文做判断。CLIP 不知道"这 7 个维度的相对重要性取决于图片的使用场景"，Gemini 知道（"如果是促销图，审美比品牌一致性更重要；如果是品牌主图，品牌一致性是最高优先级"）。

多模态大模型在这里不是"锦上添花"，而是让审核从"机械打分"变成"有上下文理解的综合判断"。

### 9.2 In-Context Learning 的业务价值

前文提到 In-Context Learning（ICL）是大模型的核心特征。在 SheLook 中，ICL 的业务价值体现在审核标准的灵活性上。

电商的视觉审核标准不是一成不变的。Amazon 每年更新图片规范，季节性的审美趋势在变化，新市场的开拓带来新的合规要求（比如进入中东市场需要增加"模特着装规范"审核维度）。如果审核是用传统监督学习实现的（收集各种类型的图→人工标注→训练分类器），每次标准变化都需要重新标注数据、重新训练、部署新模型——周期以周或月计。

有了 ICL，更新审核标准就是"修改 Prompt 中的一段文字"。五分钟完成，不需要重新训练，不需要新数据。当法务说"中东市场需要检查模特是否佩戴了宗教敏感的配饰"时，开发者在 Gemini 审核的 Prompt 中增加一条描述即可——"Check if the model is wearing any accessory that may contain religious symbols inappropriate for Middle Eastern markets"。

## 十、本章小结

1. **"大模型"的三个"大"维度——参数大（涌现能力）、数据大（训练成本千万美元级）、能力大（In-Context Learning）——共同构成了它区别于传统 ML 模型的本质特征。** SheLook 选择调用 API 而非自训练，是经济理性的必然而非技术能力的局限。

2. **多模态让模型打破信息形式的壁垒。** SheLook 利用文→图的扩散模型做高质量商品图生成、图+文→文的多模态推理做智能审核、图+文→视频的时空扩散做商品展示视频，三种方向覆盖了完整的"生成→审核→展示"链路。

3. **扩散模型的本质是"学去噪"而非"创造"。** 从纯噪点出发，以文字为引导信号逐步恢复信号，最终生成清晰的图片。扩散步数直接影响生成质量和速度的权衡——多数生产环境用 20-50 步的加速算法。

4. **多通道降级（FLUX → Gemini → SD → Mock/报错）保证系统韧性。** 任何外部服务的单点故障都有降级通道兜底，生产环境绝不用 Mock 图。品类路由根据场景特性选择最优通道——促销图用快速的 Gemini，主图用高质量的 FLUX。

5. **Gemini 审核补充了 CLIP 质检的盲区。** CLIP 做模式匹配（"这图和谐吗"），Gemini 做综合判断（"作为电商主图够好吗"）。三段阈值 40/70 的设计体现了"AI 在两端准、中间不确定"的客观认知。

6. **图生视频是 2D 扩散向 3D 时空的扩展。** 3D-VAE 编码时空潜变量，DiT 在时空维度上做去噪扩散，计算量比生图高 100-300 倍。Kling 在商品展示场景优化更好、价格更低，作为主力；Runway 作为国际备份。

7. **C2PA 是合规刚需而非可选。** EU AI Act 生效后，AI 生成内容必须有来源标识。Prompt 只存 SHA-256 哈希既保护了商业秘密，又保留了可验证性。

8. **AI 编排层的核心哲学：永远有退路（多通道）、云端重本地轻（经济理性）、模型可替换（架构抽象）。** SheLook 的产品价值不在于自己训练更好的模型，而在于将不同代际、不同供应商的 AI 能力高效编排进电商视觉决策的完整闭环。

下一章预告：**模型训练与数据飞轮**——SheLook 如何从历史数据中自动学习，实现"越用越聪明"的正向循环：GBDT 预测模型训练、分位数自动标注、数据回流机制、租户隔离下的个性化训练策略。
