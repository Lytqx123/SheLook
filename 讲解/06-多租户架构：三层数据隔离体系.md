## 一、多租户是什么？为什么电商 SaaS 必须多租户？

### 1.1 单租户 vs 多租户：一碗饭一个人吃 vs 大锅饭

多租户（Multi-Tenancy）是一种软件架构，让一个应用实例同时服务多个客户（租户），每个租户的数据完全隔离，互相不可见。

```
单租户架构：
  ├── 租户A：独立部署一套 SheLook（独立的服务器、独立的数据库、独立的配置）
  ├── 租户B：独立部署一套 SheLook
  └── 租户C：独立部署一套 SheLook
  每次新增一个客户 = 重新部署一整套系统
  运维成本 × N（N 个租户 = N 倍的服务器、监控、升级、备份工作量）
  资源利用率极低（每个租户独占服务器但平均负载可能只有 10%）

多租户架构：
  ├── 一套 SheLook 实例（一台服务器、一个数据库、一套代码）
  ├── 租户A 的数据 ←→ 租户B 看不到（逻辑隔离但不物理隔离）
  ├── 租户B 的数据 ←→ 租户C 看不到
  └── 租户C 的数据 ←→ 租户A 看不到
  每次新增一个客户 = 数据库插入一条 tenants 记录
  运维成本固定（1 套系统，不管 10 个还是 100 个租户）
  资源池化（CPU 和内存被所有租户共享，平均利用率高）
```

对于 SheLook 这种面向中小型电商团队的 SaaS 产品来说，多租户不是"可选的设计模式"，而是"必须的商业模式"。单租户模式下，服务一个年付费 $500 的小团队需要一台独立的服务器（成本 $50-200/月）——这个账根本算不过来。多租户模式下，100 个年付费 $500 的团队可以跑在同一套基础设施上，服务器成本分摊到每个租户只要 $2/月。这就是 SaaS 商业模型的经济学基础：**通过资源池化实现规模化效应。**

但"大锅饭"有一个致命的隐患：**数据隔离。** 如果租户 A 的连衣裙上架数据不小心泄露给了租户 B（一个竞争品牌），后果不是技术事故，是商业灾难。SheLook 的选择是——不在物理上分库（那样就无法享受资源池化的成本优势），而是在逻辑上用三道防线来保证"逻辑上的数据隔离和物理上的分库一样可靠"。

### 1.2 三种多租户隔离方案：从物理到逻辑的光谱

隔离方案是一个连续的光谱，不是一个二选一的"分"或"不分"：

| 方案 | 实现方式 | 隔离强度 | 资源效率 | 运维复杂度 |
|------|----------|----------|----------|-----------|
| **数据库级** | 每个租户独立数据库 | 最强（物理隔离） | 最低（N 个数据库 = N 份存储开销） | 最高（N 套迁移脚本、N 套备份、跨租户查询需要 federated query） |
| **Schema 级** | 每个租户独立 Schema（PostgreSQL 的 namespace） | 较强（逻辑隔离但共享数据库进程） | 中等（共享数据库进程但独立的表空间） | 中等（Schema 多时迁移变慢，连接池管理复杂） |
| **行级** | 每个表加 tenant_id 列，查询时加 WHERE 过滤 | 较弱（完全依赖代码正确性） | 最高（一个数据库、一套表） | 最低（标准的 SQL，不需要特殊运维） |

SheLook 选择了**行级方案 + 多层防御**。

选择行级不是因为它"隔离最强"（恰恰相反，它在三种方案中隔离最弱），而是因为它在隔离强度和资源效率之间取得了最优的折中。关键洞察：**"隔离强度不够"可以通过增加防御层次来弥补，而"资源效率不够"没有弥补的办法。** 行级方案的弱点是纯粹依赖代码的正确性——如果你忘了在查询中加 `WHERE tenant_id = ?`，数据就会泄露。但 SheLook 通过三层防御（应用层自动绑定 + ORM 层自动注入 + 数据库层强制过滤）把这个弱点变成了"几乎不可能发生"。

数据库级方案在隔离强度上是无可挑剔的，但它在业务灵活性上有致命的缺陷。假设全局数据飞轮需要分析"在所有租户的连衣裙品类中，什么样的视觉特征和 CTR 相关"——如果每个租户独立数据库，你需要查询 300 个数据库的 results 然后在应用层汇总。而行级方案只需要一条 SQL：`SELECT * FROM images WHERE category = 'dress'`（当然这条 SQL 本身会带上租户过滤——但在管理后台可以用 `skip_tenant_scope` 来绕过租户过滤做跨租户的全局分析）。

Schema 级方案的问题在于运维复杂性。PostgreSQL 的 Schema 是逻辑命名空间，迁移时需要遍历所有 Schema 逐个执行——如果 300 个租户每个一个 Schema，数据库迁移变成了 300 次 DDL 操作。

## 二、SheLook 的三层隔离体系：防御的纵深

```
层 1: 应用层 —— ContextVar 绑定
  │   每个 HTTP 请求和 Celery 任务都有自己的 TenantContext
  │   "我是谁"——当前请求属于哪个租户
  │
  ▼
层 2: ORM 层 —— SQLAlchemy 事件自动注入
  │   所有 SELECT 自动追加 WHERE tenant_id = :current_tenant
  │   所有 INSERT 自动填充 tenant_id，拒绝写入其他租户
  │   "我查询时自动帮你过滤"——开发者不需要手动写 tenant_id
  │
  ▼
层 3: 数据库层 —— PostgreSQL Row-Level Security
     每个表启用 RLS 策略，即便有人绕过 ORM 直连数据库，
     也只能看到当前会话的租户数据
     "即便是裸 SQL 也绕不过我"——数据库引擎级别强制
```

三层从高到低，防御的"强制力"从弱到强。层 1 和层 2 是"守规矩的人自动不会出错"——只要代码按常规方式写（使用 ORM、走 HTTP 中间件），就不会出问题。层 3 是"就算有人不守规矩也翻不了天"——即使用 raw SQL 绕过 ORM、即使用数据库管理工具直接连接，RLS 仍然在强制租户过滤。

这是"纵深防御"（Defense in Depth）的思想——不信任任何单一防线能永远不失效，用多道防线互相兜底。在安全工程中，任何单一防线都有失效的概率——可能是一个代码 bug、一个配置错误、或者运维人员的失误。但如果三道防线独立、失效模式不同，同时失效的概率是三个小概率的乘积——从"可能发生"变成"几乎不可能发生"。

## 三、层 1：应用层的 TenantContext —— "我是谁"的统一入口

### 3.1 ContextVar：异步 Python 中实现租户隔离的标准做法

Python 的 `ContextVar` 是 Python 3.7 引入的一个关键特性，专门解决异步编程中的"全局变量隔离"问题。在传统的同步编程中，你可以用 `threading.local()` 为每个线程创建独立存储空间。但在异步编程中，多个协程在同一个线程中交替执行——如果协程 A 设置了 `threading.local().tenant_id = "northstar"`，然后协程 B 在同一个线程中执行时读了 `.tenant_id`，读到的就是 "northstar"（而不是 B 自己的上下文）——这就是协程之间的上下文泄漏。

`ContextVar` 解决的就是这个问题：它为每个"执行上下文"（可以是一个协程，也可以是一个线程）提供独立的变量副本，自动在上下文切换时保存和恢复。

```python
# tenant.py 的核心

tenant_context_var: ContextVar[TenantContext | None] = ContextVar(
    "tenant_context", default=None
)
```

这行代码的行为：

```
FastAPI 请求处理器中的协程 A（租户=northstar）：
    │  中间件调用 set_tenant_context("northstar")
    │  ContextVar 在当前协程的上下文中 = "northstar"
    │
    ▼  async def get_products():
           tenant = get_current_tenant_id()  → "northstar"
           products = await db.query(...)    → ORM 自动过滤 northstar 的数据
           return products
       # 协程 A 挂起（await db.query），CPU 让给协程 B

FastAPI 请求处理器中的协程 B（租户=atelier）：
    │  中间件调用 set_tenant_context("atelier")
    │  ContextVar 在当前协程的上下文中 = "atelier"
    │
    ▼  async def get_products():
           tenant = get_current_tenant_id()  → "atelier"  ← 正确！没有泄漏！
           # ContextVar 自动恢复了协程 B 的上下文
```

`ContextVar` 的底层实现依赖 CPython 解释器在协程切换（`await` 点）时自动保存和恢复上下文。这个特性在 Python 3.7+ 被正式保证，SheLook 使用的 Python 3.13 完全依赖这个保证。

### 3.2 三种设置上下文的方式：不同入口，同一个目标

**方式 1：HTTP 中间件（来源 = request）**

中间件是 FastAPI 的请求处理管道中的第一站。它在请求到达业务逻辑之前被调用，设置租户上下文；在业务逻辑处理完毕、响应返回之后清理上下文。

这个设计的精妙之处在于：**业务代码完全不需要感知"租户上下文"的存在。** 路由处理器中的任何 `await db.query(...)` 都会自动带上当前请求的租户过滤，因为 ORM 层从 `ContextVar` 中读取了 tenant_id。开发者只需要关心业务逻辑，不需要在每个查询中手动传入 tenant_id。

**方式 2：Celery 任务（来源 = task）**

Celery Worker 是独立进程，不经过 FastAPI 中间件。所以后台任务的租户上下文必须显式绑定。使用 Python 的 `with` 上下文管理器，语法上优雅且强制——进入 `with` 块时设置 ContextVar，退出时自动恢复。

```python
with tenant_context("northstar", source="task"):
    process_product_images()
    generate_ctr_predictions()
    # 所有操作在 "northstar" 的上下文中
# 退出 with 块 → 自动恢复为调用前的 ContextVar 值
```

这种模式的好处是：如果开发者忘了包装 `with tenant_context(...)`，`ContextVar` 取到的是 `None`，然后层 1 的默认回落机制会触发——数据不会泄漏，但会记录一条警告日志，方便代码审查时发现遗漏。

**方式 3：默认回落（来源 = default）**

当没有前面两种方式设置上下文时（如直接运行的脚本、测试代码、命令行工具），`get_tenant_context()` 返回一个带有特殊标识的默认上下文。这不是安全漏洞——这些场景通常操作的是全局数据或测试数据，不涉及租户隔离。这个默认回落只是防止 `None` 导致代码崩溃。

### 3.3 为什么 ContextVar 是层 1 的最佳选择？

在之前，常用的做法是：
- 把 tenant_id 作为参数传递给每个函数（如 `get_products(tenant_id="northstar")`）——繁琐、易忘、污染函数签名。每个函数都要多一个参数，调用链的每一层都要传递它。
- 把 tenant_id 放在请求对象中（如 `request.state.tenant_id`）——在异步环境中 `request.state` 可能被不同协程共享导致混乱。

`ContextVar` 是 Python 标准库专门为这个场景设计的工具——全局可访问（方便）、协程安全（不会泄漏）、自动清理（`with` 块退出时恢复）。它让租户上下文成为"隐式参数"——不需要在函数签名中显式传递，但始终可被所有层访问。

## 四、层 2：ORM 层的自动租户注入 —— 让开发者"忘记"租户的存在

### 4.1 SQLAlchemy 事件系统：在不修改业务代码的前提下注入行为

SQLAlchemy 的事件系统是 ORM 框架中最强大的扩展机制之一。它允许你在 ORM 操作的各个生命周期节点注册回调函数，在框架执行某个操作的前后"插入"你自己的逻辑。SheLook 利用了三个生命周期事件：

```
do_orm_execute     → 每次执行 SQL 查询前触发（包括 SELECT、INSERT、UPDATE、DELETE）
                    在这个节点，框架还没有真正把查询发给数据库，
                    你可以在最后一刻修改查询语句

before_flush       → 每次将内存中的对象变更写入数据库前触发
                    在这个节点，所有 session.add()/session.delete() 已经执行，
                    但还没有生成实际的 SQL INSERT/DELETE 语句
    
after_begin        → 每次数据库事务开始时触发
                    在这个节点，新的数据库事务刚刚创建，
                    你可以设置事务级的配置变量
```

这三个事件分别拦截了 ORM 操作的不同阶段：读操作（do_orm_execute 拦截 SELECT）、写操作（before_flush 拦截 INSERT/UPDATE）、事务管理（after_begin 设置 RLS 变量）。每个节点都自动读取 ContextVar 中的当前租户，施加相应的隔离措施。

### 4.2 SELECT 自动注入：你写的查询会自动加上租户过滤

这是整个 ORM 层最关键的一个事件：

```python
@event.listens_for(Session, "do_orm_execute")
def _apply_tenant_scope(execute_state) -> None:
    # 三个跳过条件：开关关闭 / 非 SELECT / 显式跳过
    if (not settings.TENANT_ENFORCEMENT_ENABLED
        or not execute_state.is_select
        or execute_state.execution_options.get("skip_tenant_scope")):
        return

    tenant_id = get_current_tenant_id()
    execute_state.statement = execute_state.statement.options(
        with_loader_criteria(
            TenantScopedMixin,
            lambda entity: entity.tenant_id == tenant_id,
            include_aliases=True,
        )
    )
```

它做了什么——用最终的 SQL 来看：

```
开发者写的业务代码（不需要任何 tenant_id）：
  products = session.scalars(
    select(Product).where(Product.status == "active")
  ).all()

生成的 SQL（框架自动注入后）：
  SELECT products.id, products.name, products.tenant_id, ...
  FROM products
  WHERE products.tenant_id = 'northstar'  ← 框架自动追加的
    AND products.status = 'active'        ← 开发者写的

同样，对于 JOIN 查询：
  SELECT products.*, images.*
  FROM products
  JOIN images ON images.product_id = products.id
  WHERE products.tenant_id = 'northstar'  ← 自动追加
    AND images.tenant_id = 'northstar'     ← 自动追加（include_aliases=True）
    AND products.status = 'active'
```

`with_loader_criteria` 是 SQLAlchemy 提供的一个强大工具——它会在 ORM 生成的查询中自动为所有继承了 `TenantScopedMixin` 的模型追加额外的 WHERE 条件。`include_aliases=True` 确保 JOIN 中的别名表也会被过滤。这个机制的关键价值在于：**它让开发者永远不需要在业务代码中写 `tenant_id = ?`。** 只要模型继承了 `TenantScopedMixin`，租户过滤就是自动的、强制的、不可忘记的。

**三个跳过条件的含义：**

1. `TENANT_ENFORCEMENT_ENABLED = False`：全局开关，关闭所有租户过滤（仅在单元测试和开发环境中使用）。测试环境需要自由插入和查询任意租户的数据来验证功能。

2. `execute_state.is_select = False`：INSERT/UPDATE/DELETE 不需要在这里处理——SELECT 需要追加过滤条件（"只读我的数据"），但写操作需要的是赋值和校验（"只创建我的数据"）——这个逻辑在 `before_flush` 中处理。

3. `skip_tenant_scope = True`：最强大的跳过机制——允许特定的查询绕过租户过滤。只有管理员级别的操作（如全局仪表盘、跨租户数据汇总分析）才使用这个选项。它的使用需要在代码中显式声明 `execution_options(skip_tenant_scope=True)`，代码审查时容易发现和审计。

### 4.3 INSERT 自动填充与拒绝：两条防线防写泄漏

```python
@event.listens_for(Session, "before_flush")
def _assign_and_validate_tenant(session, _flush_context, _instances):
    tenant_id = get_current_tenant_id()
    for instance in session.new:
        if not isinstance(instance, TenantScopedMixin):
            continue  # 跳过非租户范围的表（如 tenants 表本身）

        existing_tenant_id = getattr(instance, "tenant_id", None)
        if existing_tenant_id and existing_tenant_id != tenant_id:
            raise ValueError("禁止在当前租户上下文中创建其他租户的数据")
        instance.tenant_id = tenant_id  # 自动赋值
```

写操作的两步处理：

```
步骤 1：自动赋值（你有权在我这里创建数据）
  product = Product(name="连衣裙")  ← 开发者忘记设置 tenant_id
  session.add(product)
  # before_flush 触发 →
  #   existing_tenant_id = None → 不进入拒绝逻辑
  #   instance.tenant_id = "northstar"（自动填入当前请求的租户 ID）

步骤 2：拒绝跨租户写入（你不能在我这里创建别人的数据）
  product = Product(name="连衣裙", tenant_id="competitor")
  # 恶意代码或 bug 试图创建其他租户的数据
  session.add(product)
  # before_flush 触发 →
  #   existing_tenant_id = "competitor"
  #   "competitor" != "northstar"（当前 ContextVar 中的租户）
  #   → raise ValueError → 事务回滚 → 写入拒绝
```

两步分别防御了不同的威胁模型。自动赋值防御的是"开发者的疏忽"——在数百个 API 路由中，总有人某天忘记设置 tenant_id。拒绝逻辑防御的是"恶意攻击"或者"严重逻辑错误"——比如一个 API 接收了来自客户端的 tenant_id 参数，如果代码不小心把它赋给了新创建的 Product 对象。

### 4.4 TenantScopedMixin：标记需要隔离的表

```python
class TenantScopedMixin:
    tenant_id: Mapped[str] = mapped_column(
        String(36),
        ForeignKey("tenants.id", ondelete="RESTRICT"),
        nullable=False,
        index=True,
        comment="所属企业租户",
    )
```

通过继承这个 Mixin 来标记：

```
Image(Base, TenantScopedMixin)       ← 图片按租户隔离
Product(Base, TenantScopedMixin)     ← 商品按租户隔离
Experiment(Base, TenantScopedMixin)  ← 实验按租户隔离
Scheme(Base, TenantScopedMixin)      ← 视觉方案按租户隔离
...

Tenant(Base)                         ← 租户表自身不需要隔离
TenantQuota(Base)                    ← 配额表自身不需要隔离
OrganizationUnit(Base, TenantScopedMixin) ← 组织架构按租户隔离
```

`ForeignKey("tenants.id", ondelete="RESTRICT")` 有双重保护：
- **引用完整性**：任何被 TenantScopedMixin 标记的表的 tenant_id 列，都必须是 tenants 表中真实存在的租户 UUID。你不可能因为一个拼写错误而把数据"写入一个不存在的租户"。
- **RESTRICT 删除**：如果有人在 tenants 表中试图删除一个还有业务数据的租户，PostgreSQL 会拒绝执行（因为有外键引用）。防止误删租户导致所有关联数据成为"孤儿"。

`index=True` 看似不起眼，但对性能至关重要。每一个查询都会以 `WHERE tenant_id = ?` 开头，没有索引时查询计划会变成全表扫描——对于一个有 100 个租户、每个租户有几万条记录的表来说，这会导致数倍的查询变慢。索引让"按租户过滤"始终走 B-tree 索引的 O(log N) 查找，而不是 O(N) 的全表扫描。

## 五、层 3：PostgreSQL Row-Level Security —— 数据库引擎的"最终防线"

### 5.1 RLS 是什么：内核级别的强制过滤

PostgreSQL 的行级安全（Row-Level Security, RLS）是一个从 9.5 版本开始支持的内核级特性。它的工作机制是在数据库引擎内部——不是应用层、不是 ORM 层、不是连接池层：

```
原理：

ALTER TABLE products ENABLE ROW LEVEL SECURITY;
-- 启用 RLS 后，默认行为是"拒绝所有访问"（DENY ALL）

CREATE POLICY tenant_isolation ON products
  USING (tenant_id = current_setting('app.tenant_id'));
-- 创建策略：允许访问那些 tenant_id 等于会话变量的行

之后，任何对 products 表的查询，PostgreSQL 内部自动追加：
  WHERE tenant_id = current_setting('app.tenant_id')

这发生在查询规划器和执行器之间——不是在 SQL 文本层面修改，
而是在查询执行时由数据库引擎自动施加过滤条件。
```

这里有一个关键细节：RLS 的过滤发生在**查询执行阶段**，而不是查询解析阶段。这意味着即使有人拿到了数据库连接字符串，通过 psql 或任何数据库管理工具直接执行 `SELECT * FROM products`，RLS 仍然会生效——在数据库引擎返回任何一行之前，它会先检查"当前会话的 `app.tenant_id` 是什么，只返回匹配的行"。

当然，RLS 完全依赖 `current_setting('app.tenant_id')` 这个会话变量的值。如果攻击者拿到数据库连接但不知道或没有设置正确的 `app.tenant_id`，有两种可能：
- 如果 `app.tenant_id` 不存在 → RLS 策略中的 `USING` 条件返回 NULL（不是 TRUE 也不是 FALSE，而是 NULL）→ PostgreSQL 默认不返回任何行 → 攻击者看到空数据。
- 如果攻击者手动设置 `SET app.tenant_id = 'northstar'` 并猜对了某个租户的 ID → 可以看到该租户的数据。但租户 ID 是 UUID（2^128 种可能），暴力猜测在计算上不可行。

### 5.2 事务级变量绑定：每次事务开始时设置，结束时自动清理

RLS 依赖 `current_setting('app.tenant_id')` 变量，这个变量需要有人在事务开始时设置。SheLook 在 `after_begin` 事件中做了这件事：

```python
@event.listens_for(Session, "after_begin")
def _apply_tenant_rls_setting(session, _transaction, connection):
    if not settings.TENANT_RLS_ENABLED or _database_backend != "postgresql":
        return
    connection.execute(
        text("SELECT set_config('app.tenant_id', :tenant_id, true)"),
        {"tenant_id": get_current_tenant_id()},
    )
```

`set_config` 的第三个参数 `true` 是"事务级"的标志。在这个事务中，`app.tenant_id` 对当前会话可见，但事务 COMMIT 或 ROLLBACK 后自动清除。这是事务隔离性和连接池复用配合工作的关键：同一个数据库连接在处理完"northstar"的事务后回到连接池，下一个租户的事务获取了这个连接——如果 `app.tenant_id` 没有被清除，"atelier"的事务就可能看到"northstar"的数据。事务级变量自动清除了这个风险。

```
连接池中的一个数据库连接的生命周期：

事务 1（租户=northstar）：
  BEGIN
  SET app.tenant_id = 'northstar'（事务级，仅在此事务内可见）
  SELECT * FROM products → RLS 强制：WHERE tenant_id = 'northstar'
  COMMIT → app.tenant_id 自动清除 ✓

事务 2（租户=atelier）：
  BEGIN
  SET app.tenant_id = 'atelier'（事务级）
  SELECT * FROM products → RLS 强制：WHERE tenant_id = 'atelier' ✓
  COMMIT → app.tenant_id 自动清除

"atelier"的事务永远不会看到之前"northstar"的 app.tenant_id——
因为事务级变量在 COMMIT 时已自动清除。
```

### 5.3 为什么既用 ORM 事件又用 RLS？两道锁同时上双重保险

这是"纵深防御"思想的具象化。ORM 事件（层 2）和 RLS（层 3）提供了两套独立的、不同实现机制的租户过滤。它们不是互相替代的关系，而是互相兜底的关系。

```
如果 ORM 事件（层 2）失效——例如：
  - 未来某人重构 SQLAlchemy 事件注册逻辑时遗漏了 do_orm_execute
  - 某人使用了 raw SQL (session.execute(text("SELECT ...")))
    text() 不走 ORM 的 execute state，do_orm_execute 不会触发
  → RLS（层 3）仍然生效：PostgreSQL 照样追加 WHERE tenant_id = ...

如果 RLS（层 3）失效——例如：
  - 运维误操作：ALTER TABLE products DISABLE ROW LEVEL SECURITY
  - 数据库版本升级后 RLS 策略被清空
  - PostgreSQL 的 "app.tenant_id" 变量未设置（SQLAlchemy after_begin 事件未触发）
  → ORM 事件（层 2）仍然生效：with_loader_criteria 照样加 WHERE 条件

只有两层同时失效（需要两个独立的事故同时发生），才会真正发生跨租户数据泄漏。
```

### 6.1 核心表结构

```

  ├── tenant_id → tenants.id

  ├── tenant_id → tenants.id（一对一）

  ├── generation_concurrency (同时生图任务数上限)
  ├── monthly_generation_limit (每月最多生成多少张图)

```

四张表之间的关联构成了租户数据的完整骨架。`tenant_memberships` 是连接"用户"和"角色"的核心表——用户在认证系统中有自己的身份，但在 SheLook 中的权限取决于它们属于哪个租户、分配了什么角色。同一个用户可以是"northstar"的管理员、同时是"atelier"的查看者——登录 SheLook 时切换租户，权限自动变更。

### 6.2 六种角色：从"什么都能干"到"只能看看"

| 角色 | 权限级别 | 典型用户 | 能做什么 |
|------|---------|---------|---------|
| `admin` | 最高 | 团队主管、老板 | 所有操作 + 管理组织成员 + 修改租户配置 |
| `operator` | 高 | 运营专员 | 生图、审核（L1-L3）、发布、管理方案、跑实验 |
| `reviewer` | 中 | 质检员、设计师 | 审核图片质量、给出专业意见、标记问题 |
| `analyst` | 中低 | 数据分析师 | 查看数据仪表盘、实验报告、效果趋势分析 |
| `supplier` | 低 | 外部供应商、摄影师 | 上传自己负责的图片、查看审核结果 |
| `viewer` | 最低 | 外部合作方、临时访客 | 只读查看（不能修改任何东西） |

前端和后端对角色做了双重权限控制：

```
前端（UI 层）：viewer 看不到"创建实验"按钮
后端（API 层）：viewer 的 POST /api/experiments 请求返回 403

两个层次互相冗余。前端的隐藏是"用户体验"——不让看不到的东西占界面空间。
后端的拒绝是"安全保障"——即使有人绕过了前端直接调 API，也无法越权操作。
```

前端权限是"体验层面"的守门人——它告诉用户"你能做什么"。后端权限是"安全层面"的守门人——它强制执行"你只能做什么"。任何前端权限检查都可以被绕过（浏览器的开发者工具、curl 命令），但后端权限检查是 API 服务器上的代码逻辑，无法绕过。

### 6.3 组织架构树：公司内部的"部门墙"

```
租户 "northstar" 的组织架构：

  ├── 连衣裙组 (team)
  │     └── 运营小张（operator，只看连衣裙组的数据）
  └── 上衣组 (team)
        └── 运营小李（operator，只看上衣组的数据）

男装事业部 (department)
  └── 外套组 (team)
        └── 运营小王（operator，只看外套组的数据）

设计中心 (department)
  └── 拍摄组 (team)
        └── 设计师小赵（reviewer，只看设计中心的数据）
```

组织架构的自引用外键（`parent_id → organization_units.id`）天然支持树状的层级结构。一个用户可以属于多个组织单元（`unit_ids` 是一个 JSON 数组），这对于矩阵式组织（一个人既属于"连衣裙组"又属于"设计中心"）特别重要。

## 七、租户级别的速率限制：既要公平，又要可靠

### 7.1 滑动窗口算法：每一秒都重新计算

```python
# 在 Redis 中使用 Lua 脚本原子执行

算法：滑动窗口（Sliding Window）

窗口大小：60 秒（滚动窗口，不是固定窗口）
数据结构：Redis ZSET（有序集合），score 为时间戳

每次请求时执行（Lua 脚本保证原子性）：
  1. 清理过期数据：
     ZREMRANGEBYSCORE key 0 (now - 60)
     删除 60 秒之外的所有记录
     
  2. 数当前窗口的请求数：
     count = ZCARD key
     即 60 秒内有多少条请求记录
     
  3. 判断：
     if count >= limit:
         return 429 (Too Many Requests)
         
  4. 记录本次请求：
     ZADD key now unique_id
     将当前请求插入 ZSET，记录本次访问
```

为什么用 Lua 脚本？因为这四步操作（清理→计数→判断→记录）必须原子执行。如果分成四个独立的 Redis 命令，两个并发请求可能发生：

```
请求 A（第 59 次请求）：
  读取 count → 看到 59 → 59 < 60（limit）→ 以为可以放行
请求 B（第 60 次请求，与 A 同时到达）：
  读取 count → 也看到 59 → 也以为可以放行
结果：两个请求都被放行了——虽然加起来是 61 次，超过了限额。
```

Lua 脚本让整个流程在 Redis 的单线程事件循环中作为一个原子操作执行——不可分割、不可打断。这就是"为什么用 Redis Lua"的根源：不是因为性能，而是因为原子性。没有原子性，限流在多并发场景下就是漏的。

### 7.2 租户差异化限流

```python
async def _get_request_limit(self, tenant_id: str | None) -> int:
    # 1. 先查缓存（避免每次请求都打数据库）
    cached = await redis.get(f"tenant_quota:rpm:{tenant_id}")
    if cached:
        return int(cached)

    # 2. 缓存未命中 → 查数据库
    quota = await db.get(TenantQuota, tenant_id)
    limit = quota.api_requests_per_minute if quota else self._max_requests

    # 3. 缓存结果（60 秒过期，配额变更后最多 60 秒生效）
    await redis.set(f"tenant_quota:rpm:{tenant_id}", limit, ex=60)
    return max(1, int(limit))
```

不同套餐的速率限制不同：
- 免费试用租户：60 次/分钟（以防止滥用为主）
- 标准租户：600 次/分钟（满足正常运营高峰需求）
- 企业租户：6000 次/分钟（可处理批量自动化操作）

限流不仅是技术防护（防止单个租户压垮共享基础设施），也是商业模型的体现（更高级套餐 = 更多资源）。

### 7.3 Redis 挂了怎么办？可用性优先

```python
except Exception:
    logger.error("速率限制检查失败，放行请求")
    return False, max_requests  # 放行，不拦截
```

这是"可用性优先于安全性"的经典工程权衡。如果 Redis 挂了，系统有两个选择：

- **拒绝所有请求**（安全性优先）：防止任何限流被绕过，但正常付费用户也被挡在门外。对于 SaaS 产品来说，这意味着停机——所有租户都无法使用系统。
- **放行所有请求**（可用性优先）：限流失效，恶意租户可能滥用资源，但正常用户不受影响。受损的只有"限流保护"这个功能，系统的基本可用性保持。

SheLook 选择了后者。理由有三：第一，Redis 在生产环境中是高可用部署（Docker Compose 中有健康检查 + 自动重启），挂了的情况极为罕见。第二，即使 Redis 真的挂了，后端还有 PostgreSQL 的连接池限制和 Nginx 的反向代理缓冲作为兜底——限流失效不等于系统被攻破。第三，"因为有付费用户正在用系统"比"防止某个租户滥用"优先级更高。

## 八、Celery 后台任务的租户上下文

### 8.1 问题：Celery Worker 没有 HTTP 请求，没有中间件

HTTP 请求通过中间件自动绑定 ContextVar——这个过程是对开发者透明的。但 Celery Worker 是独立进程，和 HTTP 完全无关。Celery 任务没有经过 FastAPI 中间件，ContextVar 始终是初始值（None 或默认值）。如果后台任务中的 ORM 查询读到了错误的 ContextVar，数据就会要么泄漏（如果默认租户能看到所有数据）、要么无法访问（如果默认租户配置为空）。

### 8.2 解决方案：显式绑定

```python
@app.task
def generate_variant_images(image_id: str, tenant_id: str):
    # tenant_id 是在调度任务时就确定了的，随任务参数持久化到 Redis
    with tenant_context(tenant_id, source="task"):
        image = await fetch_image(image_id)        # 自动租户过滤
        prompt = build_prompt(image)
        result = await call_replicate_api(prompt)
        save_result_to_db(result, image_id)         # 自动租户赋值
    # with 块结束时 ContextVar 自动恢复为进入前的值
```

这里有一个容易被忽视的细节：`tenant_id` 是任务调度的参数之一，在调度时就已经从 HTTP 请求的当前租户上下文中确定了。当 FastAPI 的路由处理器调用 `generate_variant_images.delay(image_id, current_tenant_id)` 时，`current_tenant_id` 被序列化到 Celery 的消息体中，存入 Redis Broker。当 Worker 从 Redis 中拉取到这个任务时，它从参数中恢复 `tenant_id`。

### 8.3 为什么需要 `source` 字段？

```
TenantContext(tenant_id="northstar", source="request")   ← HTTP 请求
TenantContext(tenant_id="northstar", source="task")       ← Celery 任务
TenantContext(tenant_id="default", source="default")      ← 默认回落
```

`source` 字段的实用价值在调试和监控场景中体现得最为明显。当安全问题发生时（比如某个操作意外地访问了不该访问的租户数据），审计日志中的 `source` 字段能立即告诉你是哪个入口点出了问题——如果是 `source=task`，问题出在 Celery 任务参数传递或绑定逻辑上；如果是 `source=request`，问题出在 HTTP 中间件或 Token 解析上；如果是 `source=default`，说明某些操作没有正确设置租户上下文就执行了。

## 九、三层隔离的安全性全景分析

### 9.1 攻击场景与防御覆盖

| 攻击场景 | 防御层 | 结果 |
|----------|--------|------|
| 开发者忘记在 SELECT 中加 `WHERE tenant_id` | 层 2：`with_loader_criteria` 自动注入 | 安全 ✓ |
| 开发者忘记在新对象上设置 `tenant_id` | 层 2：`before_flush` 自动赋值 | 安全 ✓ |
| 恶意代码试图创建其他租户的数据 | 层 2：`before_flush` 拒绝跨租户写入 | 安全 ✓ |
| 有人使用 raw SQL 绕过 ORM（`session.execute(text(...))`） | 层 3：PostgreSQL RLS | 安全 ✓ |
| 有人拿到数据库连接密码直连 | 层 3：RLS（前提：`app.tenant_id` 变量未设置 → USING 条件返回 NULL → 无行返回） | 安全 ✓ |
| 有人拿到数据库密码且猜到了某个租户的 UUID | 层 3：RLS 生效（只显示设置的那个租户的数据） | 受限（单租户泄漏） |
| Celery 任务忘记设置 `tenant_context` | 层 1：默认 ContextVar = None → 层 2 和层 3 无法正常工作 | 需代码审查 |
| ContextVar 在协程间泄漏 | Python 3.7+ 运行时保证 | 安全 ✓ |
| Redis 缓存了其他租户的数据（如 API 响应） | 缓存 key 包含 `tenant_id` 前缀 | 安全 ✓ |

### 9.2 三层防御也不能覆盖的场景

1. **管理员使用 `skip_tenant_scope` 做跨租户查询**：这是设计功能，不是漏洞。管理员确实需要跨租户的数据分析能力（如全局仪表盘、所有租户的使用统计）。这个能力通过显式的 `execution_options` 参数控制，每次使用都有完整的审计记录。

2. **数据库管理员直接 psql 登录并手动设置 `app.tenant_id`**：这属于物理安全层面——如果有人拿到了数据库 root 密码，他们可以做任何事。这种场景的防御不依赖于 RLS（RLS 可以轻易被 `ALTER TABLE DISABLE ROW LEVEL SECURITY` 或 `BYPASSRLS` 属性绕过），而是依赖于数据库用户权限管理和密钥轮换。

3. **业务逻辑 Bug**（比如 API 返回了正确的租户数据给请求它的用户，但用户本身不应该看到那个租户的一部分组织单元）：这超出了数据层的范围，需要在 API 层的业务逻辑做额外的权限过滤。

## 十、本章小结

1. **多租户架构是 SheLook SaaS 商业模型的技术基石**——通过资源池化实现规模化效应。行级方案在隔离和效率之间取得了最优折中，而"隔离不够"的弱点通过三层防御被补偿到接近数据库级隔离的水平。

2. **三层防御构成纵深防线**：应用层 ContextVar 提供"我是谁"的协程安全绑定 → ORM 层 SQLAlchemy 事件让开发者零心智负担地享受自动租户过滤 → 数据库层 PostgreSQL RLS 作为最终防线，即使绕过了 ORM 也无法跨租户读取。

3. **ContextVar 是 Python 异步环境中实现租户隔离的最佳工具**——协程安全的全局变量，支持 `with` 块自动清理。HTTP 请求通过中间件透明设置，Celery 任务通过显式 `with tenant_context()` 绑定。

4. **ORM 事件系统实现了"开发者无需关心租户"的理想**：SELECT 自动追加 WHERE、INSERT 自动赋值 tenant_id、跨租户写入被 ValueError 拒绝。`with_loader_criteria` 配合 `include_aliases=True` 覆盖所有 JOIN 场景。

5. **PostgreSQL RLS 在数据库引擎层面强制执行**：通过事务级 `SET app.tenant_id` 变量，RLS 策略自动过滤所有行。即使绕过 ORM 使用 raw SQL，RLS 也会照常生效。

6. **租户管理四张核心表**（tenants → organization_units → tenant_memberships → tenant_quotas）和**六种角色**提供了从组织架构到权限细粒度控制再到资源配额的一整套多租户管理基础设施。

7. **滑动窗口限流**（Redis ZSET + Lua 原子脚本）保证多租户环境中资源的公平分配，租户差异化 RPM 配额体现商业模型的分层。**Redis 故障时宁可放行也不拦截**——可用性优先于安全性。
