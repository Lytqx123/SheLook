## 一、为什么需要 A/B 实验？预测只是"可能"，实验才是"实际"

前面四章讲了系统如何生成图片、评估质量、预测效果。现在有一个关键问题：**凭什么说一张图比另一张好？**

这个问题比它听起来的要深刻。有人会说："预测模型已经告诉我了，A 图 CTR 可能是 3.5%，B 图可能是 3.2%，A 比 B 好，就这么简单。"但这个推理有一个致命的漏洞——**预测不等于事实**。

预测模型告诉你的不是一个确定的事实，而是一个基于历史数据的有根据的猜测。这个猜测可能对也可能错，而判断它对不对的唯一方法是——让真实的消费者来投票。

```
预测模型：A 图 CTR ≈ 3.5%，B 图 CTR ≈ 3.2%  → "A 可能更好"
          结论：有根据的猜测，置信区间 ±20%

A/B 实验：A 图上线 → 1000 次曝光 → 38 次点击 → CTR = 3.8%
          B 图上线 → 1000 次曝光 → 29 次点击 → CTR = 2.9%
          p-value = 0.23 → 差异不显著 → "不能下结论"
          结论：目前数据不足以判断

分析：为什么预测说 A 高 0.3%，实际跑出来差了 0.9% 还是"不显著"？
      因为样本量只有 1000，9 个点击的差异在统计上可能是随机波动。
      A 可能确实更好，也可能只是运气好——1000 次曝光不够判断。
```

预测告诉你往哪个方向试，实验告诉你方向对不对。没有预测，你会在海量图片组合中迷失方向（某个商品有 5 张图，两两比较有 10 种组合，如果有 100 个商品就是 1000 种组合——运营不可能手工挑）。没有实验，你会把预测当成真理，而预测必定有错——错误累积到影响真金白银的转化率时，损失已经不可挽回。

A/B 实验在 SheLook 中之所以不可或缺，还有一个更根本的原因：**图片的视觉效果是高度主观的，而主观的东西不能靠算法来判定对错。** CLIP 的审美评分可以达到和人类评审 0.85 的 Spearman 相关系数——这已经很好了，但它仍然有 15% 的排序和人类不一致。GBDT 的 CTR 预测可以做到 ±20% 的误差——这也很好了，但它仍然告诉不了你"究竟该选 A 还是选 B"。只有实验——让真实的消费者用鼠标投票——才能给出最终答案。

## 二、A/B 实验的基本概念

### 2.1 什么是一个"实验"？

```
一个 A/B 实验包含：

实验 ID：       #42
商品：          连衣裙 SKU-DR-001
版本 A（变体）： 图片 #201（白色背景平铺拍摄）
版本 B（对照）： 图片 #187（模特穿着场景拍摄）
流量分配：      A 获得 50%，B 获得 50%
状态：          运行中
开始时间：      2026-07-12 08:00
```

两个版本同时上线，随机分配流量，比较效果。

实验设计的黄金法则是：**唯一变量是图片，其他条件完全一样。** 同一件商品（不是相似的商品，是同一个 SKU）、同一个价格、同一个上线时间、同一个目标市场。如果在这些条件中混入了其他变量，实验结论就被污染了——你不知道 CTR 的差异是因为图片本身还是因为价格变了、或者是打到了不同的用户群体。

这个"单一变量原则"听起来简单，但在实践中经常被违反。最常见的错误是"用不同商品的图片做比较"——卖家想看"连衣裙的模特图"和"T 恤的生活场景图"哪个表现好，但连衣裙和 T 恤的天然 CTR 基准值就不同（连衣裙整体 CTR 约 2.5%-4.5%，T 恤约 1.5%-3.0%），品类差异远大于图片差异，实验结果毫无意义。SheLook 的前置校验强制了"同一商品"的约束，从根本上杜绝了这种无效比较。

### 2.2 实验的三种状态

```
RUNNING  ──→ COMPLETED   （自然完成：曝光量达到 10000 阈值）
   │
   └──────→ STOPPED      （人工停止：运营觉得可以下结论了）
```

两种结束方式的结论计算是一样的——系统都会计算 CTR 差异、p-value、选出胜者。区别在于触发方式：自然完成是系统认为"样本量够了"；人工停止是运营认为"我已经看明白了，不需要更多数据"。

人工停止的支持是有实际意义的。如果跑了 3000 次曝光后，A 的 CTR 是 6.5%，B 是 1.2%，运营一看——差异太大了，不可能反转，没必要再浪费 7000 次曝光去验证一个已经明显的事实。但系统不会这么想——系统严格按照 10000 次曝光的阈值行事，它不知道"6.5% vs 1.2%"意味着什么，只知道"还没跑到 10000"。"人工停止"给了运营灵活决策的空间，既保留了系统的严谨性（系统默认按规则跑完），又给人类保留了判断力。

### 2.3 创建实验的前置校验

```python
# experiment.py create_experiment() 中的三道校验

1. 不能自己和自己比
   if variant_a == variant_b: → 422 ValidationError

2. 两张图必须属于同一个商品
   if image_products[a] != product_id: → 422
   if image_products[b] != product_id: → 422

3. 商品和图片都必须存在
   if product is None: → 404
   if image_id not in image_products: → 404
```

这三道校验是最低限度的合理性保障。没有它们，系统可能创建一个"图片 A vs 图片 A"的实验（同一张图和自己比，CTR 始终一样，永远得不出结论），或者创建一个"连衣裙的图 vs 运动鞋的图"的实验（品类完全不同，CTR 基准值差 2-3 个百分点，差异来自品类而非图片）。校验 3 是纯粹的数据完整性检查——防止因为图片被删除但方案引用未清理而导致实验引用不存在的图片。

## 三、自动实验创建：系统自己找"值得比"的图片

### 3.1 为什么要自动创建？

运营人员不可能每天手动去挑"哪两张图差不多、值得比一比"。一个中等规模的卖家可能有 500 个在线商品、每个商品 3-5 张候选图片——这加起来是上千种潜在的三组合。人工筛选是根本不可能完成的。

自动创建系统每天凌晨 4:00 执行（在数据回流完成后、模型重训之前），扫描所有已审核通过的图片：

```
扫描所有已审核通过的图片（review_status == AUTO_APPROVED）
  │
  按商品分组
  │
  对每个商品：
    ├── 图片数量 < 2？ → 跳过（没得比）
    ├── 已有运行中的实验？ → 跳过（同一商品只允许一个实验）
    └── 有 >= 2 张图？
          │
          按预测 CTR 从高到低排序
          │
          遍历相邻的两张，找 CTR 差距最小的一对
          │
          差距 <= 0.02（2个百分点）？ → 创建实验
          差距 > 0.02？ → 跳过（差距太大，不值得比）
```

### 3.2 为什么选 CTR 差距最小的两张？——看似正确实则错误的选择

这是整个实验系统最反直觉的设计，也是最精妙的设计。大部分人的第一反应是"应该比最好的和最差的，这样才能看出差距有多大"。但这个想法在统计学上是错的，在业务上也没有意义。

```
场景：商品有三张已审核的图片

图片 A：预测 CTR 5.2%（模特穿着，生活场景，光影精致）
图片 B：预测 CTR 5.0%（模特穿着，白色背景，干净利落）
图片 C：预测 CTR 1.8%（平铺拍摄，光线暗淡，没有擦去背景褶皱）

如果比 A vs C：
  需要的样本量极少就能判断出结果——因为真实差距（预测 3.4%）太大，
  几乎不需要多少曝光就能验证 A >> C。
  实验的价值 = 0（你已经知道结果了，实验只是浪费时间流量）

如果比 A vs B：
  两者 CTR 预测几乎一样（差距 0.2%），预测模型说自己不确定谁更好。
  这是一个真正有悬念的比较——到底是生活场景更有吸引力，还是白底图更清晰？
  实验的价值 = 极高（你在探索一个预测模型答不上来的问题）
```

这里有一个更微妙的统计学原理在起作用。CTR 差距最小的两张图，也是**预测模型最不确定谁更好的**两张图。当两张图的预测 CTR 几乎一样时，说明在模型看到的特征空间中，它们的特征向量非常相似——模型找不到足够的特征差异来做区分。但人类的审美直觉可能会发现模型没看到的差异（比如"虽然构图和色彩差不多，但这张的模特表情更自然"）。实验就是在验证"模型盲区中的差异是否真的影响了消费者的行为"。

另一个角度来看：预测 CTR 最接近的两张图通常也是视觉上最相似的两张图。视觉最相似意味着它们在消费者眼中最可能"看起来差不多"——如果实验结果说"差别不大"，那是符合预期的。但如果实验结果说"A 显著好于 B"，那你就发现了模型没捕捉到的关键差异——这个新知识可以回馈到预测模型中，让它下次更准。而如果比 A vs C（差距大的两张图），即使结果显著，你也没学到有价值的新知识——你只是确认了一个已知事实。

**好的实验不是"验证已知"，而是"发现未知"。**

### 3.3 限制规则

```python
MAX_RUNNING_EXPERIMENTS_PER_PRODUCT = 1  # 每个商品最多 1 个实验
CTR_DIFF_THRESHOLD = 0.02                 # 差距 > 2% 的不比
```

每个商品只允许一个实验的限制是为了保证流量集中。如果同一件连衣裙同时跑了三个实验（A vs B、A vs C、B vs C），每个实验只能获得日常流量的 1/3——每个实验都需要很长时间才能积累足够的样本量，而且消费者可能会在不同实验中看到不同的图，导致混淆（同一个用户可能刷新后看到 A，再刷新看到 C——这不是 A/B 测试，这是在"洗"用户）。

## 四、UCB 算法：实验运行中的流量自动调整

### 4.1 传统 A/B 测试的问题：既要探索，又要利用

固定 50/50 是最简单、最"公平"的实验方式。但公平不等于最优。

```
一个极端的例子来感受"固定 50/50"的浪费：

假设你开了一家餐厅，有两道新菜：
  菜 A：已经被 1000 个顾客尝过，平均评分 4.8 / 5
  菜 B：只被 10 个顾客尝过，平均评分 4.2 / 5

固定 50/50 策略下：你继续把一半的顾客引导到菜 B。
但菜 B 可能真的比菜 A 差（评分低 0.6 不是偶然），那你的顾客满意度就被拖累了。
也可能菜 B 其实更好（只是前 10 个人运气不好，恰好口味偏淡）——你不知道。

最优策略：多给已知好吃的菜 A 流量（利用），但保留 10-20% 给菜 B（探索）。
如果菜 B 后续表现确实差，继续减少其流量到 20%（下限）。
如果菜 B 后续翻盘了（证明它其实更好），自动增加它的流量。
```

这个权衡在数学上被总结为"遗憾最小化"——假设菜 A 确实更好，你多给菜 B 一次曝光就损失了一次菜 A 带来的额外收益。你的目标是最小化这个总的"遗憾"（regret）。UCB 算法在理论上被证明是 regret 增长最慢的算法之一——它的 regret 以对数级增长（O(log N)），而固定 50/50 以线性增长（O(N)）。

### 4.2 UCB 算法的直觉：不只看平均，还看信息量

UCB（Upper Confidence Bound，上置信界）的核心思想是：对每一个版本，计算一个"乐观估计"——不是"当前的平均 CTR 是多少"，而是"如果这个版本真的很好，它的 CTR 最多可能有多高"。

```
核心公式：

UCB(a) = CTR(a) + sqrt(2 × ln(N) / n(a))

其中：
  CTR(a)    = A 的当前 CTR（利用项：已知 A 有多好）
  ln(N)     = 总曝光数的自然对数（总体探索压力）
  n(a)      = A 的曝光数（A 的确定程度）
  
  sqrt(...)  = 探索项（Upper Confidence Bound——"上置信界"的由来）

直觉解读：
  - A 曝光很多（n(a) 大）→ 探索项小 → 我们对 A 的 CTR 估计很确定 → 偏向利用
  - A 曝光很少（n(a) 小）→ 探索项大 → 我们对 A 还不了解 → 多给 A 一些曝光机会
  - 总曝光不多（N 小）→ ln(N) 小 → 所有探索项都小 → 平等对待（数据都不够，别急着偏）
  - 总曝光很多（N 大）→ ln(N) 大 → 但有曝光量的版本探索项仍然小 → 坚定偏向确定的好版本
```

这里有一个关键的数学细节：`ln(N)` 是取对数的，增长非常缓慢。当 N 从 100 增长到 10000（翻了 100 倍），`ln(N)` 从 4.6 增长到 9.2（只翻了 1 倍）。这意味着随着实验进行，探索项的"上界"增长得非常保守——即使你积累了海量数据，探索项也不会变得无穷大，算法对这点的克制保证了最终会收敛。

### 4.3 具体计算示例：为什么一个 CTR 更低但数据更少的版本可能获得更多流量

```
假设实验运行了一段时间：

版本 A：1000 次曝光，35 次点击 → CTR = 3.5%
版本 B：200 次曝光，8 次点击   → CTR = 4.0%

直觉：B 更高（4.0% > 3.5%），应该多用 B。

但等等——B 只有 200 次曝光，8 次点击。有没有可能 B 只是运气好？
毕竟 200 次曝光如果多 2 个点击（6→8），CTR 就从 3% 跳到了 4%。
而 A 的 1000 次曝光中多 2 个点击，CTR 只从 3.3% 变到 3.5%（影响小得多）。

UCB 计算：
  N = 1000 + 200 = 1200，ln(1200) ≈ 7.09

  UCB(A) = 0.035 + sqrt(2 × 7.09 / 1000)
          = 0.035 + sqrt(0.0142)
          = 0.035 + 0.119
          = 0.154

  UCB(B) = 0.040 + sqrt(2 × 7.09 / 200)
          = 0.040 + sqrt(0.0709)
          = 0.040 + 0.266
          = 0.306

  UCB 总量 = 0.154 + 0.306 = 0.46
  A 的新比例 = 0.154 / 0.46 = 33.5%
  B 的新比例 = 0.306 / 0.46 = 66.5%
```

结果是 B 获得 66.5% 的流量（尽管它当前 CTR 4.0% 只比 A 的 3.5% 高 0.5 个百分点）。

这个结论初看可能让人困惑——B 的 CTR 更高，而且 B 的探索项也更大（因为数据少），两者同时指向了"多给 B 流量"。但如果 B 其实是运气好呢？假设 B 的真实 CTR 是 2.5%（前 200 次运气极好），随着 B 的流量从 66.5% 跑下去，B 的 CTR 会逐渐从 4.0% 跌向真相，同时 B 的探索项随着曝光量增加而缩小，两项都下降 → UCB(B) 快速下降。

比如假设 B 的流量跑到 600 次（累计）、点击跑到 18 次（CTR = 3.0%，已经开始跌了）：

```
UCB(B) = 0.030 + sqrt(2 × ln(1800) / 600)
       = 0.030 + sqrt(2 × 7.49 / 600)
       = 0.030 + sqrt(0.025)
       = 0.030 + 0.158
       = 0.188  ← 已经开始逼近 A 的 UCB = 0.154

如果继续跑，B 会降到和 A 持平甚至更低，流量自动回弹。
```

这就是 UCB 的自我纠错机制：它给"可能的黑马"一个公平的机会，但如果黑马经不起数据检验，系统会自动减少其流量。

### 4.4 两阶段策略：先用传统方式建立基准，再切换为 UCB 加速

```python
if total_impressions < 100:
    new_ratio = 0.5    # 探索阶段：数据太少，先 50/50 公平收集
    method = "default_explore"
else:
    new_ratio = ucb_a / (ucb_a + ucb_b)  # 利用阶段：UCB 分配
    new_ratio = clamp(new_ratio, 0.2, 0.8)  # 限制在 20%-80%
    method = "ucb"
```

**探索阶段**（曝光 < 100）：固定 50/50。这是对 UCB 在极低样本量下的不稳定性做的工程保护。在曝光 < 100 时，一个版本可能 10 次曝光 0 次点击（CTR = 0%），另一个版本 10 次曝光 2 次点击（CTR = 20%）。在这个样本量下，CTR 的差异几乎肯定来自随机波动（因为真实 CTR 不可能从 0% 跳到 20% 只因为图片换了），但 UCB 的探索项会极大，导致公式不稳定。100 次曝光的固定 50/50 给两个版本一个公平的起跑机会，积累了最低限度的基准数据后，再让 UCB 接管。

**利用阶段**（曝光 >= 100）：UCB 分配，但限制在 20%-80%。这个限制区间的设计有三个考虑：

1. **防止过早收敛**。如果没有下限，假设 A 的前 200 次表现特别好（运气成分），UCB(A) 远大于 UCB(B)，算法可能在 B 还没来得及证明自己之前就把 B 的流量压缩到 5%——B 永远没有翻身的机会。20% 保底给了 B 持续积累数据的机会。

2. **保持实验的合法性**。A/B 实验的目的不仅仅是"找到最好的图"，更是"量化好多少"。即使 A 明显更好，仍然需要 B 至少有 20% 的流量来保持足够的样本量计算 p-value。如果 B 只有 5% 流量，等 A 跑了 9500 次曝光时 B 只有 500 次——B 的 CTR 方差巨大，算出来的 p-value 不可靠。

3. **防止赢者通吃**。如果某个版本因为外部原因（比如偶然赶上了双十一流量高峰）表现得异常好，UCB 可能会给它极高的流量——而这些流量其实是因为时机而非图片质量。20/80 的 clamp 给系统加了一个安全限。

### 4.5 实验自动结束

```python
if total_impressions >= EXPERIMENT_COMPLETION_IMPRESSIONS:  # 默认 10000
    计算显著性 → 标记 COMPLETED → 停止流量调整
```

为什么选 10000 次曝光作为结束阈值？这不是一个神秘的数字，而是来自统计功效分析：在 CTR 约为 3% 的典型电商场景中，10000 次曝光可以以 80% 的统计功效检测到约 0.5-0.8 个百分点的 CTR 差异（假设 α = 0.05 双侧检验）。如果差异更小（比如 0.3 个百分点），可能需要 30000-50000 次曝光。但 10000 是一个工程上的合理折中——大多数有意义的视觉改进带来的 CTR 提升在 0.8% 以上，而为了检测 0.3% 的微小差异去等待 50000 次曝光对运营来说是不现实的（50000 次曝光在中小卖家的流量规模下可能要跑一个月）。

## 五、统计显著性：差异是真的还是碰巧？

### 5.1 p-value 到底在说什么

p-value 是统计学中最常被误解、最常被滥用的概念。它不是在说"A 比 B 好的概率是 95%"。它说的是一个反事实条件概率：**假设 A 和 B 没有真正的差异（零假设成立），我们有多大可能观测到当前这样的差异（或更极端的差异）？**

```
版本 A：1000 次曝光，38 次点击 → CTR = 3.8%
版本 B：1000 次曝光，29 次点击 → CTR = 2.9%

A 比 B 高了 0.9 个百分点。

零假设（保守假设）：A 和 B 的真实 CTR 完全一样。
问题：在零假设下，观察到 ≥ 0.9% 的差异的概率是多少？
答案：p-value = 0.23

解读：即使 A 和 B 没有真正的差异，我们也有 23% 的概率"碰巧"观察到
      像现在这样 0.9% 的差距。23% 太高了——这意味着"碰巧"是很有可能的，
      我们不能下结论说 A 真的比 B 好。这个差异大概率只是随机波动。
```

0.23 意味着什么？它意味着如果你做了一个硬币实验（零假设是"硬币公平"），抛了 1000 次，正面出现 520 次而不是 500 次——这个偏差是 20 次。20 次的偏差在 1000 次抛硬币中完全正常、经常发生。你不会因为 520 vs 500 就说"这个硬币不公平"。同样，你也不应该因为 38/1000 vs 29/1000（差距只有 9 个点击）就说"A 图真的更好"。差距不够大，样本不够多，随机波动可以轻易解释这个差异。

```
另一次实验（样本量翻倍）：
版本 A：5000 次曝光，195 次点击 → CTR = 3.9%
版本 B：5000 次曝光，150 次点击 → CTR = 3.0%
p-value = 0.01

解读：如果 A 和 B 真实效果一样，只有 1% 的概率"碰巧"观察到 ≥ 0.9% 的差异。
      这种小概率事件不太可能仅仅是"运气"。我们可以说 A 确实比 B 好。
```

同样的差异幅度（0.9 个百分点），随着样本量从 2000（前一个例子）增加到 10000，p-value 从 0.23 降到了 0.01。样本量的价值在这里体现得淋漓尽致——不是让差异变大，而是让差异的信噪比变大。

这里的直觉可以用一个日常类比来理解。如果你发现你朋友的硬币掷 10 次有 7 次正面（70%），你不会觉得硬币有问题——10 次中多 2 次正面太正常了。但如果掷 10000 次有 5200 次正面（52%），你会觉得硬币确实偏了——因为 10000 次中多出 200 次正面的概率微乎其微。统计学不关心"偏差"本身的大小，而是关心"偏差除以噪音"——样本量越大，噪音越小，同样的偏差就越难用"运气"来解释。

传统阈值 p < 0.05 意味着"我们愿意接受最多 5% 的概率错误地认为有差异（第一类错误）"。

### 5.2 Z-test for Proportions 的实现

```python
def calculate_p_value(
    variant_impressions, variant_clicks,
    control_impressions, control_clicks,
):
    p1 = variant_clicks / variant_impressions    # A 的 CTR
    p2 = control_clicks / control_impressions    # B 的 CTR
    
    # 合并比例（在零假设下 A 和 B 的真实 CTR 应该相同）
    p_pool = (variant_clicks + control_clicks) / (variant_impressions + control_impressions)
    
    # 标准误：衡量"如果零假设成立，CTR 差的随机波动有多大"
    se = sqrt(p_pool × (1-p_pool) × (1/n₁ + 1/n₂))
    
    # Z 统计量：差异 / 标准误（差异是"信号"，标准误是"噪音"）
    z = (p1 - p2) / se
    
    # 双边 p-value = 2 × P(Z > |z|)
    # 取自标准正态分布的互补误差函数 erfc
    p_value = erfc(|z| / sqrt(2))
```

计算流程的每一步都有明确的统计含义：

- `p_pool`（合并比例）是零假设成立时所有人的共同 CTR 的最佳估计。它是把两组的点击和曝光加总起来算的"全局平均"。
- `se`（标准误）衡量了在零假设成立、样本量为 n₁ 和 n₂ 时，CTR 差的自然波动幅度。如果样本足够大、足够随机，CTR 差会以 se 为标准差围绕 0 波动。
- `z = (p1-p2)/se` 是"标准化后的差异"——差异是标准误的多少倍。z=1 意味着差异相当于一个标准误，很常见（p≈0.32）。z=2 意味着差异相当于两个标准误，有点异常（p≈0.045）。z=3 意味着差异相当于三个标准误，非常可疑（p≈0.003）。
- `erfc` 是互补误差函数，计算标准正态分布尾部的面积（即概率）。`erfc(|z|/√2)` 就是 P(|Z| > |z|)。双边 p-value 需要乘以 2（因为差异可能在正方向出现，也可能在负方向出现——我们关心"差异有多大"，不关心方向）。

### 5.3 为什么用 Z-test，以及其他方案的对比

| 检验方法 | 适用场景 | 为什么不选 |
|----------|----------|-----------|
| Z-test | 大样本比例比较 | **SheLook 的选择**：CTR 数据天然大样本（1000+ 曝光），中心极限定理保证小比例的样本也能用正态近似 |
| t-test | 小样本均值比较 | CTR 不是均值而是比例（伯努利实验），t-test 假设底层数据是正态分布，CTR 数据是 0/1 分布 |
| Chi-squared | 分类变量独立性 | 和 Z-test 在 2×2 表格（两组、两种结果：点击/未点击）时完全等价（χ² = z²），结果一样，Z-test 更直接 |
| Fisher's Exact | 超小样本精确检验 | 计算量大（需要枚举所有可能的分布表），当 n > 1000 时完全无必要 |
| Bootstrap | 任何场景 | 不需要正态假设，但每次都要重采样 1000+ 次，计算开销大 1000 倍，对于每周运行数百次p-value计算的系统来说不划算 |

## 六、多维度下钻归因：防止"整体好、局部差"的认知陷阱

### 6.1 为什么整体结论不够？

"版本 A 整体 CTR 比版本 B 高 0.9%（p = 0.03）"——这个结论在统计上是成立的，但在业务上存在一个致命的盲区：

```
整体结论：版本 A 比 B 好（CTR +0.9%，p=0.03）✓

下钻到市场维度后：
  美国市场：A 比 B 高 1.5%（p=0.01，高度显著）
  东南亚市场：A 比 B 低 0.3%（p=0.42，不显著但方向反了）
  
如果只看整体结论，你会把 A 推广到所有市场——包括东南亚。
但东南亚的数据说"A 可能不如 B"。

结论应该是：A 在美国市场比 B 好，东南亚市场两者差别不大。
→ 建议：美国主推 A，东南亚保留 B 或单独实验验证。
```

这就是辛普森悖论在 A/B 实验中的实际表现——整体优于局部分裂。之所以出现这种情况，通常是因为两个市场的用户偏好不同（美国用户偏好极简白底，东南亚用户偏好暖色场景），而 A 恰好偏向美国审美。在做整体汇总时，美国市场的流量占比更大（可能 70% vs 30%），A 在美国的显著优势"掩盖"了它在东南亚的微弱劣势。

### 6.2 三种下钻维度

| 维度 | 聚合方式 | 回答的问题 |
|------|---------|-----------|
| `market` | 按市场分组计算 CTR | A 在不同市场表现一样吗？ |
| `category` | 按产品品类分组计算 CTR | A 在不同品类表现一样吗？不同品类用户对图片风格的期望不同 |
| `date` | 按日期分组计算 CTR | A 的优势是持续的还是某天突然好的？某天突然好可能是外部事件而非图片因素 |

### 6.3 下钻的样本量陷阱

每个维度切片独立做 Z-test 时，样本量会显著降低。一个隐藏的陷阱：

```
整体：10000 曝光，p = 0.01（显著）
美国市场下钻：5000 曝光，p 可能变成 0.08（不显著）
东南亚市场下钻：3000 曝光，p = 0.52（完全不显著）

如果运营看了这个结果说："哦，每个市场的 p 都不显著，说明没有差异。"
那就犯了大错——不是差异不存在，而是样本量不够检测到差异。
```

这个陷阱的根源在于：p-value 同时受到效应大小（真实的差异有多大）和样本量（噪音水平）的影响。切片减少了样本量 → 增加了标准误 → z 变小 → p 变大。一个真实存在的差异在切片中可能变成"不显著"，这不代表差异消失了，只代表你的数据不够多。

SheLook 的解决方案：下钻结论前面总是标注样本量。如果某个市场切片只有 500 次曝光，结论写的是"目前样本量不足（500 曝光），建议继续观察"而不是"无显著差异"（后者会误导运营认为"确认了没差别"）。

## 七、Lift 与归因报告：不只是"好不好"，还要问"好多少"

### 7.1 Lift 的含义与解读

```
Lift = (变体 CTR - 对照 CTR) / 对照 CTR × 100%

例如：
  A（变体）CTR = 4.0%，B（对照）CTR = 3.0%
  Lift = (4.0 - 3.0) / 3.0 × 100% = +33.3%
  
解读：使用变体图片后，CTR 相对提升了 33.3%（约三分之一）。
```

Lift 是一个相对量，它的价值在于**跨品类、跨市场做归一化的比较**。CTR 从 1% 提升到 1.5% 和从 10% 提升到 10.5%，绝对差距都是 0.5 个百分点，但前者是 50% 的提升（意义重大），后者是 5% 的提升（聊胜于无）。用 Lift 而不是绝对差异来汇报实验结论，才能让运营在不同的商品之间公平地比较"哪个实验的回报最大"。

### 7.2 归因报告的三级建议

```python
recommendation = (
    "推广此方案"    if lift > 0 and p < 0.05   # 显著正面
    else "继续观察"  if p >= 0.05               # 不够显著，不能下结论
    else "考虑替换方案"                          # 显著负面（变体显著比对照差）
)
```

三级建议的设计非常克制：

- **推广此方案**：Lift 正值且统计显著。这意味着 95% 的把握认为变体确实更好。建议运营把变体设为主图、淘汰对照。
- **继续观察**：p 不显著。不管 Lift 是正是负，只要 p >= 0.05，系统就不会说"推广"或"替换"。因为数据不支持任何确定性的结论。运营可以选"继续观察"（手动延长实验）或"采纳"（凭直觉做决定——虽然统计数据不支持，但运营可能看到了统计没考虑的维度，比如新图的风格和即将到来的节日主题活动更匹配）。
- **考虑替换方案**：Lift 负值且显著。这意味着变体显著比对照差——你本来是"试试新图能不能比老图好"，结果新图反而表现更差。这个结果虽然不够"正面"，但信息量很高——它明确告诉你"这个方向不对"，省了你在错误的方向上继续投入。

"继续观察"是最常见但最考验耐心的结论。很多运营看到"不显著"就觉得实验白做了。但"不显著的零差异"和"未能拒绝零假设"是两个完全不同的结论。前者说"确认了一样"（需要做等效性检验才能得出这个结论），后者说"数据不够，还不能下结论"。Z-test 的结构决定了它只能告诉你"数据支持差异存在"或者"数据不支持差异存在"——后者不能反过来解读为"差异不存在"。

## 八、实验系统的全生命周期

```
Day 1  04:00  → 自动扫描 → 发现商品 #42 的两张图预测 CTR 接近（5.2% vs 5.0%）
              → 自动创建实验 #156（A=图201, B=图187, 50/50）
              → 状态 = RUNNING

Day 1  白天  → 100 次曝光后进入 UCB 阶段，开始根据数据调整流量分配

Day 1-6 每天 06:00 UCB 更新流量分配
              → Day 1 晚: 47/53（B 早期优势，开始略多流量）
              → Day 2: 38/62（B 持续领先，UCB 给了更多流量）
              → Day 3: 35/65（趋势维持）
              → Day 4: 32/68（稳定倾斜，接近下限）
              → Day 5: 30/70（在 30/70 左右稳定）
              → Day 6: 30/70（收敛——即使还剩一些探索，也接近稳定）

Day 7       → 总曝光量达到 10000 → 自动计算显著性 → p = 0.03
            → 实验状态 → COMPLETED，winner = 图187（B）
            → 优化建议 → "B 图比 A 图 CTR 提升 14%，建议替换主图"
            
            运营在飞轮页面看到这个结论，可以选择：
            ├── 采纳：把 B 设为主图，A 淘汰（被实验证明不如 B）
            ├── 再看：不直接替换，手动创建新一轮实验对比 B 和另一个新图

```

一个有趣的细节是：预测模型说 A 可能好 0.2%（5.2% vs 5.0%），实验最终显示 B 更好（还赢了 14%）。预测错了——但这不是预测模型的"故障"。恰恰相反，这正是实验系统存在的价值——它纠正了预测的错误。如果预测永远正确，就不需要实验了。

## 九、与预测模型的分工：假设生成器 vs 假设验证器

```
预测模型（第四章）：         A/B 实验（第五章）：
                          
回答"可能怎样"             回答"实际怎样"
基于历史数据推测            基于真实流量验证
速度快（点一下 10ms）       慢（需要跑几天到一周）
置信区间宽（±20%）          精确到统计显著性（p < 0.05）
用于筛选候选               用于最终决策
可解释性强（特征重要性）     无解释（只知道结果，不知道原因）

两者关系：
  预测 → 筛选出"值得比的"图片对 → 实验 → 验证谁更好
                                          │
                                          ▼
                            实验数据回流 → 数据飞轮标注
                                          │
                                          ▼
                            新标注 → 模型重训 → 下次预测更准
```

预测和实验不是竞争关系，而是**认知链条的上下环节**。缺了预测，你不知道该比什么（几千张图，两两组合号种可能，没法试完）。缺了实验，你不知道预测对不对（预测永远有误差，误判累积就是商业损失）。两者相辅相成，构成 SheLook 的完整决策链条：**先估算，再验证，后学习。**

## 十、核心算法公式汇总

### UCB 分配

```
UCB(i) = CTR(i) + √(2 × ln(N) / n(i))
new_ratio(i) = UCB(i) / Σ UCB(j)
clamp(new_ratio(i), 0.2, 0.8)
```

### Z-test p-value

```
p_pool = (c₁ + c₂) / (n₁ + n₂)
se = √(p_pool × (1-p_pool) × (1/n₁ + 1/n₂))
z = (p₁ - p₂) / se
p_value = erfc(|z| / √2)
```

### Lift

```
Lift% = (CTR_new - CTR_old) / CTR_old × 100%
```

### 自动实验配对

```
best_pair = argmin(|CTR(i) - CTR(i+1)|)  for all i in sorted images
if |CTR(a) - CTR(b)| <= 0.02 → 创建实验
```

## 十一、本章小结

1. **A/B 实验回答了预测模型无法回答的终极问题："差异是真的吗？"** 预测提供方向，实验提供验证——两者不是竞争关系，而是认知链条的上下环节。

2. **自动实验创建筛选"预测 CTR 最接近"的图片对**——把流量花在最有悬念的比较上，而不是验证已知事实。好的实验是"发现未知"，而不是"确认已知"。

3. **UCB 算法在两个极端之间找到平衡**：多给表现更好的图片流量（利用已知信息），保留足够流量给表现差的图片（探索未知可能），同时通过 20-80% 的 clamp 防止过早锁定和赢者通吃。

4. **两阶段策略**（探索期前 100 次曝光的 50/50 → UCB 20%-80% 区间）确保实验在数据极度匮乏的早期不会因为运气成分做出错误偏向，等到有足够基线数据后才交由算法加速。

5. **p-value 是对"差异来自运气"的概率估计**——p < 0.05 意味着如果零假设成立，观察到当前或更极端差异的概率只有 5%。Z-test 基于中心极限定理做正态近似，计算极快且适合大样本比例数据。

6. **多维度下钻（市场/品类/日期）防止辛普森悖论**——整体赢的版本可能在某个局部市场中是输的。切片后的样本量缩减会导致 p 值升高（≠差异消失），结论需配合样本量标注。

7. **Lift 量化相对提升幅度**，结合 p-value 给出三级建议："推广"（显著正面）、"继续观察"（不够显著）、"考虑替换"（显著负面）。"不显著"不等于"无差异"。
