# 第二章：深度学习 —— CLIP 模型与视觉感知体系

> 维护说明（2026-07-23）：本章的算法描述用于教学；实际模型版本、特征和降级行为以运行时配置及后端实现为准。

> 更新说明（2026-07-22）：本文解释当前 CLIP/向量检索设计的原理。模型可用性、版本和实际效果取决于已部署工件、依赖与数据，不能由教学示例推导为固定准确率承诺。

## 一、为什么是 CLIP？

### 1.1 传统路线的根本困境

在 SheLook 诞生之前，如果一个系统想判断"这张商品图好不好看"，需要做什么？

传统做法是：找设计师标注几千张图（好看/不好看），用 ResNet 做二分类，训练几小时，准确率 75%。换一个品类（裙装→鞋履）需要重新标注、重新训练。每一个新需求都是一次独立的"收集数据—标注—训练—部署"循环。

这个流程的问题不在于技术难度，而在于它的不可扩展性。当你只有一两个视觉任务时，传统方案完全可行——一个 75% 准确率的分类器已经比随机猜测（50%）好很多了。但当你需要十个、二十个视觉任务时，维护十个独立的模型意味着维护十套数据集、十次训练流程、十次部署配置，每一个模型都需要持续监控精度衰减、持续更新标注数据。在人力有限的小团队中，这很快会变成不可承受的维护负担。

根本原因在于：传统监督学习教给模型的是"如何在一个预定义的类别集合中做区分"，而不是"图片内容本身的含义"。ResNet 学会的是"把这张图和另外 999 张图区分开"，而不是"这张图里有什么、风格是什么、好不好看"。所以每换一个任务（从判断品类变成判断肤色、从检测 Logo 变成评估构图），就需要重新定义类别集合，重新收集标注数据，重新训练。

### 1.2 CLIP 如何改变了范式

CLIP 直接改变了这个范式：**不需要任何标注数据，不需要任何训练，下载模型就能用。** 因为 CLIP 不是在"学分类"，而是在"学对齐"——它在 4 亿张图文对上学会了把图片和文字映射到同一个语义空间。这意味着：

- 你不需要"标注好的数据集"来训练，你只需要"一段文字"来描述你想检测的东西。想判断肤色分布？写下"浅肤色的人""深肤色的人""不包含人物的照片"。想检测 NSFW 内容？写下"安全的商品照片""包含不适当内容的照片"。每一个新任务只增加一段文字，不增加任何训练成本。
- 一个模型解决所有视觉理解任务：搜图、分类、匹配、评分——全用同一个模型。这意味着只需要加载一次模型（约 600MB 内存），就能覆盖所有视觉感知需求。在 SheLook 中，CLIP 被 8 个不同的业务模块调用，但它只被加载了一次。
- 开发成本从"每个任务训一个模型"降到"下载一个模型"。对于需要快速迭代的产品来说，这种成本降低不是线性的，而是根本性的——它让"试一个视觉检测想法"的成本从数天降到了几分钟。你可以快速尝试"CLIP 能不能判断这张图的拍摄角度"，如果效果不行就换个描述文字再试，而不需要重新标注数据。

这是 SheLook 选择 CLIP 而非传统 CNN 的根本原因。不是精度层面的优劣——单独在肤色检测这个任务上，一个专门用 10 万张肤色标注数据训练的 ResNet 很可能比 CLIP 的零样本推断高 2-5 个百分点。但在 SheLook 需要覆盖十几个视觉任务的背景下，CLIP 的零样本能力带来的成本优势远远超过了单任务上 2-5% 的精度劣势。这是平台型系统的核心权衡：水平覆盖的广度，比垂直单点的极致精度更重要。

这张图揭示了一个关键的架构理念：CLIP 的训练和推理是分离的。OpenAI 已经完成了耗时的训练（4 亿图文对、数周的 GPU 时间），SheLook 只需要下载训练好的权重文件，就能直接进入推理。这种"预训练+冻结使用"的模式是现代 AI 应用的主流范式——少数机构承担巨额的训练成本，全球开发者以零边际成本复用训练成果。

**核心洞察**：CLIP 不是"给图片分类的模型"，而是"建立图文桥梁的模型"。桥架好之后，下游不用再训练新模型，直接用这座桥就能到对岸。你可以把 CLIP 想象成一个翻译官——它同时学会了图片的"语言"和文字的"语言"，并且知道两种语言中哪些表达对应相同的含义。一旦这个翻译官就位，任何需要"图文互译"的任务都变得极其简单。

## 三、ViT：把图片当成文字来读

CLIP 的图像编码器使用 **Vision Transformer (ViT)**，这是 2021 年之后计算机视觉领域最重要的范式转移。在 ViT 之前，CNN（卷积神经网络）统治了计算机视觉近十年。理解 ViT 是理解所有现代多模态模型的基础。

### 3.1 CNN 看见的是像素块，ViT 看见的是"视觉单词"

```
CNN 的工作方式：                     ViT 的工作方式：

输入：224×224 像素矩阵              输入：224×224 像素矩阵
  │                                  │
卷积核 3×3 扫过图像                  切成 49 个 32×32 的小块（patch）
检测边缘、角点、纹理                  每个 patch 拉平成 1024 维向量
  │                                  │
池化层压缩                           每个 patch = 一个"视觉单词"
  │                                  │
卷积核继续扫，检测更高级特征           所有 49 个 patch 排成序列
  │                                  │
... 层层堆叠 ...                     喂给 Transformer
  │                                  │
全连接层 → 输出                       输出 512 维向量
```

CNN 是通过逐层抽象：底层检测边缘，中层检测纹理，高层检测物体部件。每一层都在"向上抽象"。这个架构之所以有效，是因为它模仿了哺乳动物视觉皮层的层次化处理机制——从简单的方向检测（V1 区）到复杂的物体识别（IT 区）。但 CNN 有一个无法克服的弱点：信息跨越图片中相距较远的区域时，需要经过很多层的逐层传递。图片左上角的 Logo 和右下角的模特表情之间的关系，可能要经过几十层卷积才能间接建立。

ViT 完全不同：它把一张图切成 49 个小方块，每个方块就是一个 token（类似 NLP 中一个单词），然后用 Transformer 学习这 49 个 token 之间的关系。在 ViT 的第一层，左上角的 patch 就能直接"看到"右下角的 patch——所有的 patch 之间都是全连接的。这种全连接的全局感受野，让 ViT 天然适合理解图片中相距甚远的元素之间的语义关系。

### 3.2 Transformer 在做什么：自注意力

这是深度学习过去十年最重要的发明。让我用一个具体例子讲清楚。

假设一张商品图被切成 4 个 patch（实际是 49 个，但原理一样）：

```
Patch 1: 连衣裙的领口部分
Patch 2: 连衣裙的腰线部分
Patch 3: 连衣裙的下摆部分
Patch 4: 白色背景
```

Transformer 的自注意力机制做三件事：

**第一步：为每个 patch 生成三个向量**

```
对 Patch 1（领口）：
  Query 向量:  "我是一个领口，我应该关注什么？"
  Key 向量:    "我包含这些特征：弧形边缘、纽扣纹理"
  Value 向量:  "我的实质内容是：领口细节"
```

Query（查询）、Key（键）、Value（值）是三个不同的线性投影。这个命名来自数据库检索的类比：查询匹配键，返回对应的值。你可以把注意力机制想象成一套"兴趣导向的索引系统"：每个 patch 拿着自己的 Query（我感兴趣什么）去和所有 patch 的 Key（我有什么特点）做匹配，匹配度高的就有权读取对方的 Value（对方的实际内容）。

这个机制的精妙之处在于，Query、Key、Value 三组权重矩阵都是可学习的。模型在训练过程中自己学会了"领口区域应该关注同样是衣服结构的其他区域"——这不需要人工定义规则，而是在 4 亿张图文对的对比学习压力下自然涌现的。

**第二步：计算注意力分数**

```
领口的 Query × 腰线的 Key  →  0.8  （高相关：领口和腰线都是衣服结构）
领口的 Query × 下摆的 Key  →  0.6  （中相关：同属衣服的不同部分）
领口的 Query × 背景的 Key  →  0.02 （低相关：领口不关心背景）
```

`softmax` 归一化后：

```
领口 → 腰线: 0.52  （最高注意力）
领口 → 下摆: 0.36
领口 → 领口: 0.11  （也关注自己）
领口 → 背景: 0.01  （几乎忽略）
```

Softmax 归一化是注意力机制的关键步骤。它把原始的注意力分数（0.8、0.6、0.02）变成概率分布（总和为 1）。这样做有两个好处：第一，每个 patch 的注意力总量有上限（等于 1），防止某些 patch 无限放大自己的影响力；第二，概率化的权重有明确的解释——"领口把 52% 的注意力给了腰线"比"领口对腰线的注意力分数是 8.5"更直观。

**第三步：加权求和**

```
领口的最终表示 = 0.52 × 腰线的Value + 0.36 × 下摆的Value
               + 0.11 × 领口的Value + 0.01 × 背景的Value
```

这意味着：**领口的最终向量表示，包含了腰线和下摆的信息**。Transformer 让每个 patch 不再孤立，而是"知晓"它和其他所有 patch 的关系。

理解这一步的关键是：加权求和后的新向量不再只是"领口的描述"，而是"领口在连衣裙这个整体中的位置和角色"。它编码了"我上面是圆领、我紧挨着收腰设计、我下面连着 A 字裙摆"这种全局结构信息。而这正是 CNN 难以做到的——CNN 的每一层只能融合邻近区域的信息，要理解"领口和下摆的关系"需要信息跨越几十层才能间接建立。

这个过程在 ViT 的 12 层 Transformer 中每一层都重复一次。每经过一层，每个 patch 的表示就被更多来自其他 patch 的上下文丰富。经过 12 层之后，每个 patch 的表示已经包含了整张图片的全部信息。

### 3.3 多头注意力：同时从多个角度"看"

单头注意力只从一种关系角度理解图片。多头注意力（Multi-Head Attention）同时运行多个独立的注意力计算：

```
头 1: "哪些 patch 构成了衣服的轮廓？" → 关注边缘连续性
头 2: "哪些 patch 颜色相近？"        → 关注颜色一致性
头 3: "哪些 patch 纹理相似？"        → 关注材质
头 4: "哪些 patch 离得近？"          → 关注空间位置
...
头 8: "哪些 patch 是前景/背景？"     → 关注深度
```

8 个头各自独立计算，最后拼接在一起。每个头相当于一个"专家"，各从不同维度理解图片。

多个头的输出拼接成最终的注意力表示后，再经过一个线性投影混合所有"专家"的见解。这个投影层是可学习的——模型在训练中发现，有的头擅长空间关系（它的注意力模式是"邻近区域高权重"），有的头擅长颜色关系（"颜色相同的高权重"），有的头擅长更抽象的语义关系（"同属一个物体的高权重"）。这种分化不是人工设计的，而是在 4 亿数据的训练压力下，模型发现的最优分工策略。

### 3.4 ViT-B/32 的"B"和"32"是什么意思

```
ViT-B/32

ViT = Vision Transformer

B = Base
    ├── Base:  12层 Transformer, 768维隐藏层, 12个注意力头, 约 8600 万参数
    ├── Large: 24层 Transformer, 1024维隐藏层, 16个注意力头, 约 3 亿参数
    └── Huge:  32层 Transformer, 1280维隐藏层, 16个注意力头, 约 6 亿参数

32 = patch size = 32×32 像素
    ├── /32: 224÷32=7 → 7×7=49 个 patch → 更快（SheLook 的选择）
    ├── /16: 224÷16=14 → 14×14=196 个 patch → 更准（2-3倍推理时间）
    └── /14: 224÷14=16 → 16×16=256 个 patch → 最准（最慢）
```

SheLook 选 B/32 的原因：
- 商品图场景对细粒度要求不高（不需要分清两只猫的品种差异，只需要分清裙子还是裤子）。在电商场景中，32×32 patch 的分辨率已经能清晰区分不同服装品类、分辨拍摄角度、检测模特的有无。
- 推理速度快 3 倍意味着相同硬件可以处理 3 倍的图片量。SheLook 的以图搜图、批量质检等场景，单次操作可能需要编码几十张甚至上百张图片。从 800ms/张降到 200ms/张，意味着批量编码 100 张图从 80 秒降到 20 秒——这是可感知的用户体验差异。
- 部署用 CPU，不追求绝对精度。如果未来上了 GPU 服务器，可以通过环境变量切换到 ViT-L/14，不需要改任何代码。这种灵活性是通过配置文件实现的，体现了 SheLook 架构中"模型可替换"的设计哲学。

## 四、CLIP 是怎么训练的：对比学习

这是理解 CLIP 最关键的部分，也是它区别于所有传统视觉模型的根本。

### 4.1 传统做法 vs CLIP 做法

**传统监督学习（ResNet 等）**：

```
训练数据：(图片, 标签)  —— 比如 (猫图, "cat")
模型学习：图片 → "cat" 这个类别

问题：
- 标签空间固定：训练时是 1000 类，永远只能分这 1000 类
- 新品类需要新数据重新训练。如果你训练了一个 ImageNet 1000 类分类器后
  想识别"连衣裙"，而连衣裙不在 1000 类中，你只能重新收集连衣裙图片
  重新训练
- 标签不包含语义信息："cat"和"dog"的关系模型不知道。分类器只知道
  "第 281 类"和"第 242 类"不同，但不知道它们都是宠物，也不知道
  "第 281 类"更接近"第 242 类"还是更接近"第 500 类"
```

**CLIP 对比学习**：

```
训练数据：(图片, 文字描述)  —— 比如 (连衣裙图, "a red dress with floral pattern")
模型学习：图片和对应文字在向量空间中靠近

优势：
- 没有固定类别：任何文字描述都可以用。不需要预定义类别列表，
  不需要记住"连衣裙是第几类"。你写什么文字，CLIP 就理解什么语义
- 零样本泛化：不需要"肤色分类"的训练数据，不需要收集标注肤色图片，
  只需要文字"light skin tone"，CLIP 就能判断。原因是训练时的图文
  对齐让"light skin tone"这组词的向量和浅肤色图片的向量在空间中接近
- 语义连续："红色连衣裙"和"粉色裙子"的向量在空间中相近。而分类器
  中"红色连衣裙"和"粉色裙子"如果不在训练类别中，就完全没有关系
```

### 4.2 对比损失函数：InfoNCE Loss

训练一个批次（batch）时，假设有 N 对图文：

```
批次 = 4 对图文
  图片1 ↔ 文字1  （正匹配）
  图片2 ↔ 文字2  （正匹配）
  图片3 ↔ 文字3  （正匹配）
  图片4 ↔ 文字4  （正匹配）

计算 4×4=16 个相似度：

         文字1   文字2   文字3   文字4
图片1   0.85     0.12    0.08    0.21    ← 对角线是正匹配，应该最高
图片2   0.03     0.91    0.15    0.07
图片3   0.11     0.09    0.78    0.18
图片4   0.22     0.05    0.13    0.83

目标：让对角线（正匹配）尽可能高，非对角线（负匹配）尽可能低
```

这个 4×4 的矩阵是理解 CLIP 训练的核心。注意这里有 16 个相似度，但只有 4 个是正确的匹配（对角线）。这意味着每次训练，模型要同时处理 4 个正样本和 12 个负样本。当批次大小扩大到几百甚至几千时，负样本数量呈平方增长——在一个 256 对图文的批次中，有 256 个正匹配和 65000+ 个负匹配。这种大规模的正负对比，是 CLIP 能学到精细语义区分的关键。

InfoNCE Loss 的公式和直觉：

```
对于图片 i 和它的正确文字 i：

损失 = -log(  exp(相似度(图片i, 文字i) / τ)  /
             Σ_{j=1}^{N} exp(相似度(图片i, 文字j) / τ)  )

其中 τ（温度参数）控制"有多严格"：
  - τ 小 → 严格要求正匹配远高于负匹配 → 更艰难的区分任务 → 学得更好
  - τ 大 → 宽松 → 更容易收敛但区分能力弱
```

温度参数 τ 是一个被低估的重要细节。在训练初期，模型的能力很弱，连"猫的照片应该配'一只猫'"这种简单匹配都很困难。如果这时 τ 很小（要求严格），模型会收到巨大的损失信号，梯度过大导致训练不稳定。所以 CLIP 的 τ 是可学习的参数——训练初期模型自动调高 τ（放宽要求），等区分能力增强后再自动调低 τ（提高精度要求）。这种自适应机制让训练过程在"简单到能收敛"和"难到能学到东西"之间自动平衡。

这个损失函数同时在两个方向上优化（图文→文图），所以叫"对称交叉熵"。对称性很重要——如果只优化"图片选对文字"而不优化"文字选对图片"，模型可能退化到把所有图片映射到同一个向量。因为只要所有图片的向量都一样，那任何图片和所有文字的相似度都是同一个值，图片选文字时每次都能随机猜对（反正都一样），但文字选图片时就会完全失败。双向约束防止了这种退化。

### 4.3 为什么对比学习能泛化

本质是**强迫模型学习更本质的特征**。

监督学习的目标是"把猫和狗分开"——模型可能投机取巧：只要图里有毛就是猫（通过纹理），不需要理解"猫"是什么。这种捷径在训练集有效，换到"无毛猫"就失败。分类器可以靠任何能区分训练样本的表面特征来达到高准确率——背景的草地、照片的拍摄时间、图片的水印——这些和"猫"本身毫无关系的特征。

对比学习的目标是"让猫图和'一只猫'的文本向量靠近"——文本向量包含了"猫"的语义：四条腿、哺乳动物、宠物等。模型被迫从像素中提取能匹配这些语义的特征，而不是简单的纹理捷径。文字描述"一只猫"不包含背景信息、不包含拍照时间、不包含水印——所以模型如果只依赖这些表面特征，就无法让图片向量接近文字向量。它被迫学习"猫"的真正视觉语义。

这就是 CLIP 零样本能力的来源：**它学到的是语义特征，不是分类边界。** 分类边界只在训练时见过的类别之间有效，新的类别需要新的边界。语义特征是可组合的——"红色"+"连衣裙"+"波西米亚风"可以组合出新概念的向量，即使训练时从未见过这个组合。

## 五、CLIP 在 SheLook 中的五种用法

### 5.1 用法一：图片编码 → 向量检索

```
embedding_service.encode_image("连衣裙.jpg")
→ [0.023, -0.156, 0.487, ..., 0.341]  （512维向量）

存入 pgvector 索引 → HNSW 近似最近邻搜索
查询时：用新图片的向量，搜最相似的 N 个已存向量
```

这是**以图搜图**的基础。不是像素级比对，而是语义级匹配——不同角度、不同光照下的同款连衣裙，向量距离依然很近。因为 CLIP 在训练时已经学会了忽略无关的视觉变化（光照强弱、拍摄角度、背景杂乱的差异）而关注本质的视觉语义（产品的形状、颜色、风格、品类）。

在 SheLook 中，每张上线的商品图在发布时都会经过 CLIP 编码，向量存储到 PostgreSQL 的 product_embeddings 表中。当运营想找"和这张图风格类似的方案"时，上传参考图 → CLIP 编码 → pgvector 检索 → 返回 Top-K 相似商品。整个过程包括了 HNSW 索引的图遍历搜索（第八章详述），从百万级向量中找出最相似的 10 个，通常只需要 5-10ms。

### 5.2 用法二：图文匹配 → 质检防错

```
# image_text_matcher.py 使用
image_vec = encode_image("商品图.jpg")
title_vec = encode_text("波西米亚风碎花连衣裙")

similarity = compute_similarity(image_vec, title_vec)
# 0.78 → 图和标题一致
# 0.12 → 图文不符，可能是上传错了图
```

这是质检流水线中的图文一致性检查——AI 生成的图和商品标题描述是否匹配。

场景：运营在发品工作台为商品"A"生成了 3 张 AI 图，但因为操作失误，将其中一张错误地关联到了商品"B"。如果没有图文匹配校验，这种错误会一直带到上线——消费者看到标题是"白色简约衬衫"，但图片是一条碎花裙子。

CLIP 图文匹配在图片发布前自动校验：将商品标题编码为文本向量，将主图编码为图片向量，计算相似度。相似度低于阈值（如 0.25）时系统警告。这个机制的精妙之处在于不需要任何业务规则——不需要写"如果品类是连衣裙就检查有没有领口"这种脆弱规则。CLIP 的语义空间天然地让"波西米亚风碎花连衣裙"的向量和碎花连衣裙图片的向量接近，而远离白色衬衫图片的向量。

### 5.3 用法三：零样本分类（Zero-shot Classification）→ 多维度自动检测

```
# fairness_service.py 肤色检测
image_vec = encode_image("模特图.jpg")

# 预设文本模板，每类一个
templates = {
    "light":  "a photo of a person with light skin tone",
    "medium": "a photo of a person with medium skin tone",
    "dark":   "a photo of a person with dark skin tone",
    "no_person": "a photo of a product without any person"
}

# 分别编码
text_vecs = {k: encode_text(v) for k, v in templates.items()}

# 比较图片向量和每个文本向量的相似度，取最高的
result = max(text_vecs, key=lambda k: compute_similarity(image_vec, text_vecs[k]))
# → "medium"
```

**关键**：这段代码不需要任何肤色标注数据，不需要训练。只需定义四句话，CLIP 自己判断。这是因为 CLIP 的训练数据中包含了大量带有人物描述（肤色、年龄、性别、表情、着装）的图片，足够的训练暴露让模型对肤色这个语义维度有了稳定的认知。

同样的原理用在：

- **NSFW 检测**（compliance_checker.py）："a photo containing nudity" vs "a safe product photo"。用双通道对比判断——既需要和不安全描述的相似度低，又需要和安全描述的相似度高。只有双重条件都满足才判定为安全。这种双通道设计避免了边缘情况（比如泳装图在"不安全"和"安全"的向量之间处于模糊地带时的误判）。
- **Logo 检测**（brand_service.py）："a photo with a visible brand logo" vs "a photo without any logo"。注意这只能判断"有没有可见 Logo"，不能判断 Logo 是否侵权、属于哪个品牌——那是需要专门训练的细粒度任务，超出了 CLIP-B/32 的能力边界。
- **九维审美评分**（vision_reward.py）："a photo with excellent color harmony" vs "a photo with poor color harmony"。对每个审美维度（色彩、光影、构图、吸引力等九维），定义正反对立的两段描述，取两张描述与图片向量相似度的差异作为该维度的得分。差异越大，说明图片和其他维度上的"好"和"差"之间的区分越清晰。CLIP 在这一类任务上能达到 0.65-0.75 的 Spearman 秩相关系数（与人类专业评审的一致性），因此用作 L3 审美评分中的 60% 权重（另外 40% 来自启发式像素统计规则）。

### 5.4 用法四：文本编码 → 语义搜索

```
用户输入："韩式简约白色衬衫"
  ↓ encode_text(...)
查询向量 [0.12, -0.45, 0.67, ...]
  ↓ pgvector HNSW 索引
返回最相似的 N 张图片（即使图片标题里没有"韩式""简约"等词）
```

这不是关键词匹配，是语义匹配。CLIP 理解"韩式简约"的视觉含义——它知道这通常意味着大量留白、干净的线条、单一或极少的主色调——即使图片没有被打上"韩式""简约"这些标签，在向量空间中它的方向也会和"韩式简约"的文本向量方向接近。

而关键词检索在同样的场景下会完全失败：如果图片的标题或标签中没有"韩式"这个词，系统就不会返回这张图，即使它在视觉上非常"韩式简约"。CLIP 语义搜索弥补了关键词匹配在这个维度的致命缺陷。

### 5.5 用法五：特征提供 → 下游预测

```
# predictor.py 使用
clip_vec = encode_image("商品图.jpg")        # 512 维 CLIP 向量
# PCA 降维到 48 维（保留 85% 方差）
reduced_vec = pca_transform(clip_vec)         # 48 维
# 拼接手工特征：清晰度、色彩饱和度、白边比例、文字密度等
manual_features = extract_manual("商品图.jpg") # 31 维
# 融合
fused = concat([reduced_vec, manual_features]) # 79 维 → GBDT 预测
```

这里 CLIP 向量扮演"视觉理解"的角色——GBDT 能看到这张图的深层语义特征（风格、品类、构图），而不仅仅是表面的像素统计（亮度、色彩饱和度、留白比例）。打个比方：像素统计告诉 GBDT "这张图的红色像素占比 35%"，CLIP 向量告诉 GBDT "这张图看起来像高端设计师品牌"——后者对 CTR 预测的价值远大于前者。

降维从 512 到 48 维是关键。GBDT 是基于决策树的模型，树的分裂点需要遍历所有维度寻找最优分裂。512 维意味着每次分裂需要评估 512 个候选维度，而 48 维只需要评估 48 个。维度降低 90%，训练速度提升 5-10 倍，同时降低了过拟合风险（高维度 + 小样本 = 灾难性过拟合）。PCA 保留了方差最大的 48 个方向，相当于保留了图片语义中最显著、信息量最大的 48 个特征维度。丢失的 15% 方差对应的是细微的、噪声级的视觉差异——对"判断 CT 大概在什么区间"来说，这些噪声没有信息价值。

## 六、工程细节：模型怎么在项目中运行

### 6.1 模型加载策略

```
应用启动时（main.py lifespan）
  │
  └→ init_embedding_model()
       │
       ├── 1. 检测 GPU → 没有，用 CPU
       ├── 2. 从 HuggingFace Hub（hf-mirror.com 镜像）下载模型权重
       │        ├── CLIPModel:         约 600MB（ViT-B/32 权重 + 文本编码器权重）
       │        └── CLIPProcessor:     约 5MB（tokenizer + 图片预处理配置）
       ├── 3. 加载到内存：模型 + 中间激活 ≈ 1.2GB
       └── 4. 设为 eval() 模式（关闭 dropout、batchnorm 随机性）
```

**为什么启动时加载而不是懒加载？**

- 模型 600MB，第一次请求才下载会导致首请求超时（下载 5-15 秒 + 推理 0.2 秒）。用户点击"以图搜图"后要等 15 秒才开始搜，这个体验是不可接受的。
- 多 worker 模式（`API_WORKERS=3`）下，每个 worker 是独立进程，各自加载一份模型。Python 的多进程模型决定了内存不能共享——每个子进程有自己独立的内存空间，所以 3 个 worker 各自加载一份 600MB 的模型到各自的内存中。
- 3 worker × 1.2GB = 3.6GB 内存开销——所以 Docker Compose 给 backend 分配了 2GB（开发）到 4GB（生产）。生产环境的 4GB 刚好够 3 worker × 1.2GB = 3.6GB + 操作系统和 Python 运行时约 400MB 的开销。

### 6.2 推理流程：从接收到返回向量

```
HTTP 请求：POST /api/schemes/recommend
  │
  ├→ api/schemes.py: recommend_schemes()
  │     │
  │     ├→ image_fetcher.fetch_image(url)    # 下载图片到内存
  │     ├→ open_image_source(url)            # URL/路径→PIL Image
  │     └→ embedding_service.encode_image()  # ← 我们在这里
  │
  └→ encode_image(image_path)
        │
        ├→ 1. open_image_source(path)   # 统一转 PIL Image
        │     └→ fetch_image_sync(url)  # URL 场景：httpx 流式下载
        │        ├── 大小限制：20MB（防止超大文件撑爆内存）
        │        ├── 超时：30 秒（防止恶意慢速连接耗尽连接池）
        │        └── 安全校验：阻止内网 IP（SSRF 防护，见第八章详述）
        │
        ├→ 2. get_clip_embedding(img)
        │     ├→ img.convert("RGB")         # 强制 3 通道（处理 RGBA/灰度图）
        │     ├→ processor(images=img)      # Resize 224×224 + 归一化
        │     │    ├── Resize(224, 224)     # ViT 固定输入尺寸
        │     │    ├── CenterCrop(224, 224) # 从中心裁剪（避免边缘变形）
        │     │    └── Normalize(mean, std) # ImageNet 统计值标准化
        │     │
        │     ├→ model.get_image_features()  # ViT 前向传播
        │     │    ├── Patch Embedding: 224×224 切 49 个 32×32 patch
        │     │    │   每个 patch: 3072 维像素 → 768 维特征向量
        │     │    │
        │     │    ├── Transformer × 12 层:
        │     │    │   ├── Multi-Head Self-Attention (12 heads)
        │     │    │   │   每个 head: QKV 投影 → 注意力分数 → 加权求和
        │     │    │   │   12 个 head 输出拼接 → 768 维
        │     │    │   ├── Layer Normalization（稳定训练）
        │     │    │   └── MLP (768 → 3072 → 768)（非线性变换）
        │     │    │   ... 重复 12 次，每次 patch 之间交换信息
        │     │    │
        │     │    └── Class Token 经过 12 层后输出 768 维
        │     │         → Linear Projection → 512 维（最终输出）
        │     │
        │     ├→ torch.no_grad() 上下文     # 不记录梯度，省 ~50% 内存
        │     │                               因为不训练，梯度计算是纯浪费
        │     │
        │     └→ L2 Normalization           # 向量长度归一化为 1
        │          v = v / sqrt(v₁² + v₂² + ... + v₅₁₂²)
        │          意义：余弦相似度退化为点积，pgvector 检索更快
        │
        └→ 3. .cpu().numpy()[0].tolist()   # GPU tensor → CPU → numpy → Python list
             → [0.023, -0.156, ..., 0.341]  # 存入 pgvector 的 format

总耗时：~200ms（CPU）/ ~50ms（GPU）
```

### 6.3 模型存放与缓存

```
容器内路径：
  /home/appuser/.cache/huggingface/
    ├── models--openai--clip-vit-base-patch32/
    │   ├── blobs/              # 模型权重分片文件（二进制 blob）
    │   ├── refs/main           # 版本指针（指向当前使用的快照）
    │   └── snapshots/xxx/      # 当前版本快照目录
    │       ├── config.json     # 模型配置（12层、768维、12头、ViT-B/32）
    │       ├── preprocessor_config.json  # 图片预处理参数
    │       ├── tokenizer.json  # 文本 Tokenizer 词表（BPE 编码）

Docker 卷挂载策略：
  - hf_cache 卷挂载到 /home/appuser/.cache/huggingface
  - backend 和 celery-worker 共用同一个卷
  - 首次启动 backend 下载（5-15 分钟），celery worker 直接复用（0 分钟）
  - 卷持久化在宿主机磁盘上，容器重启不丢失模型缓存
```

这个共享卷设计解决了两个关键问题：磁盘空间（不需要两份 600MB 的模型文件）和启动时间（celery-worker 不需要重复下载）。但有一个隐含的约束：backend 和 celery-worker 必须使用相同版本的 CLIP 模型。如果未来需要定制（比如对 CLIP 做微调），需要同步更新两个容器的模型版本。

## 七、CLIP 的边界：它不能做什么

理解一个模型的局限和理解它的能力同样重要。过度依赖 AI 的后果往往比不依赖 AI 更严重——因为你会盲目相信机器的输出。

### 7.1 不能做：精确测量

```
× 这张裙子的长度是 85cm？       → CLIP 不能
× 图片里有几个纽扣？            → CLIP 不能
× 文字有没有拼写错误？          → CLIP 不能
× 这个 Logo 是不是侵权了某品牌？ → CLIP 不能（只能检测有没有 Logo）
```

CLIP 输出的是**语义级别的模糊匹配**，不是精确的物体检测或 OCR。这就是为什么 L2 质量评分中，CLIP 只提供注意力引导，具体的清晰度、分辨率等指标还是靠传统的像素分析方法（拉普拉斯算子、直方图分析等）。

### 7.2 不能做：细粒度区分

```
√ 这是连衣裙还是裤子？     → 能区分（品类差异大，patch 粒度够）
× 这是 A 品牌的连衣裙还是 B 品牌的？ → 不能（品牌差异在细节，需要更小的 patch）
× 这是真丝还是仿真丝？      → 不能（材质纹理在 32×32 尺度下被平滑掉了）
```

ViT-B/32 的 32×32 patch size 决定了它看不清纹理级别的细节。在 32×32 像素的方块中，真丝特有的光泽纹理、仿真丝的均匀反光都无法被分辨——两者在块平均后看起来几乎一样。如果需要区分面料材质，需要用 ViT-L/14（14×14 patch，能看清 4 倍的纹理细节）或专门的细粒度分类模型（如专门的材质识别网络）。

### 7.3 数据偏差：CLIP 的预训练数据是英文互联网

```
encode_text("旗袍")     → 偏向于西方视角的"东方服饰"理解
                          可能会混入刻板印象（红色、盘扣、龙凤图案）
encode_text("qipao")   → 偏向于更准确的语义
                          英文语境下的 qipao 在训练数据中通常配图是准确的

encode_text("中国风")   → 可能混入刻板印象（红色、龙纹、灯笼）
encode_text("Chinese traditional style") → 更中性的描述
```

虽然 CLIP 支持多语言，但训练数据以英文为主（4 亿图文对绝大多数来自英文网页）。中文 token 在文本编码器的 BPE 词表中可能被拆分成更碎的 subword 片段，导致语义表示不如英文紧凑和精确。中文描述的向量可能偏向于训练数据中最常与这些中文字符共现的图片——而如果这些图片本身带有文化刻板印象，偏见就会传递到下游任务。

SheLook 中的肤色分析模块对此有意识——预设文本模板使用英文而非中文。这是一个务实的处理方式：既然 CLIP 在英文上的表现更好，就在需要高精度的任务上使用英文。

### 7.4 SheLook 中的实际限制

| 限制 | 影响 | 应对 |
|------|------|------|
| 输入尺寸固定 224×224 | 小物件（耳环、戒指）在缩放后失真，细节丢失 | 目前未处理，低优先级（主打品类是服装） |
| 只能看一帧 | 视频需要抽帧分析，无法理解动态信息 | video 模块独立使用 Kling/Runway，与 CLIP 无关 |
| 不支持多图输入 | 不能理解"Before/After"对比、"搭配推荐"等需要比较两张图的场景 | ComparisonViewer 组件交给人工处理 |
| 512 维向量有信息损失 | 极相似图片（如不同颜色的同款商品）区分困难 | A/B 实验改用真实流量数据，不依赖向量距离判断 |

## 八、从 CLIP 到多模态 AI 的演进

CLIP 是 2021 年的产物。今天（2026 年）回头看，它在多模态 AI 演进中处于什么位置？

```
2020 ─── CLIP 发布：图文对齐，零样本分类
          意义：第一次证明图文对比学习的规模化可行
2021 ─── DALL·E 2：CLIP 潜空间 → 扩散模型生成图片
          意义：CLIP 从"理解"的工具变成了"生成"的条件
2022 ─── Stable Diffusion：开源，CLIP Text Encoder 作为条件输入
          意义：CLIP 成为文生图生态的标准组件，任何人可免费使用
2023 ─── GPT-4V：原生多模态，看图和看文字没有界限
          意义：突破了 CLIP 的"编码后比较"模式，实现真正图文融合理解
2024 ─── Gemini 2.0：图文音视频全模态理解和生成
          意义：多模态进入"通用智能"阶段，一个模型处理所有模态
2025 ─── Qwen-VL / DeepSeek-VL：国产开源多模态大模型崛起
          意义：多模态能力不再被少数公司垄断，开源生态繁荣
2026 ─── SheLook 使用：CLIP 做感知 + Gemini 做生图/审核
          意义：混合架构，用性价比最高的方式组合不同代际的 AI 能力
```

SheLook 的架构体现了这种演进：
- **感知层**用 CLIP（本地、离线、低成本、高频调用，一张图片 200ms）
- **生成层**用外部 API（云端、最新模型、效果好，但一张图 30-60 秒）
- **决策层**用传统 ML（数据不出租户、可解释性高、训练只需秒级）

这种分层设计让系统在未来可以灵活替换各层：如果明天出了一个比 CLIP 更好的开源视觉编码器（比如国产的 Qwen-VL 开源版本的视觉塔），只需要替换 `embedding_service.py`，上游 8 个调用方不受影响。如果 Gemini 出了更便宜的版本，只需要修改生图任务中的模型名称参数即可迁移。如果 CTR 预测需要更强的模型，可以保留 GBDT 的架构，用 XGBoost 替换 HistGradientBoosting 提升精度。每一层都是独立可替换的，没有单点锁定。

## 九、关键公式与技术细节（补充）

### 9.1 Patch Embedding 的数学

```
图片: 224×224×3（高×宽×RGB通道）
切块: 32×32, 共 (224/32)×(224/32) = 7×7 = 49 个 patch
每个 patch: 32×32×3 = 3072 个像素值

线性投影: 3072 → 768（可学习的线性变换）
E = X·W + b
  其中 X: [49, 3072], W: [3072, 768], b: [768]

加上位置编码:
Z₀ = [class_token; E₁+pos₁; E₂+pos₂; ...; E₄₉+pos₄₉]
  形状: [50, 768]（1个分类token + 49个patch）

class_token：可学习的向量，初始随机，经过 12 层 Transformer 后
作为整张图的全局表示输出
```

位置编码为什么是加法而不是拼接？因为加法保持维度不变的同时加入了位置信息，而拼接会改变维度。如果拼接位置编码（比如 768 维 + 768 维 = 1536 维），后续所有层的权重矩阵维度都要改变，模型结构变得复杂且无法复用标准 Transformer 代码。加法的代价是 patch 的内容和位置信息混合在同一组 768 维中，但实验证明 Transformer 足够强大，能在混合中同时提取内容和位置信息。

### 9.2 Self-Attention 的矩阵运算

```
Q = Z·W_Q    # [50, 768] × [768, 64] = [50, 64]
K = Z·W_K    # [50, 768] × [768, 64] = [50, 64]
V = Z·W_V    # [50, 768] × [768, 64] = [50, 64]

Attention(Q, K, V) = softmax(QK^T / √64) · V

QK^T: [50, 64] × [64, 50] = [50, 50]  ← 50×50 的注意力矩阵
每行代表：第 i 个 token 对第 j 个 token 的关注程度

√64 = 8：缩放因子，防止点积过大导致 softmax 梯度消失
为什么需要缩放？如果 Q 和 K 的各维度是独立标准正态分布，
点积的方差为 64（因为 64 个维度各自贡献方差 1）。
不缩放时，点积值可能在 0-64 范围，经过 softmax 后几乎退化为 one-hot
（最大值的概率接近 1，其他接近 0），梯度趋近于零，无法训练。
除以 √64 将方差缩回到 1，softmax 分布保持适度的"平滑"。
```

### 9.3 L2 归一化与余弦相似度

```
未归一化的向量: a, b
余弦相似度 = (a·b) / (|a| × |b|)

归一化后: a' = a/|a|, b' = b/|b|  （|a'| = 1, |b'| = 1）
余弦相似度 = a'·b'  （除法退化为点积）

为什么这样做：
1. 每张图只需归一化一次，之后的所有比较都用点积（快 2 倍）
   - 不用归一化：每次比较都要算 |a| 和 |b|（各 512 次乘加 + 1 次开方）
   - 归一化后：只需要 512 次乘加的点积
2. pgvector 的 cosine 距离算子内部也用这个优化
3. 512 维点积 ≈ 512 次乘加运算，约 5ns（现代 CPU 一条 SIMD 指令完成），几乎不耗时
```

### 9.4 PCA 降维：512 → 48

```
解释（概念示意）：

# 收集历史 CLIP 向量作为降维参考
X = [vec1, vec2, ..., vecN]  # N 张历史图片的 512 维向量矩阵

# PCA 主成分分析
cov = (X - mean).T @ (X - mean) / N      # 协方差矩阵 [512, 512]
eigenvalues, eigenvectors = eig(cov)       # 特征分解
top48 = eigenvectors[:, :48]              # 取前 48 个最大特征值对应的方向

# 降维
reduced = (vec - mean) @ top48           # [512] → [48]

# 48 维保留了原始 512 维约 85% 的信息方差
# 为什么要降维：GBDT 输入维度太高会过拟合，79 维是手工+降维后的平衡点
```

PCA（主成分分析）的思想是：数据在高维空间中通常不是均匀分布的，而是集中在少数几个方向上（想一想"一条弯弯曲曲的线在 3D 空间中，但本质上它是一维的"）。PCA 通过特征分解找到数据方差最大的方向（主成分），只保留这些方向上的投影，丢弃方差很小的方向（那些方向上的差异大概率是噪声而不是信号）。

512 维→48 维，保留了 85% 的方差，丢弃了大部分维度但只丢了 15% 的有意义信息——这说明 CLIP 向量的 512 维中有大量冗余和相关。这种冗余对于 CLIP 自己的检索效果是好事（冗余意味着鲁棒性），但对于 GBDT 的树分裂是坏事（冗余意味着不必要的维度和过拟合风险）。PCA 在保留本质信息的前提下剔除了冗余，让 CLIP 输出和 GBDT 输入之间达到了最佳匹配。

## 十、本章小结

1. **CLIP** 是 SheLook 的视觉感知底座，通过对比学习在 4 亿图文对上训练，获得了零样本视觉理解能力。它不是分类模型，而是"图文语义空间的对齐器"——这个本质决定了它能用一个模型覆盖十几个视觉任务。
2. **ViT**（Vision Transformer）是 CLIP 的图像编码器，将图片切成 patch 后用自注意力机制捕捉全局关系。与 CNN 的局部逐层传递不同，ViT 的第一层就让每个 patch 直接感知所有其他 patch——信息的全局流动不再需要跨越几十层中转。
3. **五种用法**覆盖了 SheLook 的主要视觉任务：向量检索（以图搜图的基础）、图文匹配（发布前防错）、零样本分类（肤色/NSFW/Logo/审美一键检测）、语义搜索（超越关键词的图片检索）、特征提取（为 GBDT 提供"视觉理解"，补足像素统计无法捕捉的风格和语义）。
4. **工程实现**上采用单例加载、应用预热、CPU 推理、镜像下载、L2 归一化等优化，在无 GPU 环境下仍能达到 ~200ms/张。多 worker 架构下每个进程独立加载模型（3×1.2GB 内存开销已在 container resource limits 中配置预留）。
5. **CLIP 有边界**：不擅长精确测量（纽扣数量、文字 OCR）、细粒度区分（品牌差异、材质鉴别）、中文语义（训练数据偏差）——理解局限性能避免过度依赖。在 SheLook 中，这些边界通过任务分流（精确测量用像素分析、细粒度区分交人工、中文用英文替代）来管理。
