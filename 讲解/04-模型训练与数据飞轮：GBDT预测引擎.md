# 第四章：模型训练与数据飞轮 —— SheLook 如何"越用越聪明"

> 维护说明（2026-07-23）：真实 CTR 反馈现已使用独立效果事实、预测快照和成熟标签；不得将实际值回写为预测记录。

> 更新说明（2026-07-22）：本文描述模型训练与反馈回流的实现思路。预测和训练输出是决策辅助，必须以目标租户的真实数据覆盖、评估报告与版本工件为准。

---

## 一、先回答一个问题：这个系统有"自己的模型"吗？

前三章讲了 CLIP、FLUX、Gemini、Kling——这些都是别人训练好的模型，SheLook 的角色是把它们下载或调用过来用。那 SheLook 自己训练了什么？

**答案：SheLook 自己训练了一个基于 GBDT 的预测模型，而且它会在生产环境中持续自动进化。**

这个自我训练的模型负责回答三个具体到不能再具体的问题：

1. 这张商品图上线后，大概会有多少人点击？—— **CTR 预测**（Click-Through Rate，点击率），这是一个回归问题，输出一个 0% 到 100% 之间的数值。
2. 这张图会不会成为爆款（表现超过同品类 75% 的图片）？—— **爆款概率预测**，这是一个二分类问题，输出"是爆款"的概率。
3. 买了这个商品的人退货率会高吗？—— **退货风险预测**，这也是一个二分类问题，输出"高退货风险"的概率。

这三个预测不是学术实验——它们直接决定了运营人员每天的操作决策。如果 CTR 预测说这张图可能只有 1.2% 的点击率（而同品类平均是 2.8%），运营就不会发布它，而是要求重新生成。如果退货风险预测说某个风格方案的退货率可能高达 12%，运营会避开这个方案。如果爆款概率预测给出了 85% 的置信度，这张图会被优先分配给流量最高的商品位。

这个模型就是 SheLook 的"决策大脑"。它不是靠人工编写的规则（"如果模特穿着 → CTR+1%"），而是从每一个租户自己的历史数据中自动学习"在你的品类、你的市场、你的价格带，什么样的图卖得好"。这个学习过程不需要人工干预——数据飞轮自动完成"收集反馈 → 标注好坏 → 训练模型 → 应用预测"的完整闭环。

## 二、为什么要自己训练模型？大模型不行吗？

每一个接触 SheLook 的人几乎都会问这个问题。Gemini 不是能看图吗？它能看出"这张图清晰、构图好、色彩和谐"，那让它直接猜 CTR 不就行了？

### 2.1 大模型不会做精确数值预测

你把一张连衣裙商品图发给 Gemini，让它预测 CTR，Gemini 的回答大概是这样：

> "Based on the image quality and composition, this product photo could achieve a click-through rate between 2% to 5% on an e-commerce platform."

你觉得这个回答有用吗？"2% 到 5%"是一个宽度 3 个百分点的区间，而一个品类的真实 CTR 分布通常只在 1.5%-4.5% 之间——这意味着 Gemini 的预测覆盖了几乎整个可能的 CTR 范围。这种预测对决策没有任何帮助。运营需要的是"这张图预估 CTR 约 3.4%，比同品类平均高出约 0.9 个百分点"这种精确度——这样才能在不同的方案之间做出取舍。

大模型不适合做精确数值预测有几个根本原因：

**第一，大模型的"数值感"很差。** 大模型的本质是预测下一个 token（一个字或一个词）的概率分布，它学会的是语言的流畅性和逻辑性，而不是精确的数值校准。当你问它"CTR 是多少"，它会在概率最高的那些数字 tokens 中采样——"3%""4%""2.5%"这些在训练数据中经常出现的数字更有可能是它的输出。但"3.42%"这种精确到两位小数的数字，大模型几乎不可能稳定地生成——它的训练目标不包含数值精度，它的损失函数不会因为它把 3.4% 说成 3.7% 而惩罚它。

**第二，你无法验证它的推理过程。** 你问 Gemini "为什么预测 3%"，它可能回答"因为图片清晰度高、色彩搭配好"——这听起来合理，但它真的是通过"清晰度→CTR"这个因果链得出的结论吗？还是先凭直觉说了一个数字，再用语言模型的能力"编"了一个合理的解释？这个问题叫"推理与解释的分离"——大模型善于解释，但不一定是在对自己真实的推理过程做解释。你无法区分它是在"诚实报告"还是在"事后合理化"。

**第三，同一天问同一个问题可能得到不同答案。** 大模型推理时有一个叫"温度"的参数控制输出的随机性。即使温度为 0（最确定模式），底层 GPU 的浮点运算不确定性也可能导致微小差异。而更为关键的是，Gemini 模型自身在持续更新——你 7 月问的 Gemini 2.0 和 9 月问的 Gemini 2.1，内部权重已经不一样了，对同一张图的"判断"也可能不一样。对于需要持续追踪和对比（"这个月的图比上月好还是差"）的业务场景来说，不可复现的判断是一个严重问题。

### 2.2 预测需要"记忆"私有历史数据

这是 GBDT 相对于大模型最根本的优势，也是为什么不管大模型多强大都不能替代一个自己训练的预测模型：

```
某卖家的连衣裙品类，过去 30 天：
  - 白色背景 + 45° 侧拍 + 无模特的图，平均 CTR 4.1%
  - 生活场景 + 正面拍摄 + 无模特的图，平均 CTR 2.3%
  - 白色背景 + 正面拍摄 + 模特穿着的图，平均 CTR 5.8%
  - 暖色调背景 + 俯拍的图，平均 CTR 1.9%（惨淡）

这些数据是只属于这个卖家、这个品类、这个市场的私有信息。
它们不在任何大模型的训练数据中，Gemini 也不可能"知道"这些信息。
```

大模型可以评估单张图片的"通用质量"——构图好不好、光线是否均匀、色彩是否和谐——但它不知道"在这个卖家的连衣裙品类中，什么样的图历史上卖得好"。就像一个外聘的设计顾问，她能评价一张图的专业水准，但她不知道你的客户喜欢什么风格。

GBDT 模型不需要理解什么是一张"好图"。它只需要记住一件事：在过去 30 天里，那 50 张 CTR 超过 4% 的图都有什么共同特征（比如大量使用了白色背景、模特穿着的比例很高、价格都在 $15-25 之间），而那 50 张 CTR 低于 2% 的图又有什么共同特征（比如拍摄角度偏俯拍、画面中文字密度高、颜色饱和度太低）。它学到的是纯粹的数据规律，不涉及任何审美判断。

### 2.3 三者配合：CLIP 做翻译，GBDT 做决策，Gemini 做裁判

SheLook 中三个 AI 角色的配合是理解整个系统运行逻辑的关键：

```
CLIP（视觉编码器）：    "我把图翻译成 512 个数字。我不懂这些数字代表什么，
                        但我知道相似的图数字也相似。"
                        
GBDT（决策模型）：      "我不知道什么叫'好看'，什么叫'品牌调性'。
                        但我记住了：在你店铺里，第 23 维 > 0.5 且第 7 维 < -0.2 的图，
                        历史上 CTR 比平均高 40%。"

Gemini（审核大模型）：   "我能评价这张图的审美、品牌一致性、整体吸引力。
                        CLIP 看不懂这些主观维度，GBDT 记不住这些审美规律。
                        我的判断可以作为 GBDT 特征之外的辅助参考。"
```

三者的关系不是竞争，而是互补。CLIP 弥补了 GBDT 不懂视觉语义的短板（把图翻译成数字），GBDT 弥补了 CLIP 不懂历史数据的短板（从数字中学习规律），Gemini 弥补了前两者都不懂主观审美的短板（提供审美判断）。

## 三、GBDT 是什么？为什么选它而不是神经网络？

### 3.1 梯度提升树：让一百棵小树精诚合作

GBDT（Gradient Boosting Decision Tree）的算法名称听起来复杂，但思想极其朴素。理解它的最好方法，是跟随着一个具体的预测过程：

```
问题：预测一张连衣裙商品图的 CTR

第一步：瞎猜。所有人一开始都猜同品类的平均 CTR = 3.2%
      你给每一张图都猜 3.2%——当然不准确，但这是一个起点。

第二步：看错误。我们看看每一张图实际是多少，和我猜的差了多少：
       图A 实际 5.1% → 我猜 3.2% → 差 +1.9%（我猜低了）
       图B 实际 2.1% → 我猜 3.2% → 差 -1.1%（我猜高了）
       图C 实际 4.5% → 我猜 3.2% → 差 +1.3%（我还是猜低了）
       这个"差值"叫"残差"——就是系统还没学到的规律。

第三步：派第一棵小树去学"残差"。
       这棵小树（通常只有 3-5 层深，非常浅）被训练来预测"差值"而不是"CTR"。
       它很快发现一个规律：
          如果图片有模特穿着 → 差值一般是正的（实际 CTR 比平均高）
          如果图片是平铺拍摄 → 差值一般是负的（实际 CTR 比平均低）
       这棵小树输出：模特图 +1.5%，平铺图 -0.8%。
       
       现在的预测不再只是 3.2% 了：
       图A（模特穿着）：3.2% + 1.5% = 4.7%（更接近真实的 5.1%了）
       图B（平铺）：    3.2% - 0.8% = 2.4%（更接近真实的 2.1%了）
       
       新的残差：
       图A 残差：5.1% - 4.7% = +0.4%（还有正的小偏差）
       图B 残差：2.1% - 2.4% = -0.3%（还是猜高了一点）

第四步：派第二棵小树去学"新的残差"。
       这棵小树的目标不再是 CTR，而是第一棵树没学完的小偏差。
       它发现：
          如果图片是暖色调 → 小偏差一般是正的（暖色调图实际 CTR 比第一棵树预测的还高一点）
          如果图片是冷色调 → 小偏差一般是负的
       这棵小树输出：暖色调 +0.3%，冷色调 -0.2%。
       
       现在的预测：
       图A（模特+暖色调）：3.2% + 1.5% + 0.3% = 5.0%（更接近 5.1%了！）

第五、第六……第一百步：重复这个"看残差→派一棵小树→修正预测"的过程。
       每一棵新树只负责修正"之前所有树的预测和真实值之间的小偏差"。
       越往后，残差越来越小，每棵新树"要修正"的东西越来越少。
       到了第 100 棵树，残差可能只剩 ±0.02%——训练停止。
       
最终预测 = 3.2% + 树1的修正 + 树2的修正 + ... + 树100的修正
         = 一个接近真实 CTR 的、精确到 0.01% 的预测值
```

这就是"Boosting"（提升）的含义——**后面每一棵树都在弥补前面所有树的不足，一棵一棵地"提升"预测精度。** 每一棵树本身都很弱（只学习了一个简单的规则，如"模特穿着的图 CTR 偏高"），但 100 棵弱树的组合可以学到非常复杂的规律（"模特穿着 + 暖色调 + 白色背景 + 价格中等 → CTR 特别高"）。

### 3.2 梯度提升的"梯度"是什么？

很多人记住了 Boosting 的直觉，但对"梯度"这个数学词感到困惑。这里的梯度不是神经网络中的反向传播梯度，而是一个数学优化中的泛化概念：**梯度代表了"朝哪个方向走能最快减少误差"**。

在 GBDT 中，每棵新树不是在拟合残差本身，而是在拟合损失函数（比如均方误差）对预测值的一阶导数——也就是梯度。对于回归问题（预测 CTR），这个梯度恰好等于残差（乘以一个常数），所以"拟合残差"和"拟合梯度"是等价的。但 GBDT 之所以叫"梯度"提升而不是"残差"提升，是因为它可以扩展到任意可微的损失函数——分类问题（预测爆款概率）使用的对数损失函数的梯度就不再等于残差。

这个细节对理解算法的通用性很重要，但对理解 SheLook 如何使用 GBDT 不那么关键。你只需要知道：对于 CTR 预测（回归），"拟合残差"这个直觉是完全准确的；对于爆款概率和退货风险（分类），这个直觉仍然基本成立，只是数学上稍微复杂一点。

### 3.3 为什么用 HistGradientBoosting 而不是 XGBoost 或 LightGBM？

SheLook 使用的是 scikit-learn 的 `HistGradientBoostingRegressor`（回归）和 `HistGradientBoostingClassifier`（分类）。这三个 GBDT 变体在算法原理上高度相似，但工程实现上各有侧重：

| 特性 | HistGradientBoosting (sklearn) | XGBoost | LightGBM |
|------|-------------------------------|---------|----------|
| 缺失值处理 | 原生支持（训练时自动学习"缺失值应该走左边还是右边"） | 需手动填充或设置 missing 参数 | 原生支持 |
| 安装依赖 | 只有 numpy/scipy（Python 纯实现） | 需要编译 C++ 扩展 | 需要编译 C++ 扩展 |
| 训练速度 | 中等（纯 Python 但有 Cython 加速） | 快（C++底层） | 最快（基于直方图的优化算法） |
| 模型文件大小 | 较小 | 中等 | 较小 |
| Docker 部署友好度 | 极高（pip install scikit-learn 零编译） | 中等（可能出现 C++ ABI 不兼容） | 中等（同 XGBoost） |
| 生态成熟度 | 最高（sklearn 的标准 API 生态） | 高 | 高 |

SheLook 选择 HistGradientBoosting 的核心原因有两个：

1. **零编译依赖**。Docker 镜像构建时，`pip install scikit-learn` 纯粹是复制预编译的 wheel 文件，没有任何 C++ 编译过程。而 XGBoost/LightGBM 需要编译 C++ 代码——在 slim Docker 镜像中（SheLook 使用的是 python:3.13-slim-bookworm），可能缺少必要的编译工具链（gcc、libstdc++），导致安装失败。生产环境中 Docker 构建的可靠性比算法性能更重要。

2. **"够用就行"的哲学**。SheLook 的训练样本量通常在 50 到 5000 之间（每个租户每类商品的图片数量），训练特征 79 维，树的数量 100 棵。在这个规模下，HistGradientBoosting 的训练时间是 0.5 到 5 秒——XGBoost 可能更快（0.3 到 3 秒），但 2 秒的差异在每周只训练一次的场景下毫无意义。省下的 2 秒不足以补偿承担编译依赖的运维风险。

### 3.4 为什么不直接用神经网络？

这是 SheLook 技术选型中最容易被质疑的一点——现在都 2026 年了，不用深度学习做预测是不是太保守了？

```
神经网络（MLP）的要求：
  数据量： 至少 1000+ 样本才能不严重过拟合
           在小样本情况下，MLP 的 VC 维（模型复杂度）远超 GBDT，
           会"记住"训练样本中的噪声而不是学到真实规律
  硬件：   需要 GPU（训练 > 1000 样本时 CPU 太慢）
           意味着更贵的云服务器或更复杂的部署配置
  超参数： 层数、每层神经元数、激活函数、学习率、batch size、
           dropout 率、weight decay、早停 patience……
           GBDT 的超参数只有 3-4 个（树数、深度、学习率、最小叶子样本数）
  可解释性：黑盒。无法回答"为什么预测这张图 CTR 高"
           GBDT 天然提供特征重要性（feature importance）
           "第 23 维 CLIP 特征 + 品类是连衣裙 + 价格中等 = CTR 高"
           这个解释对于运营人员和产品经理来说是至关重要的
```

在 SheLook 的多租户场景中，GBDT 相对于神经网络的压倒性优势更加明显。一个新入驻的租户可能只有 30 张上线过的商品图——GBDT 可以勉强训练（虽然效果一般但有参考价值），神经网络会完全过拟合（在这 30 个样本上训练误差为零，但对新数据完全瞎猜）。GBDT 小样本有效的根本原因是决策树的归纳偏置——树模型天然假设决策边界是轴对齐的阶梯函数，这个假设虽然限制了表达能力，但也极大地降低了模型的复杂度，使它在小样本时不容易过拟合。神经网络的假设空间是光滑的非线性函数，表达能力强，但对数据的"饥渴"程度也是指数级增长的。

还有一个关键的现实因素：如果一个租户只有 50 张图片，你给她预测说"这张图的 CTR 估计 3.5%"，她能接受。但如果你告诉她"这个预测是用神经网络基于 50 个样本做的"，从统计学的角度来看这就是在过拟合。GBDT 在小样本场景里至少"诚实"——它的预测区间很宽（±40%），明确告诉运营"数据不够，预测仅供参考"。而神经网络即使在小样本上过拟合，也可能输出一个看起来很精确但完全错误的结果（比如"CTR 4.87%，置信度 95%"——这是在自欺欺人）。

## 四、79 维特征：模型到底"看"什么？

### 4.1 特征全景图

79 维融合特征 = 31 维手工特征 + 48 维 CLIP 降维特征。这两大类特征的来源和性质完全不同：手工特征是人类专家定义了"什么因素可能影响 CTR"然后用程序提取出来的数值；CLIP 降维特征是让 AI 模型替我们"感受"图片的视觉语义并把它压缩为紧凑的数值表示。两者的结合既利用了人类的领域知识（手工特征），又利用了 AI 的感知能力（CLIP 特征）。

```
31 维手工特征包含：
  ├── 10 维品类 one-hot 编码（dress/shoes/tops/accessories/...）
  ├──  1 维价格区间归一化值（0.0-1.0）
  ├──  1 维目标市场编码（us/eu/me/seasia）
  ├──  7 维视觉像素统计（留白、文字密度、饱和度、熵、宽高比偏离、历史CTR、复杂度）
  └── 12 维 CLIP 零样本分类得分（拍摄角度3维 + 模特数量4维 + 质检相关5维）

48 维 CLIP 降维特征：
  CLIP 编码 512 维 → 分块平均降维 → 48 维
```

### 4.2 手工特征详解

**品类 one-hot（10 维）**

为什么要浪费 10 个维度来编码一个品类？品类是图片效果预测中最重要的上下文变量。同一张"白色背景+平铺展示+侧光"的图：

- 如果是连衣裙 → CTR 可能很高（连衣裙品类用户偏好简洁展示）
- 如果是运动鞋 → CTR 可能很低（运动鞋用户想看多角度、户外场景）

不同品类的 CTR 基准值天差地别。连衣裙品类整体 CTR 在 2.5%-4.5% 之间，配饰品类只有 1.0%-2.0%。如果模型不知道这张图是什么品类，它会把连衣裙的正常 CTR 误判为"极高的 CTR"，或者把配饰的正常 CTR 误判为"极低的 CTR"。

One-hot 编码（每个品类一个独立的 0/1 维度）让模型学到了"品类特有的权重因子"——在连衣裙品类中，"模特穿着"这个特征的权重可能是 +0.8%，但在运动鞋品类中，同一个特征的权重可能是 +0.2%（因为运动鞋的展示逻辑和服装不同）。

**价格区间（1 维）**

价格和视觉呈现之间有深刻的互动关系。高价商品（>$25）通常需要更精致的视觉呈现（更好的光影、更高质量的面料展示、更专业感的背景），因为消费者花更多钱时对图片质量的期望更高。如果一张高价位商品的图看起来"廉价"（过度压缩、苍白的光线、简陋的背景），CLIP 可能给不了这个主观判断，但 GBDT 能从历史数据中学到：高价位 + 某些 CLIP 特征的特定组合 → 退货率高（因为图片让商品"看起来不值这个价"）。

**目标市场（1 维）**

不同市场的用户审美差异巨大。东南亚用户偏好暖色调、生活化场景、模特表情亲切的微笑；美国用户偏好冷色调、专业白底图、模特表情自信但不过于情感化；中东用户对模特着装的保守程度有隐性期望。同一个视觉方案在不同市场的表现可能差 50% 以上。市场编码让 GBDT 学到了"在美国市场，极简白底权重 +1.2%；在同个方案在东南亚市场，极简白底权重 -0.3%"这种市场差异化的规律。

**视觉像素统计（7 维）——纯数学计算，不涉及 AI**

这 7 个特征完全不依赖任何模型，是从像素值直接计算出来的统计量。它们衡量的是图片的"物理属性"：

| 特征 | 计算方法 | 物理含义与业务意义 |
|------|----------|-------------------|
| whitespace_ratio | 亮度 > 240 的像素占全图的比例 | 留白越多通常 CTR 越高（干净背景让商品更突出）。Amazon 要求主图白色背景占比 > 85%。但这个规律在生活场景风格中不成立（留白少但 CTR 可能更高，因为场景感增加了吸引力）。GBDT 会从数据中学到"留白"在什么品类中重要、在什么风格中不重要。 |
| text_density | 高对比度边缘的密度 → 推断文字/水印的数量 | 文字多通常 CTR 低（消费者不喜欢图片上有大字遮挡商品），但在促销图中文字是必需的。GBDT 会学习"商品主图文字密度高 = 负面信号，但 promo 品类 = 中性信号"。 |
| saturation_mean | HSV 色彩空间中 S 通道的所有像素均值 | 饱和度适中通常 CTR 最高。饱和度太低（灰蒙蒙）→ 图没有吸引力。饱和度太高（鲜艳刺眼）→ 看起来不真实、像过度 PS。最优区间通常在 0.3-0.6 之间。 |
| image_entropy | 256-bin 灰度直方图的香农熵 | 信息复杂度。熵太低 → 画面太空旷（只有背景没有商品细节）。熵太高 → 画面太杂乱（背景里东西太多）。合适的复杂度取决于品类——连衣裙可以有较高的熵（面料花纹细节），但电子产品通常应该熵较低（简洁展示）。 |
| aspect_ratio_deviation | |宽/高 - 1.0| 的绝对值 | 偏离正方形。大多数电商平台要求主图接近 1:1（正方形）。偏离太多 → 图片被平台裁切 → 商品展示不完整 → CTR 下降。 |
| similar_ctr_mean | 用 CLIP 向量找到最相似的 K 张历史图，取它们的平均 CTR | 这是外推信息的特征——如果这张图和历史上 CTR 很高的那些图在视觉上很相似（CLIP 向量距离近），那它自己的 CTR 也可能高。这个特征在样本量不足时特别有价值——即使租户没有足够多自己的数据，也能通过"和历史上好图有多像"来辅助判断。 |
| complexity | 清晰度评分 / 100 | 图片细节丰富程度的代理指标。和 image_entropy 有相关性但不完全一样——entropy 衡量信息分布的均匀性，complexity 衡量纹理和多尺度的结构细节。对于服装来说，一定的面料纹理复杂度是正面信号（说明图片足够清晰能看到质感）。 |

这 7 个视觉像素统计特征的价值在于：它们是**低成本、可解释、不依赖 GPU 的**。每张图的提取只需要几个毫秒的 CPU 时间（不需要任何 AI 模型），但能捕获大量关键的视觉线索。

**CLIP 零样本分类（12 维）**

这 12 个特征与第二章讲的"零样本分类"是同一个机制：定义几段描述文字，用 CLIP 比较图片向量和这些文字向量的相似度，把相似度作为特征值。但这 12 个分类不是"判断图片好不好"（那是质检和审核的工作），而是"提取关于图片内容的客观事实，作为预测模型的输入"。

特别是拍摄角度（俯拍/平拍/仰拍）和模特数量（无人/一人/2-3人/多人），这些都是已经被电商行业验证过的、和 CTR 高度相关的因素。用 CLIP 零样本分类来提取这些信息，成本几乎为零（不需要额外训练任何模型），但精度足够（在一个已知答案的测试集上，CLIP 判断拍摄角度的准确率约 80-85%）。

### 4.3 CLIP 向量降维：512 → 48

这是特征工程中最关键的一个操作。降维的出发点是：GBDT 不需要也不适合在 512 维上工作。

512 维 CLIP 向量虽然信息丰富，但对于基于决策树的 GBDT 来说是"维度灾难"。树模型寻找最佳分裂点的算法需要遍历所有维度——512 维意味着每个内部节点要做 512 次评估，100 棵树、每棵树几十个节点，总计算量是 512 维 × N 个样本 × 100+ 个节点。而 48 维只有原来的不到 10%。

更重要的是，512 维中大量的维度携带的是冗余信息——CLIP 为了在检索任务上保持鲁棒性（对光照、角度、背景变化不敏感），特意在多个维度上编码了类似的语义，这样即使某些维度被噪声干扰，其他维度仍然能保持正确的语义方向。这种冗余对以图搜图是好事（容错率高），但对 GBDT 是坏事（多余的特征增加了过拟合的风险）。

降维策略的选择也有讲究。SheLook 使用分块平均（将 512 维按顺序分为 48 组，每组约 10-11 个值取平均）而不是 PCA（主成分分析）。原因有三：

1. **计算简单**：分块平均不需要任何训练数据，不需要计算协方差矩阵和特征分解。对于每个租户独立训练的模型来说，不需要额外的降维参考数据。

2. **可解释性**：虽然 512 维 CLIP 向量中的单个维度没有直观含义，但相邻维度之间的相关性假设是成立的（ViT 输出的 patch 特征是局部有序的）。分块平均保留了这种"局部平滑"的特性——每个块的平均值可以理解为"这个语义子区域的大致方向"。

3. **不需要存储降维参数**：PCA 需要存储均值向量（512 维）和投影矩阵（512×48），分块平均只需要存储分组索引——但分组是固定的（从维度 0 到维度 511 每连续约 10 个一组），所以连分组索引也不需要存储。

分块平均的代价是损失约 15% 的方差（信息）。对于 GBDT 来说，15% 的信息损失换来了 90% 的维度压缩和 5-10 倍的训练加速——这笔交易非常划算。

## 五、数据飞轮：从线上数据到更好的模型

这是 SheLook 最核心的长期价值来源。前面三章讲的 AI 能力（CLIP 感知、大模型生图和审核）都是"首次交付"的价值——系统第一次使用时就能提供的能力。数据飞轮提供的是"持续增长"的价值——用一个月之后系统比第一周更"懂"你的业务。

### 5.1 飞轮的四个阶段

```
        ┌──────────────────────────────┐
        │                              │
        ▼                              │
  ① 线上运行                           │
  图片发布到电商平台                     │
  真实消费者看到、点击、购买、退货           │
  所有的行为数据被平台 API 回传         │
        │                              │
        ▼                              │
  ② 数据回流（每天凌晨 2:00 自动执行）     │
  从平台拉取过去 24 小时的图片级表现数据     │
  聚合计算每张图的 CTR、CVR、退货率         │
  写入 daily_metrics 表（支持 upsert）    │
        │                              │
        ▼                              │
  ③ 自动标注（每周日凌晨 3:00 前完成）     │
  对每张图按 CTR 在品类内排序               │
  P75 以上 → 正样本（"好图"）             │
  P25 以下 → 负样本（"差图"）             │
  中间 50% → 不参与训练（中性样本）         │
        │                              │
        ▼                              │
  ④ 模型重训（每周日凌晨 3:00）            │
  用新的正负样本训练三个 GBDT 模型           │
  CTR 预测器 / 爆款分类器 / 退货分类器       │
  保存为新版本到磁盘                       │
  API 进程通过 mtime 检测自动热加载          │
        │                              │
        └──────────────────────────────┘
```

这个闭环每转一圈，模型对"在你的品类和市场里什么视觉特征对应好的商业表现"的理解就更深一层。第一周模型可能只能说出"模特穿着的图比平铺的图 CTR 高"这种粗粒度的规律；到了第十周，模型可能已经学到了"在连衣裙品类、目标美国市场、价格 $20-30 区间，暖色调+侧光+模特微笑的图，CTR 比同品类平均高 32%"这种细粒度的规律。

### 5.2 为什么标注用分位数而不是固定阈值？

这是一个看似微小但后果巨大的设计决策。

**固定阈值的方式：**
```
CTR > 3% → 标记为"好图"（正样本）
CTR < 2% → 标记为"差图"（负样本）

问题：旺季来了（比如黑五、圣诞节），所有品类的 CTR 整体上涨 1-2 个百分点。
      结果几乎所有图都 > 3%（因为整体市场活跃，不是因为图特别好）。
      标注系统把几乎所有图都标记为"好图"——正负样本严重失衡。
      用这样的数据训练的模型，学到的是"几乎所有图都是好图"的错误规律。
```

**分位数的方式：**
```
不管市场涨跌，始终取 CTR 排名前 25% 的图作为正样本，后 25% 作为负样本。

旺季：CTR 整体上涨 → P75 阈值自动从 3.5% 涨到 5.2%
      仍然是前 25% 的图作为正样本，后 25% 作为负样本
      标注的"好"和"差"始终是相对于同品类同期图片的排名，不是绝对数字

淡季：CTR 整体下跌 → P75 阈值自动从 3.5% 降到 2.8%
      标注逻辑不变——仍然是同期的相对排名
```

这个设计的核心洞察是：**"好图"和"差图"在商业上始终是一个相对概念。** 在旺季，所有人 CTR 都高——但"相对于其他同期图片，你的 CTR 排名前 25%"才意味着你的图在竞争中赢了。在淡季，所有人 CTR 都低——但只要你的图排名前 25%，就说明你的视觉策略在当下是有效的。

分位数标注还有一个隐含的好处：它让标注"无视"了外部因素的整体波动。经济衰退、季节变化、平台改版——这些宏观因素让所有人 CTR 同涨同跌。分位数法把这种整体涨跌的影响"归一化"掉了，模型学到的是"哪些视觉特征能在竞争中给你相对优势"，而不是"CTR 的绝对水平是多少"。

### 5.3 额外规则

除了分位数标注，还有一条硬规则：**退货率 > 10% → 无论 CTR 多高，强制标为负样本。** 这是一个业务智慧的体现——CTR 高的图不一定都是"好图"。一张标题党式的图（比如用极具误导性的特效渲染让商品看起来和实物差距极大）可能点击率极高（消费者被吸引进来），但退货率也极高（消费者收到实物发现完全不是图上那样）。这种"高 CTR、高退货"的图从商业角度看是有害的——它不仅浪费了广告费，还损害了店铺的信誉评分。强制将高退货图标记为负样本，让模型学习到"哪些视觉特征会带来退货风险"。

## 六、租户隔离的训练策略

### 6.1 为什么每个租户要独立的模型？

这是一个深刻的原则问题。把所有租户的数据合并起来训练一个"全局模型"，在技术上是可行的（数据量更大、模型可能更稳定），但它在业务上是灾难性的：

```
全局模型（所有租户的数据混在一起训练）：
  租户 FastFashion（快时尚女装）：模特穿着 + 暖色调 → CTR 高
  租户 EliteMens（高端男装）：    模特穿着 + 暖色调 → CTR 低
                                  （男装买家偏好冷色调、专业风格）
  
  全局模型学到：模特穿着 + 暖色调 → 和 CTR 的关系"不明确"
  （因为在全局数据中，这个组合有时好有时差，互相抵消）
  
  结果：FastFashion 得到的建议是"暖色模特图 CTR 不确定"
        但事实上在 FastFashion 的品类和市场里，暖色模特图 CTR 远高于平均
        全局模型把真实存在的规律给"抹平"了
```

租户独立模型避免了这个"辛普森悖论"式的统计陷阱。每个租户在自己的数据中学习自己的规律：

```
FastFashion 的独立模型 → 学到了"我的客户喜欢暖色模特图"
EliteMens 的独立模型   → 学到了"我的客户偏好冷色平铺图"
```

两个矛盾的规律都是"正确的"——因为它们描述了不同目标市场的真实偏好。全局模型无法同时包含这两个矛盾但都正确的规律（除非模型知道以"租户"作为条件变量），而独立模型天然地避免了互相干扰。

### 6.2 新租户的"冷启动"

新租户的数据不够 50 条，独立模型没法训练。SheLook 的处理策略是：

1. **优先使用租户独立模型**（如果训练过且有足够的样本质量）。
2. **退回到全局基线模型**（用所有租户的历史数据训练），同时附带更宽的置信区间（±40% 而不是 ±20%），明确告知运营"数据不够，预测仅供参考"。
3. **一旦租户数据积累到 50 条以上**，触发自动训练，生成租户专属模型，预测精度立即提升（置信区间从 ±40% 缩小到 ±25%），之后随着数据持续积累继续改善。

这个三阶段策略既保证了"新用户第一天就有预测可用"（虽然不太准），又确保了"老用户享受越来越精准的个性化预测"，同时整个过程中零人工干预。

### 6.3 租户模型存放路径

租户 ID 通过 SHA-256 哈希映射为目录名，防止路径中明文暴露租户身份信息：

```
models/
├── global/
│   └── ctr_predictor.pkl            ← 全局基线模型（所有租户共享）
├── tenants/
│   ├── 3f8a2c1b9d4e5f6a7b8c9d0e/   ← SHA256("northstar") 前 32 字符
│   │   ├── ctr_predictor.pkl        ← 当前活跃版本
│   │   ├── ctr_predictor_20260713.pkl  ← 历史版本（用于回滚）
│   │   └── ctr_predictor_20260706.pkl
│   └── 7d1e9f2a4b7c8d0e3f5a6b9c/   ← SHA256("atelier") 前 32 字符
│       ├── ctr_predictor.pkl
│       └── ctr_predictor_20260706.pkl
```

### 6.4 热加载：模型更新无需重启

预测器在每次接收到预测请求时，会检查磁盘上对应租户的模型文件的修改时间（mtime）。如果 mtime 比上次加载时更晚（说明 Celery Worker 在后台训练了新版模型并写入了新文件），预测器自动重新加载新模型到内存。

关键工程点：

- **零停机**：API 进程不重启，不中断正在处理的请求。热加载在单个预测请求的间隙完成——老模型处理完当前请求，接下来自动用新模型处理下一个请求。
- **租户级隔离**：只更新有新版模型的租户的预测器，其他租户的预测器完全不受影响。
- **训练与预测分离**：Celery Worker 负责训练（重计算，GPU 可选），API 进程负责预测（轻推理，纯 CPU，10ms）。训练写入文件，预测读取文件——文件系统是两者之间的"异步通信信道"。

## 七、模型版本管理与回滚

### 7.1 版本策略

每周重训 → 生成新版本文件名 `ctr_predictor_YYYYMMDD.pkl`，最多保留 4 个历史版本，超过 4 个的自动删除最旧的。

`ctr_predictor.pkl` 始终是"当前活跃版本"的副本。回滚时，复制指定日期的历史 pkl 文件覆盖当前 `ctr_predictor.pkl`。

### 7.2 为什么需要回滚？

场景示例（真实发生过的情况）：

```
旧模型（v20260706）：预测 CTR 2.85%，实际 CTR 2.91% → 误差 0.06%（很好）
新模型（v20260713）：预测 CTR 4.12%，实际 CTR 2.73% → 误差 1.39%（灾难）

原因是上周搞了一场大型促销活动（Prime Day 级别），促销期间的 CTR 异常高
（对比平时约高 1-2 个百分点），大量促销图片被分位数标注系统标记为"正样本"，
但它们的高 CTR 是因为促销本身而不是因为图片好。
这些"伪正样本"污染了训练数据，模型学错了规律。
```

回滚操作：调用管理 API 或后台操作，选择目标日期（如 20260706），系统复制 `ctr_predictor_20260706.pkl` 覆盖 `ctr_predictor.pkl`。下一次预测请求自动使用老模型。新模型文件不删除——保留供人工分析问题原因。

### 7.3 监控指标

正常运作的数据飞轮应该呈现几个趋势：
- 预测模型的误差（MAE 或 RMSE）随时间逐渐降低（表示模型越来越准）
- 爆款分类器的准确率随时间提升（从最初的 60% 左右到 75-85%）
- 退货风险预测的 F1 score 随着数据积累改善

如果这些指标突然恶化（如误差增加 200% 以上），触发回滚流程。

## 八、预测输出：不只是"一个数字"

### 8.1 CTR 预测的完整输出结构

```json
{
    "predicted_ctr": 0.0342,
    "normalized_ctr": 1.28,
    "confidence_interval": {
        "lower": 0.0274,
        "upper": 0.0410
    }
}
```

三个数字各有各的用途。

**predicted_ctr（点估计）**：最直观的数字。运营看到"3.42%"就知道大概的水平。但点估计单独使用时有一个陷阱——它没有告诉你"预测有多不确定"。

**normalized_ctr（品类归一化后的 z-score）**：用来跨越不同品类做比较。不同品类的 CTR 自然水平差异巨大——连衣裙品类平均 2.5%，配饰品类平均 1.2%。一张连衣裙图 CTR 2.5% 和一张配饰图 CTR 1.8%，哪个更好？看 normalized_ctr：连衣裙的 z-score 是 (2.5-2.5)/1.0=0（刚好在平均线），配饰的 z-score 是 (1.8-1.2)/0.5=1.2（远高于平均线）。结论：配饰的那张图虽然绝对 CTR 更低，但在它的品类中竞争力更强。

**confidence_interval（置信区间）**：告诉运营"我们都多确定这个数字"。通过一个简化的 ±20% 公式计算。实际的统计学置信区间需要结合样本方差和 t-分布计算，但 ±20% 的工程近似对于业务决策已经足够——运营看到 3.42%（2.74%-4.10%）就知道"大概在 3% 到 4% 之间"，这比一个孤零零的"3.42%"有用得多。如果置信区间很宽（比如样本量少时可能会到 ±40%），运营就知道"数据还不多，仅供参考"。

### 8.2 退货风险的降级策略

退货风险有两套方案并行：

- **模型方案**：训练了专门的退货风险分类器，输入 79 维特征，输出高风险概率。
- **启发式方案**：`风险分 = CTR × 100 × (1 - 爆款概率)`——这个公式的业务直觉是：高点击率但爆款概率低的图，往往意味着"图看起来很好但实物跟不上"，就是典型的退货风险高。比如一张效果图渲染得特别漂亮（高 CTR），但实际商品的做工无法达到图中的质感水平（爆款概率低 → 说明类似的图历史上消费者买完后悔了）。

当退货分类器尚未训练或训练失败时，使用启发式方案作为兜底。启发式虽然不如训练好的模型精确，但它是建立在业务直觉上的、零训练成本的替代方案。

## 九、与前面章节的技术衔接

数据飞轮不是SheLook的一个独立模块，而是连接了前三章所有 AI 能力的"闭环引擎"：

- **第二章的 CLIP** 提供了图片的视觉编码——飞轮的 79 维特征中，48 维来自 CLIP。没有 CLIP，GBDT 只能看到像素统计，无法感知风格、构图、美学等深层视觉信息。
- **第三章的大模型** 生成图片——飞轮的"起点"是生图，"终点"是预测这张图的效果。生图 + 预测 = "先创造多种可能性，再用数据选择最优解"。
- **第五章的 A/B 实验** 是飞轮的验证环节——GBDT 预测"这张图可能点击率 3.4%"，实验告诉你"实际上它确实比另一张高 0.6%"。预测 + 实验 = "先估算，再验证"。实验数据也是飞轮最重要的真实反馈来源——实验的 CTR 数据比自然上线的 CTR 数据更干净（不受商品位、时间段等混淆因素干扰）。

## 十、本章小结

1. **SheLook 自己训练的是一个 GBDT 预测模型**，回答三个问题：CTR 多少、会不会成爆款、退货率高不高。它是系统的"决策大脑"。

2. **大模型不适合做数值预测**——没有数值精度、不可验证推理过程、不可复现。GBDT 的记忆能力（"在你的历史数据中什么规律成立"）是更匹配的解决方案。

3. **GBDT 的思想是"一百棵弱树的精诚合作"**——每一棵新树修正前几棵树的残差，最终逼近真实规律。选择 HistGradientBoosting 而非 XGBoost 是为了零 C++ 编译依赖（Docker 部署的可靠性 > 训练速度的 2 秒优势）。

4. **79 维特征 = 31 维手工（人类专家提取） + 48 维 CLIP 降维（AI 视觉感知）**，两者各有分工。512 维 CLIP 向量通过分块平均降到 48 维，代价是 15% 方差损失，换来 90% 维度压缩和显著的反过拟合效果。

5. **数据飞轮的四个阶段**——线上运行 → 数据回流 → 分位数自动标注 → 模型重训——完全自动化，零人工干预。分位数标注的"相对论"适应了市场波动，退货率硬规则过滤了"高 CTR 高退货"的陷阱。

6. **租户隔离训练**避免了"辛普森悖论"——不同租户的客户群偏好不同，全局模型会"抹平"真实存在的差异化规律。

7. **热加载 + 版本回滚**实现了零停机更新，每周重训一次，API 进程通过文件 mtime 检测自动加载新模型。4 个历史版本保留用于回滚。

下一章预告：**A/B 实验与效果归因**——预测说"好"还不够，要让真实数据来说话：UCB 算法做流量分配、Z-test 做统计显著性检验、多维度下钻避免辛普森悖论。
