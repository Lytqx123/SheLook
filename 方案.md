# SheLook 当前产品与系统方案

> 文档基线：2026-07-23。本文与 `重构方案/07-企业数据接入与CTR闭环六阶段路线图.md` 共同构成当前方案说明。

## 方案目标

SheLook 的当前实现围绕一个明确的产品中心：帮助跨境电商团队把“做素材”升级为“做视觉经营决策”。系统把商品视觉从生产、审核、预测、验证到复盘连接为一条可追溯链路，让团队能在每次活动中明确目标、处理风险、验证结果并沉淀可复用策略。

本文件描述仓库已经落地的工作方式与运行边界，不将容量目标、未来集成或尚未完成的验证写成既成能力。

2026-07-23 的本机部署复核确认：迁移 `019`、PostgreSQL、Redis、MinIO、API/Nginx 路由、分队列 Celery、前端生产服务和观测组件可以协同启动。异步任务通过 fork 后连接池重置与任务结束前释放异步连接，避免连接跨事件循环复用；这补齐了 Outbox 周期分发的运行闭环。业务供应商 API 已统一迁移到受权限保护的网页配置中心，运行时按租户读取加密配置。

## 已落地的业务闭环

~~~mermaid
flowchart LR
    A["运营活动<br/>商品、市场、目标"] --> B["策略与素材生产"]
    B --> C["质量审核<br/>自动检查与人工决策"]
    C --> D["效果预测<br/>机会、风险、建议"]
    D --> E["A/B 实验<br/>真实流量验证"]
    E --> F["指标与活动复盘"]
    F --> G["洞见、样本与模型学习"]
    G --> A
~~~

活动并不取代商品、素材、审核或实验这些既有实体，而是把它们聚合到同一条经营主线中。活动详情会读取关联的商品、方案、生成素材、审核记录、预测记录、实验、指标与洞见，并给出当前待办和下一步建议。

## 产品结构

| 层次 | 当前职责 |
| --- | --- |
| 日常运营 | 运营活动、行动式首页、商品管理、发品工作台、审核工作台、任务中心和视频生成。 |
| 经营决策 | 预测比较、A/B 实验、供应商协同。 |
| 分析与治理 | 公平性、聚类、指标与模型、数据飞轮、审计日志。 |
| 组织控制 | 租户、成员、角色、组织单元、权限、配额和功能开关。 |

默认首页优先呈现用户今天需要处理的事项，而不是将所有指标和图表平铺展示。运营、审核、分析、供应商和管理角色看到的导航和行动重点不同；高级治理能力保持可访问，但不干扰高频业务路径。

## 系统结构

~~~mermaid
flowchart TB
    U["运营、审核、分析、供应商、管理者"] --> F["Next.js 前端"]
    F --> A["FastAPI API"]
    A --> T["认证、组织、权限与租户上下文"]
    A --> P["PostgreSQL + pgvector"]
    A --> R["Redis"]
    A --> O["MinIO"]
    A --> W["Celery 工作队列"]
    W --> G["生成、审核、预测、实验、飞轮任务"]
    G --> X["外部模型与平台适配器"]
    A --> M["Prometheus 指标与结构化日志"]
~~~

同步 API 承担业务状态、授权和查询；耗时操作进入 Celery 队列。现有 Compose 配置将编排、生成、模型和分析任务分开运行。PostgreSQL 保存业务事实，MinIO 保存对象，Redis 用于任务、缓存、限流和 Pub/Sub；Prometheus、Grafana、Flower 提供运行观察入口。

## 关键设计取舍

### 用活动串联模块，而不是重做所有模块

商品、方案、生成图片、审核记录和实验仍是各自领域的事实来源。视觉运营活动通过受校验的引用把它们关联起来，避免一次性改写所有存量数据模型，同时让用户获得完整的经营上下文。

### 人工决策保留在高风险节点

自动审核和预测用于减少筛选成本、给出依据和风险提示。审核人员仍可做最终质量判断；实验与经营结果用于验证策略，系统不把预测结果伪装为真实收益，也不在未经确认的情况下替用户完成外部经营操作。

### 权限与数据隔离在服务端闭环

前端只负责可见性和操作引导，后端才是授权来源。业务表以 tenant_id 为边界，成员关系和权限控制访问范围，PostgreSQL 行级安全在生产环境提供额外保护。外部 SSO 或飞书身份必须映射到本地租户和成员关系。

### 业务 API 在网页配置，部署密钥留在运行环境

管理员可在“系统集成 / 外部 API 配置”维护视频生成、图片生成、AI 审核和现有指标同步提供商。浏览器只提交一次性写入字段；后端以 `provider_configs` 按租户加密保存并通过 RLS 隔离，运行时才解密使用。数据库、Redis、对象存储、JWT、TLS、监控入口与加密根密钥不是业务供应商参数，仍必须由部署 Secret 管理，不能迁入浏览器。

### 数据飞轮必须让运营人员可感知

活动的时间线和洞见不是独立的“模型页面数据”：它们把审核、预测、实验和经营反馈沉淀为可复盘的策略记录。数据飞轮模块继续负责样本同步、训练与版本管理，使反馈可以进入下一轮决策。

## 部署与运行方案

日常开发、预发布和生产环境均以 Docker Compose 为基础，使用根目录的 setup.ps1 或 setup.sh 管理环境文件、镜像、独立迁移、对象存储初始化和健康检查。开发环境从源码构建；预发、生产环境只接受带 digest 的不可变镜像，并分别使用 `.env.staging`、`.env.prod`。Compose 包含 PostgreSQL、Redis、MinIO、FastAPI、Celery Worker/Beat、Next.js、Nginx、Prometheus 和 Grafana；Flower 在开发环境可用，在预发、生产仅作为内部 `ops` profile 按需启动。PgBouncer 与本地 SD WebUI 为可选 profile。演示数据默认不写入，只有 `dev` 下显式传入 `-SeedDemo`/`--seed-demo` 并二次确认才会填充；数据脚本只允许在 development/test 环境执行。

当前迁移头为 `019`。API 和 Worker 容器不再在启动时迁移数据库，独立 `migrate` 服务持有迁移连接并在应用启动前完成 schema 更新。预发、生产覆盖会关闭生成模拟、移除源码挂载和非必要端口，并要求企业认证、显式租户映射、RLS、受限数据库账号、C2PA 证书/私钥文件、指标密钥文件、独立凭据加密根密钥与受控迁移连接；缺少任一关键条件时，启动脚本会在不输出密钥值的前提下拒绝部署。

Kubernetes 清单与部署说明位于 [deploy/kubernetes/README.md](./deploy/kubernetes/README.md)。

## 当前验收边界

当前仓库已包含多租户和权限基础、工作流与 Outbox、活动聚合、角色化前端、异步任务、指标与模型模块，以及性能/数据一致性验证脚本。它们说明系统具备企业化演进的结构基础，但不等同于“已经验证可支撑固定规模的并发用户”。

任何面向大规模团队的上线结论都应至少基于目标硬件、目标数据量、真实身份接入、外部服务配额和压测报告共同确认。压测方法、指标口径和已知验证限制请查看 [PERFORMANCE.md](./PERFORMANCE.md)。
