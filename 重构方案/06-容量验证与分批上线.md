# 第六阶段：容量验证、迁移与分批上线（当前状态）

> 文档基线：2026-07-23。容量数据必须通过目标环境演练获得，不能从本地 Compose 健康状态推导。

本文定义当前版本的发布与容量验证边界。SheLook 已具备压测、数据一致性与发布门禁脚本，但尚未产生可以直接证明“1000 人以上团队已通过生产容量验收”的报告；在得到目标环境的实测证据前，不能作出该结论。

## 已具备的验证与发布基础

| 能力 | 当前实现 | 用途 |
| --- | --- | --- |
| 只读并发压测 | `backend/scripts/phase6/run_read_load.py` | 无额外依赖地压测健康、看板与商品列表，并输出 P95/P99、错误率和吞吐 |
| 业务压测场景 | `scripts/locustfile.py` + `perf` 依赖 | 覆盖已认证的看板、商品、实验、审核、审计和模型读取；写操作需显式开启 |
| 数据一致性检查 | `backend/scripts/phase6/verify_data_integrity.py` | 校验迁移版本、租户外键、跨租户关联、配额与 RLS |
| 发布门禁 | `backend/scripts/phase6/release_gate.py` | 汇总负载报告和一致性报告，按请求量、错误率、P95 与数据一致性生成可审计结论 |
| 交付基础 | CI、Compose、Kubernetes Job/HPA/PDB/探针 | 运行质量检查、迁移和受控滚动发布 |
| 灰度控制 | 租户级功能开关和生成/预算配额 | 按租户启停 AI 生成、视频生成与自动实验 |

## 当前必须如实保留的限制

1. `verify_data_integrity.py` 已与当前迁移头 `018` 对齐，并覆盖 `visual_operation_campaigns`、`campaign_insights`、经营事实、效果事实、预测快照和反馈标签的租户外键、关联与 RLS 检查。它仍只是发布门禁的一个输入，必须在目标 PostgreSQL 上实际执行并归档结果。
2. 仓库提供测试工具和阈值默认值，但没有提交某一目标硬件、数据库规格、外部 AI 配额和真实数据规模下的压测报告。默认阈值不是容量承诺。
3. Kubernetes 清单提供扩缩容与可用性骨架，但不包含生产入口、跨可用区数据库、备份服务、密钥轮换或告警接收方。部署清单不能替代灾备演练。
4. 生成、平台同步和企业登录依赖外部服务。压测应将它们与普通读路径分开计算，不能让第三方波动掩盖或虚构 API 容量结论。

## 当前推荐的验收顺序

在 `backend` 目录、已连接目标 PostgreSQL/Redis 且已完成数据脱敏的预发环境执行：

```powershell
# 1. 代码与迁移基线
uv run ruff check .
uv run pytest
uv run alembic upgrade head

# 2. 当前完整性校验已覆盖迁移 head 018、活动表和反馈数据表
uv run python -m scripts.phase6.verify_data_integrity --output artifacts/integrity.json

# 3. 只读并发基线；生产/认证环境应传入有效 token 和 tenant_id
uv run python -m scripts.phase6.run_read_load `
  --base-url https://staging.example.com `
  --concurrency 20 `
  --duration-seconds 300 `
  --tenant-id <tenant-id> `
  --token <access-token> `
  --output artifacts/read-load.json

# 4. 生成发布门禁报告；阈值要按目标环境和业务 SLO 审批
uv run python -m scripts.phase6.release_gate `
  --load-report artifacts/read-load.json `
  --integrity-report artifacts/integrity.json `
  --minimum-requests 1000 `
  --max-error-rate 0.005 `
  --max-p95-ms 500 `
  --output artifacts/release-gate.json
```

需要模拟更接近真实用户行为时，安装 `perf` 依赖并运行 Locust。默认场景只读；预测和生成写入必须显式设置 `SHELOOK_ENABLE_MUTATIONS=true`，并传入专用测试素材 ID，禁止在生产数据上无控制地启动写压测。

```powershell
uv sync --locked --extra dev --extra perf
uv run locust -f ..\scripts\locustfile.py --host https://staging.example.com --headless -u 50 -r 5 -t 10m --csv artifacts/locust
```

## 分批发布与停止条件

将新能力按租户和角色分批开放：先在内部受控租户验证活动主线、审核、任务恢复和模型解释，再扩大到真实业务租户。每一批都应保留以下证据：迁移版本、租户隔离报告、关键 API 与 Worker 健康、错误率/P95、队列积压、预算拒绝率、审核阻塞量、活动完成率和已知外部依赖异常。

出现以下任一情况，应暂停扩大范围并回到问题定位，而不是继续加机器或扩大用户数：

- 发布门禁未通过、迁移版本不一致、RLS/租户引用校验失败；
- 读路径 P95、错误率或数据库连接等待超出已批准阈值；
- Outbox/生成队列持续积压，失败任务无法在任务中心解释或恢复；
- AI 预算预留异常、功能开关无法即时生效，或权限/会话撤销出现跨成员影响；
- 外部平台或模型提供商不可用，却被页面错误显示为生成、同步或投放成功。

## 数据迁移与回退原则

- 数据库结构通过独立迁移 Job 执行，API/Worker 不持有 schema-owner 账号。迁移前备份并验证恢复路径；迁移后运行健康、隔离和核心业务回归检查。
- 采用“扩展—回填—切换—收缩”的兼容方式。新字段与新路由可以先并行存在，数据和访问链路未验证前不删除旧字段。
- 应用版本可以回退，但已提交的数据迁移、活动洞察和异步事件不能靠简单回退镜像“撤销”。需要针对每个迁移准备前向修复或经验证的 downgrade 策略。
- 仓库没有自动化 PostgreSQL/MinIO 备份恢复编排。实际生产必须使用受管备份或等效方案，并定期在隔离环境执行恢复演练。

## 容量结论的正确表达

当前系统具备面向大团队的隔离、队列、限流、配额、缓存、探针与扩缩容基础；这说明可以进入分批验证，而不是已经证明能够服务任意 1000 人组织。只有在完成目标环境压测、验证外部依赖限额、完成恢复演练，并由发布门禁留下可复现报告后，才可以对外给出相应规模的容量承诺。
