# 第六阶段验收运行手册（当前版本）

> 文档基线：2026-07-23。验收先执行迁移和就绪检查，再进行带权限、租户和回滚边界的业务验证。

本手册用于在**预发或隔离测试环境**验收 SheLook 当前版本。它以当前代码的实际脚本为准，不代表已完成千人规模生产验收。

## 0. 先确认边界

- 使用脱敏或合成数据；不要把真实生产凭据、真实用户 token、生产对象桶或生产 AI 预算用于写压测。
- 目标环境应启用认证、受限 PostgreSQL 运行账号、`TENANT_ENFORCEMENT_ENABLED=true`、`TENANT_RLS_ENABLED=true`，并将 `ALLOW_GENERATION_MOCKS=false`。
- API/Worker 的 `DATABASE_URL` 不能使用 schema-owner、超级用户或 `BYPASSRLS` 账号；迁移作业才使用独立的 `DATABASE_MIGRATION_URL`。
- 当前一致性脚本已对齐迁移头 `018`，并覆盖 `visual_operation_campaigns`、`campaign_insights`、经营事实、效果事实、预测快照和反馈标签的租户外键、关联与 RLS 检查。仍应在目标 PostgreSQL 上执行并保存报告，不能以源码常量代替实际隔离验收。

## 1. 交付物和服务健康

```powershell
# 在仓库根目录：检查编排配置
docker compose -f docker-compose.yml -f docker-compose.staging.yml config --quiet

# 在 backend 目录：检查后端质量与迁移
uv run ruff check .
uv run pytest
uv run alembic upgrade head

# 在 frontend 目录：检查前端生产构建
npm ci
npm run lint
npx tsc --noEmit
npm run build
```

启动后确认：

```powershell
Invoke-WebRequest https://staging.example.com/api/health/live
Invoke-WebRequest https://staging.example.com/api/health/ready
Invoke-WebRequest https://staging.example.com/api/health
```

存活探针只说明进程响应；就绪和综合健康还应确认数据库、Redis、Worker、对象存储及外部服务配置满足本次验收范围。

## 2. 租户、登录与权限验收

1. 为两个独立租户创建最小测试数据，分别使用运营、审核、分析、供应商和只读成员登录。
2. 验证同一成员仅能读取本租户的商品、活动、素材、任务、审核、预测、实验、指标和审计记录。
3. 验证前端菜单会按角色收敛，但更要直接调用 API，确认无权限的读写返回 `403`。
4. 禁用或降权成员后，用其既有 token 验证会话撤销是否生效；生产环境 Redis 故障时，权限和成员变更应按失败关闭处理。
5. 使用 OIDC 或飞书时，验证外部组织只通过受控映射进入本地租户，未受邀成员、未知 `tenant_key` 或未知 OIDC claim 必须被拒绝。

## 3. 视觉运营活动业务闭环

对一个隔离租户执行以下最小业务场景：

1. 新建视觉运营活动，选择商品、市场和可量化经营目标。
2. 从活动进入方案/素材生产，提交生成任务并记录任务 ID；确认生成没有把“已提交”显示成“已完成”。
3. 完成自动/人工审核，检查质量与合规结论、审计追踪和活动阶段是否能解释下一动作。
4. 运行预测，确认决策卡展示证据、置信度、风险和推荐动作；必要时从两个候选素材创建 A/B 实验。
5. 写入或同步获授权的经营指标，确认实验和活动详情反映真实数据状态而非填补数值。
6. 在活动复盘中写入洞察，确认其可追溯到素材、预测、审核、实验或指标来源。

## 4. 可靠任务验收

- 重复提交同一图片生成请求（相同幂等键），确认只复用同一个工作流任务和预算预留。
- 在 Outbox 待发布、队列中、失败和已运行状态分别测试任务中心的取消/重试限制。
- 人为停止一个 Worker 后恢复，检查任务、Outbox 和素材状态是否可通过任务 ID、请求 ID 和审计 ID 解释。
- 模拟外部生成超时、Redis Pub/Sub 不可用和预算耗尽；页面应展示等待、失败或降级状态，而不是伪造成功。
- 确认只有失败的图片生成任务支持任务中心人工重试；其他任务类型尚未统一接入该恢复入口。

## 5. 数据一致性与性能门禁

在目标环境完成迁移后，执行：

```powershell
# backend 目录
uv run python -m scripts.phase6.verify_data_integrity --output artifacts/integrity.json

uv run python -m scripts.phase6.run_read_load `
  --base-url https://staging.example.com `
  --concurrency 20 `
  --duration-seconds 300 `
  --tenant-id <tenant-id> `
  --token <access-token> `
  --output artifacts/read-load.json

uv run python -m scripts.phase6.release_gate `
  --load-report artifacts/read-load.json `
  --integrity-report artifacts/integrity.json `
  --minimum-requests 1000 `
  --max-error-rate 0.005 `
  --max-p95-ms 500 `
  --output artifacts/release-gate.json
```

门禁中的 1,000 请求、0.5% 错误率和 500 ms P95 只是脚本默认值。验收负责人必须根据机器规格、数据库连接预算、租户数、数据量、外部依赖和业务 SLO 审批实际阈值。不能以本地一次通过替代生产容量证明。

## 6. 发布、观察与回退

1. 先运行独立迁移 Job，等待完成后再滚动 API/Worker；不要把迁移 owner 密钥挂载给运行工作负载。
2. 先向内部或白名单租户开放新功能，再使用租户功能开关扩大范围；AI 生成、视频生成和自动实验均可单独熔断。
3. 观察 Prometheus/Grafana、结构化日志、健康探针、队列积压、Outbox 失败、数据库连接池、预算拒绝和审核积压。
4. 若隔离校验、发布门禁或核心业务回归失败，停止扩大范围。优先关闭相应租户功能开关或回退应用镜像；对已执行的数据库迁移采用经过验证的前向修复，而不是盲目回退数据。
5. 发布前后均应保留备份与恢复演练证据。仓库没有自动化数据库/对象存储备份编排，生产团队需提供受管备份和恢复时限承诺。

## 验收输出

一次可追溯验收至少应归档：版本与镜像标识、迁移版本、环境与数据规模、身份/租户测试结果、核心闭环记录、任务故障注入结果、`integrity.json`、负载报告、`release-gate.json`、监控截图或链接、已知风险及审批结论。缺少这些证据时，只能说明“功能可运行”，不能说明“已完成生产规模验收”。
