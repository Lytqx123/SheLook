# 企业数据接入与 CTR 闭环：六阶段开发路线图

> 文档基线：2026-07-23。路线图中的“真实同步”以企业获得供应商授权接口契约为前提。

> 状态：已启动（2026-07-23）  
> 范围：优先完成店小秘接入、可由 Web 管理的运行时配置，以及真实 CTR 驱动的模型校正；马帮、易仓、领星不在本轮实现范围内。  
> 架构原则：保持“模块化单体 + 独立 Worker”，不为本次改造拆分微服务。

## 目标与边界

系统将从“基于素材的 CTR 预测工具”演进为“视觉运营决策平台”：店小秘提供商品、刊登、库存、订单和履约等经营事实；店铺分析或广告/BI 数据源提供真实曝光、点击和 CTR；两类数据在统一映射后分别服务于经营分析和模型校正。

店小秘的商品、订单和库存数据**不能替代**真实 CTR。CTR 必须由可提供 `impressions` 与 `clicks` 的店铺分析、广告平台或企业 BI 数据源单独接入；缺少其中任一项时，系统不得推算或伪造 CTR。

“转到 Web 配置”仅适用于业务运行时配置和第三方业务凭据。数据库连接、Redis、消息队列、根加密密钥、JWT/会话签名密钥、TLS 证书与部署端点等启动和平台安全密钥，仍必须放在 `.env`、Docker/Kubernetes Secret 或受管密钥服务中，不能暴露到浏览器或普通 Web 配置页。

## 目标架构

```text
Web 配置中心（权限、版本、审计、只写密钥）
                |
                v
运行时配置服务 ──> Redis 配置失效事件 ──> API / Worker 重新加载
                |
                v
店小秘适配器 ──> 同步任务 / 游标 / 原始响应留痕
                |
                v
外部实体映射 ──> 统一经营事实（商品、刊登、库存、订单、履约）
                |
真实效果数据源 ──> 曝光、点击、真实 CTR、成熟窗口标记
                |
                v
预测快照 + 真实标签 ──> 校准、评估、受控再训练
```

## 六个阶段

| 阶段 | 主题 | 状态 | 阶段出口 |
| --- | --- | --- | --- |
| 1 | 安全接入基座与店小秘连接管理 | 已完成 | 可在 Web 中安全保存、更新、校验和撤销店小秘连接；密钥不回显，有租户隔离与审计 |
| 2 | 统一运行时配置中心 | 核心已完成 | 白名单租户业务配置可从 Web 版本化保存、恢复默认，并由 API/Worker 在执行时解析 |
| 3 | 统一经营事实与外部实体映射 | 核心已完成 | 外部 ID 映射、幂等经营事实和真实效果事实已独立建模，重复同步不会重复创建当前事实 |
| 4 | 店小秘真实同步适配器 | 框架已完成；等待供应商契约 | 同步任务、运行记录、游标字段、幂等写入和明确的契约门禁已上线；真实网络调用等待授权资料 |
| 5 | 真实 CTR 接入与模型反馈闭环 | 核心已完成；等待真实效果数据 | 预测快照、成熟效果事实和反馈标签已分离，飞轮不再用实际值覆盖预测记录 |
| 6 | 生产治理与多平台扩展准备 | 部署基线已验证 | 迁移、容器构建、健康检查和契约门禁已验证；运营 SLA/告警需在真实数据上线后按基线配置 |

## 阶段 1：安全接入基座与店小秘连接管理（已完成）

### 已交付

- 增加租户级店小秘连接、同步运行记录模型和数据库迁移 `016`。
- 增加 `/api/integrations/dianxiaomi` 与集成中心页面；支持连接创建、编辑、连接本地校验和删除。
- API Key、Secret、Access Token 只可写入；后端使用 Fernet 加密保存，响应和审计日志不返回密钥明文。
- 已记录配置版本、操作者、时间、店铺引用、同步范围和间隔；只有管理员或 `tenant:manage` 权限可操作。

### 刻意未做

尚未假定店小秘存在某个公开的通用第三方 ERP API 或猜测签名规则。因此“校验”目前只验证本地配置和凭据可解密，不会宣称已连通或同步成功。

## 阶段 2：统一运行时配置中心（核心已完成）

### 要实现的能力

1. 已建立租户级白名单运行时配置模型与修订历史：`ctr.dashboard_baseline`、`experiments.completion_impressions`、`ctr.minimum_mature_impressions`。
2. 已提供 `/api/runtime-settings` 的读取、更新、恢复默认和历史接口，以及管理员 Web 页面；配置无须重启容器。
3. API 在看板计算时解析 CTR 基线及其版本，后台实验/反馈任务在执行时解析相应阈值；看板缓存键包含配置版本，避免旧缓存显示新配置。
4. 店小秘连接级业务配置仍属于连接模型；凭据仅可写、后端加密、不会进入运行时配置值或浏览器响应。
5. 平台级跨租户覆盖策略尚未实现，因为需要明确平台管理员身份和租户覆盖优先级，不能由当前租户管理员越权设置。

### 验收标准

- 一次配置变更不重启容器也能被相应 API 和 Worker 识别，并能查询到两端使用的配置版本。
- 未授权用户、跨租户用户和审计查询均无法读取密钥明文。
- 失败的配置解析不会覆盖最近一次有效配置，且会留下可操作的错误状态。

## 阶段 3：统一经营事实与外部实体映射

### 要实现的能力

- 建立 `ExternalEntityMapping`：以 `provider + connection + shop/account + marketplace + external ID` 唯一定位，并关联内部商品、刊登、SKU、图片/素材和后续创意。
- 建立可追溯的原始同步记录和标准化事实层。原始响应只用于排错/回放并按保留策略脱敏；业务查询使用标准化商品、刊登、库存、订单和履约事实。
- 统一幂等键、游标、水位线、时区、币种和删除/下架语义；把“外部实体未映射”作为明确的数据质量问题，而不是静默丢弃。
- 提供人工映射和建议映射流程，任何自动匹配都必须可修正、可审计。

### 验收标准

- 同一批外部数据重复导入不会创建重复的商品、订单或映射。
- 每条标准化事实都可追溯至连接、同步批次和外部 ID。
- 仪表盘能展示未映射率、重复率、字段空值率和同步延迟。

## 阶段 4：店小秘真实同步适配器

### 前置条件（需要企业提供）

- 店小秘为当前企业开通的接口地址、应用/商户授权方式、签名算法、字段契约、分页/增量游标规则、限流规则和测试账号。
- 首批接入范围确认：建议顺序为“商品与刊登 → 库存 → 订单 → 履约”，而非一次性接入全部范围。

### 要实现的能力

- 用 Provider Adapter 接口封装认证、请求签名、分页、限流、重试、错误分类和字段转换；网络调用只在 Worker 中执行，不阻塞 Web 请求。
- 使用连接级锁、幂等写入、持久化游标和退避重试，避免并发重复同步或部分失败造成的数据错乱。
- 提供首次全量同步、定时增量同步、指定范围补数、失败重试和人工暂停/恢复；每次运行生成可审计的 `SyncRun`。
- 对接口权限不足、字段变化、限流、授权失效和映射缺失给出不同状态和处置建议，绝不显示为“同步成功”。

### 验收标准

- 使用店小秘测试授权完成端到端的首批范围同步；至少覆盖一次全量、一次增量、一次失败重试和一次游标续跑。
- 同步结果与店小秘侧抽样数据可对账，差异能定位到字段、时间窗或映射。

## 阶段 5：真实 CTR 接入与模型反馈闭环

### 数据定义

真实 CTR 统一定义为：`sum(clicks) / sum(impressions)`，必须在相同的店铺、市场、刊登/创意、归因窗口和日期粒度上聚合。不得平均每日 CTR，也不得将订单量、访问量或店小秘履约数据当作点击/曝光。

### 要实现的能力

- 接入实际效果数据源（优先企业已有 BI 导出/API 或已授权的店铺/广告分析数据），保存原始曝光、点击、花费、订单等事实及其来源、延迟和口径版本。
- 将效果事实通过阶段 3 的映射关联到刊登和素材；映射缺失或粒度不足的数据不能进入训练标签。
- 每次预测保存模型版本、特征版本、预测值、预测时间和实体快照；标签达到最小曝光量与成熟窗口后才进入评估。
- 区分模型监控、校准和再训练：先计算偏差/分桶校准和漂移，再由受控任务产生候选模型，人工或发布门禁决定上线。

### 验收标准

- 能对每个进入评估的样本追溯“预测时看到了什么”和“真实 CTR 来自哪里”。
- 仪表盘同时展示覆盖率、样本量、成熟度、校准误差、按市场/店铺/类目的偏差；样本不足时明确显示不可结论。
- 不会因延迟数据、映射变化或模型版本变化而回写篡改历史标签。

## 阶段 6：生产治理与多平台扩展准备

### 要实现的能力

- 建立连接健康、同步 SLA、数据质量、配置版本、密钥轮换、模型版本和反馈质量的统一运营看板与告警。
- 设置数据保留、脱敏、最小权限、导出审批、审计检索、灾难恢复演练和回滚手册；定义供应商接口异常时的降级策略。
- 将共享能力固定为 Provider Adapter、配置中心、映射、事实层和同步运行框架。马帮、易仓、领星未来只新增适配器、字段映射和授权页，不复制同步调度或 CTR 闭环代码。
- 只有店小秘与真实 CTR 链路经过稳定运行和对账后，才评估接入其他 ERP；本阶段不承诺它们的具体接口实现。

### 验收标准

- 故障、数据异常、配置回滚和密钥轮换均有演练记录和可执行 Runbook。
- 新 Provider 的接入工作量主要集中在供应商协议和字段映射，而非修改预测、权限或任务核心逻辑。

## 当前执行顺序与决策点

本轮立即按“阶段 2 → 阶段 3 → 阶段 4 → 阶段 5 → 阶段 6”推进；阶段 1 已作为现有基座交付。阶段 4 需要真实店小秘接口资料和测试授权，缺失时可以继续完成框架、映射和模拟契约测试，但不能声称完成真实同步。

当前不存在需要业务方裁决的功能冲突。后续若企业希望把某个**平台级**业务配置自动下发到所有租户，或允许租户自行覆盖它，将先提交明确的覆盖优先级与权限方案供确认；该选择会影响数据隔离与责任边界，不能擅自决定。

## 不做的事项

- 不把数据库、JWT、主加密密钥等基础安全配置放入前端页面。
- 不根据猜测的店小秘 URL、签名或字段编写“真实同步”。
- 不将 ERP 订单/库存事实冒充曝光、点击或 CTR。
- 不在本轮接入马帮、易仓、领星，也不为此拆分微服务。
