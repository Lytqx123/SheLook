# 第三阶段：工作流与异步任务可靠性（当前状态）

> 文档基线：2026-07-23。异步任务均须显式绑定租户上下文；外部同步不会在请求线程中执行。

本文记录当前已经接入的可靠任务链路，以及它明确尚未覆盖的范围。它不把设计中的通用编排能力误写为现有功能。

## 当前结论

SheLook 已把图片生成的提交、任务记录、预算预留和异步投递拆成可恢复的事务链路：API 在同一数据库事务内创建素材占位记录、`workflow_tasks` 记录、`outbox_events` 事件和 AI 用量预留；提交成功后唤醒 Outbox 发布器，Celery Beat 还会每 10 秒扫描未发布事件。即使首次唤醒失败，已提交事件仍会留在数据库中等待后续投递，而不会因进程瞬时故障直接丢失。

这一机制当前重点服务于**图片生成**。模型训练、质量评估、向量索引、指标同步、实验辅助等也由 Celery 执行，但并非所有任务都已统一写入 `workflow_tasks` 或支持同样的人工恢复能力。因此“统一工作流”是已落地的可靠生成链路与任务中心，不是已经完成的全业务编排平台。

2026-07-22 的运行复核发现并修复了 Celery prefork 与 asyncpg 的跨事件循环连接复用问题。所有同步 Celery 任务现在统一经 `app.tasks.async_utils.run_async_task` 执行：fork 后重置继承连接池，任务结束前释放当前异步连接。实际 Worker 中连续 Outbox 分发及 Beat 的多个 10 秒周期均成功完成。

## 当前任务模型与状态

`workflow_tasks` 保存租户、任务类型、关联资源、幂等键、请求追踪 ID、执行尝试、结果、错误信息和关键时间点；同一租户中的相同幂等键只会返回已有任务。状态枚举覆盖 `created`、`queued`、`running`、`waiting_external`、`waiting_human`、`retrying`、`succeeded`、`failed`、`cancelled`，为任务中心和后续场景提供统一语言。

当前图片生成的实际流转为：

```text
提交生成请求
  → created / Outbox pending
  → queued
  → running
  → succeeded | retrying | failed
```

- 提交前会校验功能开关、租户生成并发、月度次数和预算；同一幂等键不会重复创建可计费任务。
- Worker 启动、成功、可恢复失败和终止都会更新任务状态与素材状态，并结算或标记 AI 用量。
- 未执行的 `created`、`queued`、`retrying`、`waiting_external`、`waiting_human` 任务可在任务中心取消；取消会同时终止仍待发布的 Outbox 事件。
- 失败的图片生成任务可人工重试，重试会新增一条可审计的 Outbox 事件，并基于原始参数重新投递。运行中的任务不能由页面强制取消，以避免留下不一致结果。

## 队列、调度与可恢复性

Celery 使用 Redis 作为 Broker/结果后端，按 `orchestration`、`generation`、`model`、`analytics` 队列隔离不同资源类型。Worker 开启 `acks_late`、`task_reject_on_worker_lost`、低预取和超时控制，降低 Worker 异常时静默丢失任务的概率。Beat 定时执行数据同步、模型再训练、实验辅助、流量更新和 Outbox 扫描。

Outbox 发布器按可用时间锁定待发布事件，使用指数退避；单个事件超过当前上限后标记为失败并保留错误信息。事件和工作流记录都带租户范围，发布和消费时显式恢复租户上下文。生成进度优先经 Redis Pub/Sub 推送，Pub/Sub 不可用时前端可退回状态轮询。

## 人工决策与活动链路

人工审核、预测、A/B 实验和飞轮复盘仍由各自领域模型负责业务事实与决策。视觉运营活动把这些事实连接为可读的业务阶段，并沉淀“下一步建议、决策证据、风险、实验结果和学习洞察”。这样运营人员无需在各模块间自行拼接上下文，但系统不会把“已提交给外部服务”显示为“已完成”，也不会用模拟结果替代真实审核或经营指标。

## 当前运行边界

- Outbox 当前只分发 `generation.requested` 事件；它不是可消费任意领域事件的通用事件总线。
- 当前没有 Temporal、Saga 补偿、Inbox 去重表、死信队列界面或跨服务工作流版本治理。若将支付、发布或长时间人工等待纳入同一可靠链路，必须先补齐这些能力，而不能仅复用状态字符串。
- `waiting_external` 与 `waiting_human` 已在任务状态模型中预留，但图片生成之外的异步工作尚未完全统一为可暂停、可恢复的任务状态机。
- 任务中心目前对失败图片生成提供人工重试；其他任务类型的恢复策略仍以各自 API/Worker 的日志和重试机制为准。
- 可靠投递不等于外部供应商一定成功。供应商超时、限流、返回不合规内容或成本变化，必须在任务状态、用户提示和审计记录中如实暴露。

## 运行检查要点

- 观察 Outbox `pending`/`failed` 数量、最早待发布时间、事件尝试次数和生成队列长度。
- 观察各任务的成功率、重试率、P95 完成时间、人工取消/重试次数，以及按租户的预算预留与结算差异。
- 对 Worker 重启、Redis 短暂不可用、外部生成超时、重复提交和人工取消做故障注入；验收标准是任务可定位、可重试或可解释，而不是仅仅 API 返回 202。
- 通过任务 ID、请求 ID 和审计追踪 ID 关联 API 日志、Outbox 事件、Worker 日志和业务素材记录。
