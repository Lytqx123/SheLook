# 第四阶段：数据平台与弹性服务（当前状态）

> 文档基线：2026-07-23。经营事实、真实效果事实、预测快照与反馈标签在迁移 `018` 后已分离保存。

本文描述 SheLook 当前数据层和弹性运行方式。系统已具备可扩展的基础设施边界，但尚未实现独立数仓或完全拆分的微服务体系。

> 复核说明（2026-07-22）：development 环境的 PostgreSQL、Redis、MinIO 已由 API 就绪探针同时确认可用，MinIO 业务桶完成初始化。该连通性检查不等同于备份恢复、跨区复制或高负载弹性验证。

## 当前数据平面

| 组件 | 当前职责 | 运行约束 |
| --- | --- | --- |
| PostgreSQL + pgvector | 业务事实、租户关系、审核/实验/指标、活动洞察与向量索引 | 生产 API/Worker 必须使用受限运行账号；迁移使用独立 owner 账号 |
| Redis | API 限流、短期缓存、Celery Broker/结果、WebSocket Pub/Sub、会话撤销 | 缓存失败不应阻断普通读取；生产授权校验按失败关闭 |
| MinIO | 私有草稿素材、公开发布素材、预签名读取 | 草稿默认进入私有桶；发布后复制到公开桶 |
| Celery | 生成、编排、模型与分析类后台执行 | 队列独立配置并发，避免生成任务挤占轻量任务 |
| 模型工件卷 | 预测模型与相关工件 | Compose 卷与 Kubernetes PVC 均已提供基础挂载 |

数据库连接池、语句超时、锁超时、连接回收和应用标识已配置为环境变量。Kubernetes API 清单以多个 API 副本为基础，并配置 HPA；Worker 按队列部署为不同工作负载，因此 API 扩缩容不会直接等同于生成能力扩缩容。

## 数据读写与缓存策略

- 商品列表和看板汇总使用租户范围的 Redis 短期缓存。缓存键包含租户和筛选条件；商品写入通过版本号失效缓存，缓存异常时回落到数据库查询。
- 列表 API 使用分页和服务端筛选，避免把全量商品、审核、任务或实验一次性传到前端。当前缓存只覆盖热点读模型，不把 Redis 当作业务事实来源。
- 图片对象以 bucket + object key 为稳定定位；私有对象返回短期签名 URL，公开对象返回稳定 URL。数据库不应把会过期的签名 URL 当成永久事实。
- 生成、审核、预测、实验和指标数据保留在同一事务型数据库中，便于活动详情按当前租户聚合证据、风险与结果。

## 平台数据与模型数据

指标采集层提供 Shopee、Amazon 和 Lazada 的适配接口，并将外部 listing 映射为内部图片/素材维度的日指标。Shopee 与 Amazon 的凭据、区域和 Marketplace 信息由管理员在“外部 API 配置”页面按租户录入并加密保存，Worker 只读取已启用配置才会真正拉取；Lazada 的公开接口没有经过验证的 listing CTR 数据源，当前要求将已获授权数据通过批量指标接口导入。Amazon Sales and Traffic 报告没有 listing 点击量时会保留 CTR 为空，不会伪造点击或 CTR。

店小秘采用独立的租户连接模型，而不是复用环境变量中的全局凭据。管理员通过 Web 集成中心维护店铺引用、同步范围、间隔和只写授权字段；后端加密保存凭据，记录配置版本和审计事件。同步任务由 Worker 执行，并写入 `IntegrationSyncRun`；在获得商户授权、接口地址、签名规则、字段契约、分页/游标和限流规则前，任务会明确标记为 `awaiting_provider_contract`，不会报告“同步成功”。

企业数据接入新增 `ExternalEntityMapping`、`CommerceFact`、`PerformanceFact`、`PredictionSnapshot` 和 `ModelFeedbackLabel`。ERP 的商品、刊登、库存、订单和履约以经营事实保存；真实 CTR 必须来自包含曝光和点击的店铺分析、广告或企业 BI 原始事实，并以 `sum(clicks) / sum(impressions)` 计算。只有完成映射、达到成熟时间且满足最小曝光阈值的数据才生成反馈标签；反馈标签不会回写或篡改预测记录。

向量检索使用 pgvector 与 CLIP 特征，预测与飞轮任务使用模型工件目录和版本化训练产物。活动洞察记录的指标快照、置信度和建议用于复盘和下次决策，但不是替代原始指标表的第二份数据源。

## 当前弹性边界

- 当前后端仍是模块化单体，工作负载按 API/Worker 类型拆分，而不是按每个业务领域拆成独立服务。这样可先按生成、模型、分析和 API 的资源差异扩缩容，同时保持事务和权限边界清晰。
- PostgreSQL 当前没有仓库内定义的读写分离、分区表、跨区域复制或 OLAP 数仓；高基数报表、长期归因和超大规模导出不能假设已具备无限横向扩展能力。
- Kubernetes 清单提供 API HPA、PDB、资源请求/限制和安全上下文，但不包含云厂商数据库、对象存储、入口、证书或跨可用区策略。它是部署起点，不是完整生产平台。
- 模型文件目前通过共享卷挂载。若模型版本体积、发布频率或跨区域读写显著增长，应以对象存储制品库和显式版本发布替代共享文件系统假设。

## 运行检查要点

- 按 API、生成、模型、分析队列分别观察排队时间、吞吐、失败率与资源占用；不要只监控总 CPU。
- 观察数据库池等待、慢查询、锁等待、缓存命中率、Redis 连接错误和 MinIO 签名/上传失败。
- 平台同步要记录凭据状态、拉取范围、映射缺失、字段空值和重复写入，而不是把未授权或数据缺失显示为“同步成功”。
- 在扩大副本或连接池前，先核算所有 API 与 Worker 实例的连接总量，确保低于 PostgreSQL 的实际连接预算。
